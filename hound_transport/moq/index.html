§MoQ media transport over an iroh Endpoint

The IO shell that moves encoded media over a MoQ session (moq-net) carried on a raw-QUIC iroh connection (web-transport-iroh). MoQ’s group/object model owns framing and (for video) loss recovery, so the custom packetizer/reassembler in [hound_media] is not on the video path.

§Sans-IO boundary

This shell only does IO. The sans-IO [hound_media] cores it sits in front of — jitter/playout ([hound_media::PlayoutBuffer]), the A/V clock ([hound_media::CaptureClock]), the BWE→encoder loop ([hound_media::RateController]/[hound_media::QualityLadder]) and AEC — are never touched here. The publisher takes encoded [hound_media::Frame]s (exactly what the encoder produces) and the subscriber hands back reassembled [hound_media::Frame]s (exactly what jitter/playout expects). The clock is injected as now_us.

§Frame ↔ group/object mapping

one media flow   →  one TRACK              (the subscribe unit)
one GOP          →  one GROUP              (its own QUIC uni-stream)
one encoded frame →  one OBJECT (frame)    (opaque payload, byte-for-byte)
  • keyframe → NEW group boundary. A keyframe finishes the open group and create_groups the next sequence; the keyframe payload is the first object in the new group.
  • delta → object appended to the open keyframe group (write_frame). A delta before any keyframe is rejected — there is no decodable group to anchor it.
  • object payload is forwarded verbatim — nothing decodes or rewrites it.

§Delivery mode & why it kills the keyframe-after-deltas freeze

moq-net serves each group on its own QUIC uni-stream, so:

  • within a GOP: ordered + reliable (QUIC stream); a lost delta is repaired by QUIC stream retransmit — we do not hand-roll NACK for video.
  • across GOPs: independent streams; an old GOP’s stalled stream can never head-of-line-block a newer one. The subscriber drains with [TrackConsumer::next_group], which discards a late group at the boundary instead of waiting — freeze prevention expressed as transport semantics rather than app-level NACK.

Audio takes the same group/object path but rolls groups by COUNT, not keyframe: successive Opus frames ride one ordered group, rolled every few frames (see AUDIO_GROUP_OBJECTS).

§Live-join gate (correctness, not timing)

moq-net is live pub/sub: a subscriber joins at the latest group and does not replay groups produced before its per-track SUBSCRIBE landed. The session handshake completing does not imply the SUBSCRIBE arrived, so the publisher gates first-frame production on MoqPublisher::wait_subscribed ([TrackProducer::used]) — which resolves deterministically once a consumer attaches — guaranteeing the opening keyframe (group 0) is delivered with no sleep/yield timing dependency.

Structs§

MoqPublisher
MoQ publisher over an iroh connection. Owns the moq-net session (held alive for the track’s lifetime), the broadcast producer, and the open track producer. Hand it encoded [Frame]s via MoqPublisher::publish_frame; it maps each onto the GOP group / object model in §2.
MoqRelaySession
One MoQ relay session multiplexing MANY tracks — publish AND subscribe — over a SINGLE iroh/QUIC connection.
MoqRelaySubscriber
MoQ subscriber over the relay path (services/sfu::moq_relay).
MoqSubscriber
MoQ subscriber over an iroh connection. Accepts the moq-net session, waits for the publisher’s broadcast announce, and subscribes to a track. Drains groups/objects and reassembles [Frame]s for the jitter/playout core.
MoqTrackPublisher
One publish track on a multiplexed MoqRelaySession. Same GOP/object framing as MoqPublisher (it shares gop_publish_frame); the only difference is the TrackProducer comes from a shared session’s origin rather than its own connection. Holds its [BroadcastProducer] so the announce persists until drop.
MoqTrackSubscriber
One subscribe track on a multiplexed MoqRelaySession. Shares the priority-drop drain_next_frame discipline with MoqRelaySubscriber; the only difference is the TrackConsumer came from a shared session.

Constants§

ALPN
MoQ version-negotiation ALPN (moq-net’s ALPN_LITE). In raw mode web-transport-iroh reports the iroh connection’s ALPN as the MoQ protocol; moq-net rejects an unrecognised ALPN with UnknownAlpn, so the wire ALPN is part of the MoQ handshake contract and must be a moq-net ALPN — not an invented hound-* one.
AUDIO_GROUP_OBJECTS 🔒
Audio objects per group before rolling the group boundary (the ordered audio substream model). Audio is sequential and stateful — not GOP-structured like video — so instead of one group (= one QUIC uni-stream) PER FRAME (which gives ~50 unordered racing streams/s whose cross-stream reordering corrupts playout), successive Opus frames are appended as objects to ONE group so QUIC delivers them in order. The group is rolled every AUDIO_GROUP_OBJECTS frames because a single never-rolled group is not viable in moq-net 0.1.9 (the 1024-object / 32 MB eviction cap → fatal CacheFull; late-join only attaches at a group boundary; and a lost object head-of-line-blocks the whole group until retransmit).
DEFAULT_BROADCAST_PATH
A default broadcast path for the peer-to-peer types. In a real call the call-membership/namespace model that decides which broadcast a participant may announce or subscribe to lives above MoQ in hound-control, and the relay enforces it via the space-scoped token (see MoqRelaySession).
MAX_AUDIO_OBJECT 🔒
Per-object payload bound for AUDIO tracks (defense-in-depth; see MAX_VIDEO_OBJECT). A real Opus frame is on the order of tens to low hundreds of bytes — 256 KiB is orders of magnitude above any legitimate frame, so this only ever catches a malformed/hostile object, never real traffic.
MAX_VIDEO_OBJECT 🔒
Per-object payload bound for VIDEO tracks (defense-in-depth on top of moq-net’s own transport-level frame cap). Deliberately equals moq-net 0.1.9’s MAX_FRAME_SIZE (16 MiB): sizing here is trusted to that cap today, but a relay/moq-net change should not silently widen what this layer accepts without the bound being visibly re-examined.
