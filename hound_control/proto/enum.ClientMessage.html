MediaFeedback

A receiver’s reception report for a subscribed remote track, relayed to the track’s producer so its sender-side BWE can see loss on the downstream leg the relay hides (the receiver→producer half of the BWE loop).

Counts are since this reporter’s previous report for the same track (deltas, not cumulative), so the producer can fold them into a per-interval loss fraction directly. A reporter sends these on a ~1 s cadence while it has something to report.

Additive variant (protocol-compatible): an older server that does not know it skips it; an older client never sends it. See PROTOCOL_VERSION.

Fields

§track: TrackId
§received: u32

Frames the receiver actually decoded/accepted for this track since its last report (the denominator of the loss fraction).

§lost: u32

Frames the receiver determined missing/skipped since its last report — frame_seq gaps plus deliberate priority-drops (both signal pressure; see the receiver measurement in src/voice/session).

§jitter_us: Option<i64>

Optional inter-arrival jitter estimate (µs) for the track, if measured. Informational: producers fold only the loss fraction.

§

RequestQuality

A receiver’s standing quality request for a subscribed remote video track, relayed to that track’s producer to drive demand-driven simulcast.

cap_bps: None = “I can take the source” (no lower layer needed for me); Some(bps) = “I can sustain at most bps”. The producer coalesces every receiver’s cap into at most one shared low layer (see hound_media::plan_layers) and drops the source if no one can take it. Defensive: one standing request per (reporter, track), only meaningful below the source (a cap ≥ source is just “source”), and it is a level, not a stream demand. A receiver re-sends it only when its estimate crosses the hysteresis threshold, not every tick.

Additive variant (protocol-compatible) — see PROTOCOL_VERSION.
