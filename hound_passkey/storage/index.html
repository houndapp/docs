The platform-keyring session-token store. Behind the storage feature because it links the OS credential stores (Keychain / Secret Service) — enabled by app hosts (the FRB layer), not the backend. Platform-keyring persistence for the session token pair — the high-level storage API the app hosts consume (the FRB layer adapts it onto hound-client’s SessionStore seam).

The token pair is persisted as ONE JSON blob under a single credential, so a refresh (which rotates BOTH tokens) can never leave a torn pair. Backend per platform is selected at compile time:

  • macOS — the login Keychain (apple-native-keyring-store/keychain).
  • Linux — the freedesktop Secret Service (gnome-keyring / KWallet) over pure-Rust zbus (zbus-secret-service-keyring-store), no system libdbus/openssl link.
  • elsewhere (Windows/mobile, not yet brought online) — the in-memory mock: builds everywhere, persists nothing.

Linux blocking hazard: the secret-service store drives zbus through its blocking API, and zbus panics if a blocking call is made from inside an async runtime — which is where app hosts call this (FRB commands run on FRB’s tokio runtime). [run_blocking] hops every keyring operation onto a scratch thread on Linux (the sanctioned keyring-rs pattern for async apps); the macOS keychain is plain sync FFI and is called in place.

run_blocking 🔒
Run a keyring operation somewhere it is safe to block. On Linux the secret-service backend’s blocking zbus calls panic when made from a tokio worker thread — so hop onto a scratch thread (scoped, so closures can borrow). Keyring ops are rare (login / refresh / logout) and fast; briefly parking one async worker is fine. On macOS the keychain is synchronous FFI with no runtime detection: call in place.
