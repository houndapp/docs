hound-passkey — the sans-IO passkey / WebAuthn protocol core.

One vocabulary for the passkey ceremony, shared by the relying party (the backend) and the caller (hound-client):

  • proto — the canonical WebAuthn wire types (re-exported from webauthn-rs-proto), so every caller speaks the same options/credential shapes instead of serde_json::Value.
  • dto — the /auth/{register,login}/{start,finish} request/response structs, defined ONCE here (snake_case, matching the live wire).
  • rp — a sans-IO RelyingParty wrapping webauthn_rs::Webauthn: the four ceremony calls plus the RP-config accessors the .well-known endpoints need.
  • authenticator — the CTAP2 trait seam for client-side implementations.
  • error — PasskeyError, wrapping WebauthnError.

It performs NO IO. The DB + HTTP transport stay in the callers.

error
The error surface for the passkey core. Wraps webauthn-rs’s WebauthnError (every ceremony failure — only under the rp feature, since the relying-party state machine is what produces it) plus a generic variant for the authenticator seam and anything that doesn’t originate in the crypto machine.
proto
The canonical WebAuthn wire types — re-exported from webauthn-rs-proto so every caller (the relying party AND the client authenticator seam) shares ONE vocabulary rather than passing opaque serde_json::Values around.
rp
The relying-party ceremony wrapper. Behind the rp feature because it links webauthn-rs (and transitively openssl) — enabled by the relying party (the backend), not the client. The sans-IO relying party: a thin wrapper over webauthn_rs::Webauthn exposing the four ceremony calls plus the RP-config accessors the .well-known endpoints read. It owns NO state beyond the configured Webauthn — the in-flight ceremony state (PasskeyRegistration / PasskeyAuthentication) is returned to and stored by the caller, and the Passkey credentials live in the caller’s database.
