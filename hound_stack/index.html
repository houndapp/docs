hound-stack: the dev/CI stack lifecycle library backing the hound-stack CLI (src/main.rs). One implementation shared by every caller that needs “bring the dev backend stack up, wait for it to be healthy, tear it down” — the e2e and visual subcommands, plus the up/down manual-dev commands. Replaces the hand-synced tools/houndbot/e2e.sh / ui/integration_test/visual.sh shell scripts.

Noise contract (every flow): the console gets one line per lifecycle step; every subprocess’s full output lands in a separated artifact file (build.log for all build phases, compose.log, backend.log, sfu.log, bot logs, test output); binaries are built quietly first and then exec’d so runtime logs hold runtime lines only; a failure surfaces the relevant log’s tail inline.

Module map:

  • ports — host port preflight (fail fast on a leftover/concurrent run).
  • compose — the docker compose/docker-compose/mise fallback chain, output to compose.log.
  • wait — deadline-bounded polling (HTTP reachability, container health, log lines) — no bare sleeps beyond the poll interval.
  • cargo — quiet builds to build.log; runtime logs never see compiler chatter.
  • logs — log sectioning + failure-tail helpers.
  • backend — spawns the (prebuilt) host backend with the shared dev env.
  • sfu — forces the SFU’s one-shot registration POST to re-land.
  • artifacts — E2E_OUT_DIR/VISUAL_OUT_DIR-style output dir resolution.
  • tee — runs a child process, teeing its output to console + a log file (the Rust equivalent of cmd | tee log).
  • lifecycle — lifecycle::LiveStack: bring-up plus idempotent, drop-safe teardown (partial bring-up failures included), composing the modules above.
  • e2e — the hound-testkit scenario suite flow.
  • visual — the visual snapshot harness flow (bootstraps via hound_testkit::Env rather than re-implementing it in shell).
  • updown — manual up/down dev commands.

Modules§

artifacts
Artifact directory resolution: an env var (E2E_OUT_DIR/VISUAL_OUT_DIR — a durable, CI-collectible dir) if set, else a throwaway temp dir left in place for local inspection. Same compat contract the two shell scripts documented.
backend
The host backend child process: the exact dev env both shell harnesses ran the backend under — this is the one place that env now lives.
cargo
Quiet cargo builds, separated from runtime logs. Every binary the harness runs (backend, houndbot, snapdiff, the nextest test binaries) is built HERE first — full compiler output captured into the run’s build.log, surfaced only on failure — and then executed directly, so each runtime artifact (backend.log, bot logs, test output) contains runtime lines only, never pages of “Compiling…” chatter.
compose
Compose driver: resolves the same docker compose (plugin) → docker-compose (standalone) → mise-pinned fallback chain the shell harnesses used, so a CI host that ships bare docker (no plugin, no standalone binary — the X64 runner) still works without a host install.
e2e
hound-stack e2e: bring the stack up, run the hound-testkit scenario suite (cargo nextest run -p hound-testkit --profile e2e) against it, collect artifacts, tear down. Mirrors the retired tools/houndbot/e2e.sh. Extra CLI args forward to cargo nextest run (e.g. an -E filter for a fast local iteration loop against one scenario file).
lifecycle
The shared stack lifecycle: compose bring-up, health waits, host backend, SFU re-registration — and its mirror-image teardown (backend kill, SFU log capture, compose down --volumes). One LiveStack per invocation of e2e/visual/up; LiveStack::teardown is idempotent, so it’s safe to call both explicitly (to observe teardown before the process exits) and again from Drop — a panic, early return, or PARTIAL bring-up failure (compose up but the backend never came) still tears everything down, mirroring the shell scripts’ trap cleanup EXIT.
logs
Log-file helpers for the harness’s noise policy: every subprocess writes its full output to a per-concern artifact file, the console gets one progress line per lifecycle step, and a failure surfaces the relevant log’s tail inline — never the whole stream.
ports
Port preflight: fail fast if this run’s fixed host ports are already taken by a leftover/concurrent stack, instead of racing it through compose up/backend-startup timeouts.
sfu
SFU (re-)registration: the SFU registers its node ID with the backend at ITS OWN startup — compose up -d only (re)creates containers whose config changed, so a hound-sfu container left running from a previous invocation may be mid-interval against a freshly MIGRATE=fresh’d backend. Restarting it forces an immediate registration, then we poll the backend’s registry endpoint until a live node appears — the typed contract for “is voice routable”, never a log-line scrape (see MEMORY.md “SFU registration fragility” / services/backend/src/api/internal.rs).
tee
Run a child process, teeing its stdout/stderr to both this process’s own stdout/stderr and a log file — the Rust equivalent of the shell scripts’ cmd 2>&1 | tee "$LOG".
updown
hound-stack up/down: manual dev convenience — bring the stack up and hold it open (Ctrl-C tears it down), or force a standalone teardown when a prior up didn’t get the chance to run its own cleanup (killed terminal, crashed host).
visual
hound-stack visual: bring the stack up, bootstrap the shared community via [hound_testkit::Env] (no bash curl/jq re-implementation — see its module doc), spawn one houndbot --testcard peer into the voice call, run the real-app Flutter integration test, then snapdiff compare against the accepted baselines. Mirrors the retired ui/integration_test/visual.sh.
wait
Deadline-bounded polling — no bare sleeps beyond the poll interval (see AGENTS.md’s “Test determinism”). Every waiter here bounds total wall time and returns a descriptive eyre error on timeout instead of hanging or (worse) racing a downstream timeout with a less useful cause.
