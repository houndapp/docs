A second, test-owned hound-sfu process — the seamless-rollout migration scenario (tests/call_scenarios.rs’s sfu_seamless_rollout_migrates_clients_and_drains) spawns it from the SAME binary hound-stack e2e already builds for the stack’s own sfu1 (a compose service), on fresh ports and a short DRAIN_DEADLINE_SECS, to prove the backend’s newest-registration routing + SfuMigrate broadcast + SIGTERM-drain contract end-to-end.

Plumbing contract with hound-stack: BIN_ENV/RELAY_SECRET_ENV/ RELAY_URL_ENV are set on every cargo nextest run invocation hound-stack e2e drives (see tools/stack/src/e2e.rs) — BACKEND_URL is already provided for every scenario (see crate::env::Env::from_env). SecondSfuConfig::from_env returns None when they’re absent (e.g. a bare cargo nextest run -p hound-testkit without the hound-stack harness), which the scenario treats as a clean skip rather than a spawn-failure — every other scenario only needs BACKEND_URL/ MAILPIT_URL, which default to the compose stack’s fixed dev ports, but a SECOND sfu binary has no sane default to fall back to.
