Crate hound_testkit

Expand description

Layer-1 e2e scenario framework: drives real hound-client instances (magic-link auth, voice calls, messaging) against a live backend + Mailpit — exactly the surface a real Flutter/native client drives, nothing engine/transport-internal. hound-bot’s CLI delegates its sign-in/ stats-sampling internals to auth/call; this crate’s own tests/*.rs is the scenario suite proper.

Entry point: Env::from_env → Env::user → Participant::join / Participant::open_text for the InCall/TextSpace expectation DSL. The stack (compose + backend) is provided externally — see crates/bot/e2e.sh / mise run e2e:call.

Errors are eyre (not a typed error enum like the product crates): test support wants rich contextual reports and has no consumers that match on variants — a deliberate deviation, not an oversight.

Modules§

auth
Magic-link sign-in: trigger the email over REST (bypassing the loopback listener HoundClient’s interactive flow binds — a headless caller never clicks the emailed link, so that listener is dead weight, and skipping it makes concurrent sign-ins trivially port-safe), poll Mailpit for the code, and adopt the session. Ported out of crates/bot so both houndbot and the scenario suite (crate::env::Env::user) share one implementation.
call
InCall: a joined participant’s call handle. Every expect_* is deadline-bounded and watch-driven — it polls a tokio::sync::watch channel to a condition (never a blind sleep-then-assert), and fails with the last-observed state on timeout.
env
Env: the shared per-process scenario harness — backend/Mailpit endpoints, a lazily-bootstrapped default community (general/voice space pair), and per-run-unique participant emails.
mailpit
Mailpit polling: find the magic-link email the backend sent and pull out its sign-in code. Moved here (from crates/bot) so both houndbot and the scenario suite share one implementation.
messaging
Messaging over an opened text space: send + a deadline-bounded, watch-driven wait for a message with matching contents to arrive via the realtime plane (or be present already — history + a same-tick echo both land in the same watch_messages channel).
participant
A signed-in scenario participant: an in-process Arc<HoundClient> plus the call/messaging handles built on top of it.
sample
Test-facing glue over [hound_client::call_health]’s steady-state audio-starvation monitor. houndbot’s CI verdict and crate::call::InCall::expect_steady_audio both gate on the same hound-client type, so a call that passes one passes the other by construction — this module just adapts it to a &HoundClient.
