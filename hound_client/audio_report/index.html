§Agent-readable audio-quality debug report

A pure function — audio_report — that turns one StatsSnapshot into a compact, deterministic, paste-able text block that pre-attributes an audio glitch to a pipeline stage. The point is that a reader (a human or an agent) sees a cause, not a wall of counters: a diagnosis: headline names the most likely fault, a per-stage table flags each signal ok / HIGH / BAD, and a glitch log lists the recent discrete events with timing.

§The pipeline stages (any can cause a stutter)

sender ─▶ network ─▶ receiver jitter buffer ─▶ mix tick (RT) ─▶ device ring ─▶ DAC

The receiver can directly observe everything from network rightward; the report is built entirely from signals the crate::stats snapshot already carries (inter-arrival jitter, loss/reorder, jitter-buffer depth vs the recommended depth, drop counts, per-peer underflow, the mix-tick lateness percentiles, output-ring underruns) plus the discrete glitch log.

§Why a text format (not just the JSON)

The raw snapshot is for the live panel; this is for diagnosis. It is stable, self-describing, and small enough to paste into a chat with an agent — and the diagnose rules collapse the six-stage signal space to a single most-likely cause so the reader doesn’t re-derive it every time.

Pure + deterministic ⇒ golden-tested per fault class (shallow buffer / RT descheduled / loss / healthy), which is what keeps the agent-facing format honest.
