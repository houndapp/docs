Pure reducers: fold REST loads + realtime events into the typed view-models the UI streams. Each reducer is a plain struct with no IO and no async, so it is trivially unit-testable and deterministic — the caller drives it from the crate::realtime event stream and REST results, then snapshots it for the FRB layer.

Three reducers, matching the three things the UI renders:

  • RosterView — display identities keyed by user id, merged from every source (member list, message authors, voice peers). Last writer with real fields wins.
  • MessageList — an ordered, de-duplicated per-space message list that applies create/edit events.
  • CommunityTree — communities and their spaces, applying space create/remove.

Structs§

CommunityTree
The community → spaces tree the sidebar renders. Holds communities and, per community, its spaces; applies space:create/space:remove realtime events.
MessageList
An ordered per-space message list. Ordered by sent_at_ms then id (the v7-uuid tiebreak keeps same-ms messages stable). De-duplicates by id so a create event for a message already loaded via REST is idempotent; an edit updates contents in place.
RosterView
The display roster: a Member per user id, the UI joins messages + call peers against. Merged from many sources; an entry with a real name/picture is never overwritten by one with empty fields (so a bare member-list user_id doesn’t clobber a name learned from a message author).
