§Steady-state audio-starvation monitor

A warmup-anchored pass/fail gate over the same StatsSnapshot stream the audio debug report (crate::audio_report) diagnoses per-tick — the difference is this module answers “is the call healthy”, not “why isn’t it”. Shared by any long-running call harness that needs a boolean verdict: the hound-testkit scenario suite’s InCall::expect_steady_audio and houndbot’s CI smoke-test verdict both gate on sample_call_health, so a call that passes one passes the other by construction.

§Why warmup-anchored

Every peer’s playout jitter buffer starts EMPTY and primes over its first few seconds — on the raw counters that priming is indistinguishable from starvation. Gating on absolute counts would fail every honest run. Instead: once min_peers have been continuously present for WARMUP, the starve-signal total observed at that instant becomes the baseline; starvation is then any LATER window’s total exceeding it.

§Never a blind sleep

sample_call_health is watch-driven: it re-checks the observation deadline on every StatsSnapshot change (capped so a quiet stream still re-checks), never a fixed sleep-then-assert.

§The starvation signal

Sums the exact counters crate::audio_report::diagnose attributes a glitch to — device-side output underruns/mix-starves, plus each peer’s jitter-buffer underflow/late-drop/overflow-drop/loss — into one pass/fail total instead of naming a cause. Reach for the audio debug report when this fails and you need to know why.

Structs§

CallHealthSample
One observation window’s rollup. Peer/frame counts are high-water marks over the whole window (a peer that briefly joined/left, or a frame count that can only grow, are both captured correctly by tracking a running max rather than reading only the final sample); starved is the warmup-gated verdict.

Constants§

WARMUP
How long min_peers must have been continuously present before the starve-signal baseline is taken. See the module doc for why this can’t be zero.

Functions§

sample_call_health
Watch stats for duration, gated on min_peers (see WARMUP). The caller supplies the receiver (e.g. HoundClient::watch_stats()) — this function has no dependency on any particular client/frontend, only the wire snapshot type.
starve_total 🔒
Every starvation signal sample_call_health gates on: device-side output underruns/mix-starves plus each peer’s jitter-buffer underflow/late-drop/ overflow-drop/loss counters. Cross-references crate::audio_report::diagnose’s per-stage rules — this sums the same counters into one pass/fail total rather than attributing a cause.
