§hound-client — the unified, headless client core

The single source of truth for every hound client. It owns auth, the REST data plane, the realtime (/community) plane, and the media/call engine; reduces them into the typed view-models in model; and exposes commands + snapshot streams. UI layers (via hound-frb) are thin views over this — no client logic lives in Dart.

Layout:

  • app — the HoundClient orchestration facade: observable watch channels + the realtime→reducer task.
  • error — the typed HoundError surface.
  • model — typed view-models the UI renders.
  • rest — the REST data plane.
  • realtime — the socket.io /community plane.
  • session — auth/session lifecycle: SessionStore, token refresh, AuthState.
  • reducer — pure folds of REST loads + realtime events into view-models.
  • call — the voice/video call orchestration engine: join → control WS → MoQ → publish/subscribe → decode, reusing the engine/transport/ control/media crates. UI-free: the decode→render seam is a call::VideoSink the FRB layer supplies.
  • stats / audio_report / call_health — the RTC debug-stats snapshot, its agent-readable diagnosis report, and the warmup-anchored pass/fail starvation monitor built on top of it.

app
The orchestration facade: ties the REST data plane, the session lifecycle, and the realtime (/community) plane into one observable, UI-free app core.

Voice/video call orchestration — the UI-free engine that drives a live call from HoundClient.

Steady-state audio-starvation monitor
error
The single, typed error surface for the whole client. Every command returns Result<T, HoundError>; the FRB layer maps this 1:1 onto a Dart sealed class so the UI matches on variants instead of parsing strings. Keep variants coarse + stable — they are a public contract the UI switches on.
model
The typed view-models the UI renders — the data plane’s shape, owned here so there is one source of truth (no drift between a Dart model and a Rust model). These mirror the backend contract (ui/src/lib/openapi/*, services/backend/src/api/*) but are client view-models: only the fields the UI needs, in client-friendly forms. They are plain serde types so the FRB layer mirrors them into Dart and reducers can snapshot them.
realtime
The realtime plane: a socket.io client for the backend /community namespace (server is socketioxide 0.18 — Socket.IO v5 / Engine.IO v4, which [rust_socketio] 0.6 speaks). It connects with { token } handshake auth, lets the caller subscribe/unsubscribe to per-space rooms, and decodes every server event into a typed RealtimeEvent delivered over a [tokio::sync::mpsc] channel — so the reducer consumes a single typed stream, never raw JSON.
reducer
Pure reducers: fold REST loads + realtime events into the typed view-models the UI streams. Each reducer is a plain struct with no IO and no async, so it is trivially unit-testable and deterministic — the caller drives it from the crate::realtime event stream and REST results, then snapshots it for the FRB layer.
rest
The REST data plane: a thin, typed wrapper over the backend HTTP API. Holds the session bearer token in a lock-free cell (read on every request, republished by the auth refresh task) and maps transport/status failures onto HoundError.
session
Auth/session lifecycle: where tokens live, how they persist, and how they refresh.
stats
RTC debug stats (call:stats)
