Auth/session lifecycle: where tokens live, how they persist, and how they refresh.

The crate must NOT depend on any platform keyring, so persistence is injected via the SessionStore trait — the host (Flutter via FRB) supplies a keyring-backed impl; tests use MemorySessionStore. Session owns the live tokens, drives refresh (proactive at ~half the access-token lifetime, or on demand after a 401), and republishes the access token onto the shared RestClient so every request reads the current value. It also tracks an AuthState snapshot for the reducer.

user_id_from_jwt 🔒
Extract the sub (user id) claim from a JWT access token without verifying the signature. The client trusts a token the server just minted; signature verification is the server’s job on every request, not the client’s — we only need to read who the token is for. Returns None if the token isn’t a well-formed JWT or carries no parseable sub. (Backend claim shape: services/backend/src/api/auth/jwt.rs.)
