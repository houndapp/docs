§RTC debug stats (call:stats)

The shared counter/gauge registry + 1 Hz emitter behind the call:stats snapshot — the data source for the in-app RTC debug panel. One structured snapshot per second of everything the live call can observe, grouped by section (transport / outbound audio / outbound video / per-peer inbound).

§Design: cheap hot paths, rates computed off the hot path

Every media loop (capture / encode / decode / playback / publish / subscribe) only ever does a single relaxed atomic increment to record a stat — no locks, no allocation, no clock read. The registry is a bag of AtomicU64 / AtomicU32 counters (monotonic totals) and gauges (last-written values), grouped into per-section structs and shared as one Arc<CallStats> across the pipelines exactly the way the call’s AdaptShared seam is.

Windowed rates (fps / kbps) are NOT measured on the hot path. The emitter samples the monotonic counters once per second and divides the delta since the previous sample by the elapsed wall time — so a frame-per-second or kilobit-per-second figure costs the hot path nothing but one fetch_add.

§Extending it (the whole point)

Adding a stat is two edits: bump one atomic from the producing loop, and add one field to the matching *Snapshot struct (a counter total, or a rate derived in StatsEmitter::sample). The UI renders the snapshot generically, so a new self-describing field (snake_case, units in the name — _ms, _kbps, _pct, _fps) appears in the panel automatically.

§The payload contract (consumed verbatim by the UI)

The serialized StatsSnapshot is the wire contract. See its doc comment and each section struct for the exact field list and units. This is intentionally separate from call:levels (10 Hz, hot speaking meter) — do not merge them.

StutterEpisode 🔒

Emission only — callers still see the on-window d_under/d_starve deltas via the existing playout-health counters, so no gauge/counter this pins on changes.

Enums§

InboundAudioKind
Which inbound audio stream a stat applies to. A peer’s microphone and its shared screen-share app audio ride SEPARATE mic/screen_audio tracks and are counted separately (a single peer can have both at once).
InboundVideoKind
Which inbound video stream a stat applies to. A peer’s camera and its shared screen ride SEPARATE cam/screen tracks and are counted separately (a single peer can publish both at once) — without this split a peer sharing screen AND camera would have both tracks’ frames land in one bucket, so decoded_fps would read as the SUM of the two rather than each stream’s true rate. Mirrors InboundAudioKind.
StutterLog 🔒
What StutterEpisode::observe tells the caller to do with this window’s log.
TransportState
Coarse transport state, stored as a u8 gauge so the supervisor can write it from any task without a lock. Mirrors the call:connection lifecycle but is folded into the one-snapshot-per-second debug event as a string.

Constants§

GLITCH_HISTORY_CAP 🔒
Hard cap on retained glitch events (drop-oldest) so a sustained-stutter storm can’t grow the history unbounded between the time-based evictions.
GLITCH_WINDOW_US 🔒
How far back (µs) the rolling audio-glitch history retains events — the window the debug report’s glitch log covers (~12 s of recent past).
SHORT_ID_LEN 🔒
How many leading characters of the SFU node id to surface (enough to disambiguate which relay a session landed on without dumping the whole key).
STATS_EVENT
Event/channel name for the RTC debug snapshot.
STATS_INTERVAL
Emit rate for call:stats: 1 Hz. A debug panel does not need to update faster than once a second, and the per-second cadence makes the windowed rates (fps/kbps) read directly as “per second” without scaling. Deliberately an order of magnitude slower than call:levels (10 Hz) — this carries a much larger payload and is advisory diagnostics, not a live meter.
STUTTER_SUMMARY_WINDOWS 🔒
How many consecutive imperfect windows form a “summary due” period. At the 1 Hz tick cadence this is ~30 s — often enough that a live run still sees the episode is ongoing, rare enough that it reads as a heartbeat rather than per-second spam.
SUSTAINED_STUTTER_WINDOWS 🔒
Consecutive glitchy 1 Hz windows before the emitter auto-logs the full audio debug report once (so a live run self-documents bad audio without a reproduction). Three seconds of sustained stutter is well past transient and worth a loud, diagnosable log line; it re-arms only after a clean window.
