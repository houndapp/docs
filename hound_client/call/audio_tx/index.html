Module audio_tx 

Source
Expand description

Outbound audio: the mic capture loop + Opus publish, and the screen APP-AUDIO leg (its !Send capture thread + publish task). Encode + publish run on the RT audio runtime (“audio is sacred”: a screenshare encode can’t stall the mic), and audio frames publish via the ordered-substream path (count-rolled groups) so successive Opus frames ride one in-order QUIC stream — never one stream per frame.

Functions§

mic_capture_loop 🔒
The mic capture loop: pull PCM, gate on mute, meter the level, forward to the encoder. Hot-swaps the device when device_rx changes; re-opens on error.
publish_audio_one 🔒
Publish one AUDIO frame via the ordered-substream path (count-rolled groups), so successive Opus frames ride one in-order QUIC stream instead of one stream per frame (the source of the cross-stream reordering). Use for mic + screen-audio.
screen_audio_capture_loop 🔒
The !Send screen APP-AUDIO capture thread: block on the engine ScreenAudio (the mic-thread analogue — no APM, no levels, no hot-swap) and forward each 20 ms PCM frame to the publish task. A next_frame error/panic is the leg’s clean end (its parent screen stream stopped); drop the leg on this owning thread and exit.
spawn_mic_publish 🔒
Spawn the mic publish task. A blocking capture thread pulls PCM from the engine Microphone (hot-swappable via device_rx); the async half Opus-encodes and publishes each frame on the relay session’s mic track. Muting gates the encoder (no frames sent) and zeroes the reported level.
spawn_screen_audio_publish 🔒
The async screen AUDIO publish task: the mic publish path MINUS APM / levels / hot-swap (app audio is not a mic). Opus-encodes each forwarded PCM frame and publishes it on the screen_audio track; the receiver plays it on the shared mix bus exactly like a mic.
