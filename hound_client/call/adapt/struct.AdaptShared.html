Expand description

The shared seam between the async adaptation tick (owns the sans-IO [VideoAdaptation] core + observes RTT) and the !Send capture/encode thread (owns the encoder, applies the decision per frame). Plain atomics — no lock crosses the thread/async boundary and each field has exactly one writer.

Data flow:

  • capture thread → tick: produced / dropped (per-tick backpressure).
  • tick → capture thread: target_bps (encoder retarget), target_fps (fps gate), target_w/target_h (downscale to the rung), paused (AudioOnly ⇒ skip encode/publish), force_keyframe (a throttle-approved IDR).
  • keyframe listener → tick: pli_pending; feedback listener → tick: remote_loss_bps/remote_reported.

Fields§

§produced: AtomicU32

Encoded access units handed to the publisher since the last tick.

§dropped: AtomicU32

Of those, how many were dropped at the bounded channel (publisher behind).

§target_bps: AtomicU64

The encoder’s current target bitrate (bits/sec); the capture thread retargets only when it changes. 0 until the first decision.

§target_fps: AtomicU32

The chosen rung’s frame rate (fps); the capture thread downsamples to it. 0 = no limit (no decision yet / AudioOnly — the publish path is skipped).

§target_w: AtomicU32

The chosen rung’s encode geometry (width/height); the capture thread downscales the frame to this when smaller than capture. 0 = no decision yet / AudioOnly.

§target_h: AtomicU32§paused: AtomicBool

Whether video is paused (the ladder chose AudioOnly). The capture thread keeps capturing (self-preview, A/V-sync clock) but skips encode/publish.

§pli_pending: AtomicBool

A PLI arrived for our track (set by the keyframe listener); the tick drains it into the keyframe throttle to coalesce a storm.

§demand_seen: AtomicBool

Latched (never cleared) once ANY keyframe request named our track — proof a receiver wants this video. The publish loop’s live gate accepts it as a fallback for the moq subscriber signal, which can be lost when a broadcast is replaced on the same path (a share restart): every subscribe sends RequestKeyframe, so demand always reaches us on the control plane even when used() never fires.

§remote_loss_bps: AtomicU32

Worst receiver-reported loss fraction this interval, in basis points (loss × 10_000), max-reduced across reporters. Read+cleared by the tick.

§remote_reported: AtomicBool

Whether any receiver reported this interval — lets the tick pass None (no remote signal) vs Some(0.0) (a receiver explicitly reported a clean leg).

§force_keyframe: AtomicBool

A throttle-approved keyframe is due — the capture thread swap-clears this and forces the encoder’s next frame to an IDR.

§target_codec: AtomicU8

The codec the encoder should be producing, as a codec_rank byte. Seeded at publish from the roster pick and re-driven by the mid-call re-negotiator (spawn_codec_renegotiator); the encode loop rebuilds its encoder when this no longer matches what it built. Defaults to 0 = H.264 (the floor).

fn record_remote_feedback(&self, received: u32, lost: u32)

Record one receiver’s reception report, max-reducing into this interval’s worst loss fraction (serve-the-worst-leg). received/lost are deltas since the reporter’s previous report. An empty report (received + lost == 0) is ignored.

Source

fn take_remote_loss(&self) -> Option<f64>

Drain this interval’s worst receiver-reported loss fraction (and clear it). None when no receiver reported — so the tick does not pin the loss-equivalent to 0 and mask the local signals on a feedback-free interval.
