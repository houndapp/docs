The shared per-frame encode→adapt→publish core for both video capture loops. Retargets the encoder to the BWE’s latest bitrate (only on a real move), honours a throttle-approved keyframe (swap-clears so it fires once), downsamples to the rung’s fps, scales the frame to encoder_dims when it doesn’t already match (the caller keeps encoder_dims in sync with the live encoder — see the geometry-rebuild note below), encodes, and forwards each access unit via forward. forward returns false on backpressure (its bounded channel was full); the caller’s drop bumps dropped, which the adaptation tick reads as the congestion signal. A forced keyframe is re-armed when it is dropped, so the next captured frame re-emits it.

encoder_dims is the geometry the passed-in encoder was ACTUALLY built at — not necessarily frame‘s own (capture) geometry. The caller (the capture loop) is responsible for rebuilding the encoder to the adaptation’s dwelled rung geometry (AdaptShared::encoder_w/encoder_h) and passing the matching encoder_dims here; this function’s job is only to make the FRAME match whatever the encoder actually is, scaling down when the two diverge and skipping gracefully (never spam-rejecting) if a scale is impossible. This is the fix for the original resolution-adaptation regression: a rung-downscaled frame was once fed to an encoder still fixed at a different geometry (“frame WxH != encoder W’xH’”), rejecting ~every frame (produced≈0, dropped≈all). Now the two can never diverge for long — the capture loop rebuilds the encoder to match the dwelled rung, and this scale-to-match is the defensive fallback for the one-tick window before that rebuild lands.
