Module reconnect 

Source
Expand description

Reconnect policy + the retry loop: the pure exponential Backoff with bounded jitter, the fatal-vs-retryable classify verdict over HoundError, and reconnect_loop — which re-runs establish_live until success, a fatal error, or cancel. Invariant: only an auth/permission outcome is fatal; everything transient retries FOREVER (each attempt re-mints the relay token via voice/join, so a stale token heals on retry).

Structs§

Backoff 🔒
A pure exponential-backoff sequencer with bounded jitter. Sans-IO: it computes delays but never sleeps, so its whole policy is unit-testable. jitter is injected (a closure returning a fraction in [-1.0, 1.0]) so tests can pin it.

Constants§

BACKOFF_FACTOR 🔒
Per-attempt growth factor: 0.5s → 1 → 2 → 4 → 8 → (cap) 15s. Classic doubling.
BACKOFF_INITIAL 🔒
First reconnect delay. Short enough that a blip (a relay restart, a brief network stall) heals almost immediately, long enough not to busy-loop a restarting backend.
BACKOFF_JITTER_FRAC 🔒
Jitter band, ±20% of the base delay — de-correlates a fleet that all dropped at once (a relay bounce) so they don’t reconnect in a thundering herd.
BACKOFF_MAX 🔒
Ceiling on the base delay; past this every attempt waits ~15s (± jitter).

Functions§

classify 🔒
Classify a join/re-establish failure. Only an auth/permission outcome is fatal: Http {401|403} or HoundError::Auth. Everything else — network, realtime (control WS), media (relay dial/handshake), internal blips, an invalid-state or not-found/decode hiccup mid-redeploy — is transient and retried forever (a relay token is re-minted fresh each attempt, so a stale token heals on retry).
reconnect_loop 🔒
Retry establish_live with backoff until it succeeds, hits a fatal error, or is cancelled. Publishes Reconnecting { attempt } per attempt. The backoff sleep is inside a select! against cancel, so a leave mid-backoff wakes immediately.
rng_jitter 🔒
A lightweight jitter source for production: a fraction in [-1.0, 1.0] derived from the wall-clock nanosecond field hashed through a SplitMix64 step. No RNG dependency needed — the goal is only to de-correlate a fleet of reconnecting clients, not cryptographic randomness. (Tests pin next_delay’s jitter directly, so this stays out of the deterministic path.)
