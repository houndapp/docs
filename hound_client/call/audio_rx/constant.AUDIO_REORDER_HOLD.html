const AUDIO_REORDER_HOLD: usize = 3;
Expand description

Frames held in the encoded-audio reorder buffer to undo network reordering before the (stateful) Opus decode. recv_group delivers frames in ARRIVAL order; on the shared connection a screenshare reorders the tiny per-frame audio streams, so a frame can arrive after a later one. Decoding in arrival order forces dropping the late frame and concealing it with PLC — reordered becomes lost, and on a busy connection that is continuous distortion. Holding a few frames and releasing the oldest by timestamp restores a clean in-order stream into the decoder; genuine loss still surfaces as a real gap. 3 frames (~60 ms) absorbs the multi-frame reordering the screenshare induces; the added latency is recovered by the PCM jitter buffer shrinking once arrival is smooth. NOT done via a recv timeout — the moq group read is not cancel-safe mid-object — so release is purely count-based.
