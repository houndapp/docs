hound_client::call

Module session 

Source
Expand description

The live call: CallSession (join/leave/commands + the cross-reconnect command seams), establish_live (one full control+MoQ session), and the auto-reconnect supervisor — which owns the live session, the backoff, and the screen-share state (so a share is republished against each fresh session). Invariant: recovery is a FULL re-establish (fresh PeerId) — the broken session’s tasks/threads are fully torn down before a rebuild, and only an auth/permission failure (or an explicit leave) ends the call.

The handles for one running screen share: the capture-thread stop flag and the async publish tasks (video + optional app-audio). Stopping it latches the flag (the !Send capture thread drops its engine handles on its own thread) and aborts the tasks. Owned by the supervisor; replaced on a new share, cleared on stop/leave.

Enums§

ScreenCommand 🔒
A screen-share command routed to the supervisor (which OWNS the running share so it can republish on reconnect). Start carries a reply oneshot so the async screen_share_start can await the open result (a permission/unsupported error must surface to the caller, not be swallowed). Ended/Resized/Retry are the capture thread’s self-reported exits (via ScreenExitNotify): Ended is a clean user/OS stop (tear down + unpublish + drop the desire), while Resized/Retry RESTART the same share in place against the current session (the share must NOT end on a resize or a transient glitch).

establish_live 🔒
Build one full control+MoQ LiveSession. Each establish re-mints the relay token by calling voice/join, so a token-expiry SFU drop heals via reconnect — that re-mint is what keeps a call alive past the relay token’s TTL. The capture loops subscribe to params.devices.* (so device selection survives a reconnect) and read params.mute/params.camera_enabled. Seeds the identity map from the fresh voice/join each time.
handle_screen_command 🔒
Handle one ScreenCommand inside the supervisor loop, against the CURRENT live session. Start replaces any running share (stop-old-first + unpublish), starts a fresh pipeline, records the desired source for cross-reconnect republish, and replies on the oneshot. Stop and Ended both tear the pipeline down + unpublish the tracks + drop the desire (a clean user/OS stop). Resized/Retry RESTART the same share in place (the share must NOT end on a resize or a transient glitch).
restart_screen_share 🔒
Tear down the running screen pipeline and start it again against the SAME source (recorded in desired_source). Shared by the Resized and Retry exits: receivers see the screen track replaced, the desire is untouched on success, and a restart whose spawn_screen_publish fails — the source truly vanished — degrades to a clean end (drops the desire so a later reconnect won’t republish it).
spawn_heartbeat 🔒
Keep the backend voice-membership alive (resets its eviction timer). No relay-token refresh here: a reconnect re-mints the token via voice/join, so a token-expiry SFU drop (moq code=4) heals through the supervisor with a fresh session + token.
spawn_supervisor 🔒
Spawn the auto-reconnect supervisor. It owns the live session + a Backoff + the screen-share state, monitors every layer, and on a failure tears the broken session fully down and rebuilds it with exponential backoff + jitter — re-minting via voice/join each attempt (which also refreshes the relay token). It gives up only on a FATAL (auth/permission) error or an explicit leave (cancel); transient errors retry FOREVER. State transitions are published on channels.connection.

The cross-reconnect outbound-control seam. control_outbound changes on every (re-)establish (it is the new control WS’s writer queue), so commands must reach the CURRENT session: the supervisor swaps the live sender in here on each establish, and set_muted (etc.) loads it to send.
