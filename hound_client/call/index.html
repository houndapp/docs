Module call 

Source
Expand description

Voice/video call orchestration — the UI-free engine that drives a live call from HoundClient.

It parallels the legacy Tauri orchestration in src/voice/session/* but reuses the engine/transport/control crates directly and stays free of any UI/FFI machinery (no Tauri Channel/AppHandle/emit). The decode→render seam is a VideoSink trait the FRB layer supplies (its closure calls video_render::push_frame(...)); decoded frames reach the right tile keyed by peer without this crate ever knowing about textures or Dart.

§What a join wires up

REST voice/join  ──► relay_url + relay_token + sfu_node_id + peers
       │
       ├─► control WS (hound-control)  ──► SessionView reducer ──► watch<CallView>
       │
       └─► dial iroh ──► MoqRelaySession ─┬─► publish mic (Opus)
                                          ├─► publish camera (H.264) [if enabled]
                                          └─► subscribe loop: for each remote track
                                                video → decode → VideoSink (push_frame)
                                                audio → decode → mix bus (PlayoutBus)

§Module tree

This module is the shared hub — the public seams (VideoSink, RenderSurfaceHandle), the picker enumerations, the observable snapshot types, the dedicated runtimes, and the cross-module tuning constants. The orchestration lives in the submodules:

  • session — CallSession (join/leave/commands), establish_live, the auto-reconnect supervisor + screen-share command handling.
  • reconnect — the pure backoff/classification policy + the retry loop.
  • control — the /control WebSocket IO shell + the iroh relay dial.
  • policy — the pure, unit-tested decision policies (codec switch, layer hysteresis, restart pacing, resize debounce, gain, gap math…).
  • adapt — sender-side BWE/congestion adaptation glue (adapt::AdaptShared, the 1 s tick, the keyframe/feedback listeners, the shared encode core).
  • simulcast — demand-driven simulcast (the layer planner + low publish).
  • audio_tx / camera_tx / screen_tx — the publish paths.
  • subscribe — the roster-diffing subscribe loop + per-track pipelines.
  • audio_rx / video_rx — the decode pipelines.

§Observable state

Three watch channels, driven the same way as the realtime→reducer→watch task in crate::app: CallView (self id, peers, tracks) is derived from the control plane’s [SessionView]; ConnectionState reflects the control WS + reconnect attempts; LevelsView carries mic + per-peer speaking levels.

§Auto-reconnect supervisor

A live call heals a dropped control WebSocket or MoQ/iroh connection without the user re-joining — “nothing drops a call except total network loss.”

  • reconnect::Backoff + [reconnect::classify] — the pure, headless policy (exp-backoff+jitter; retryable-vs-fatal over HoundError). Only an auth/permission outcome (Http 401/403 or Auth) is fatal; everything transient retries FOREVER.
  • session::LiveSession — the per-establish layers (relay/conn/control/tasks). Its monitor selects over the control-WS-closed signal and the iroh Connection::closed(), naming which layer dropped.
  • session::ConnectionParams — the Arc-cloneable cross-reconnect state. establish_live builds one LiveSession (re-minting the relay token via voice/join each time, so no standalone token-refresh task is needed).
  • [session::spawn_supervisor] — owns live + a Backoff + the screen-share state; monitors, tears down on failure, and rebuilds with backoff. Recovery is a FULL re-establish (fresh PeerId); the reactive subscribe loop re-diffs the roster for free and re-publish is a fresh mic/cam announce.

Modules§

adapt 🔒
Sender-side BWE/congestion adaptation. The camera + screen video the client PUBLISHES adapts its bitrate/resolution to network conditions and serves keyframes on demand. The brain — [hound_media::VideoAdaptation] — is shared; this is the local IO glue:
audio_rx 🔒
Inbound audio: pull → reorder by timestamp → Opus decode (bounded PLC on genuine gaps) → per-peer jitter buffer on the mix bus, plus level metering. The encoded-frame reorder hold undoes the cross-stream reordering a screenshare induces on the shared connection BEFORE the stateful decoder; release is purely count-based — the moq group read is not cancel-safe mid-object, so no recv timeout is ever raced against it.
audio_tx 🔒
Outbound audio: the mic capture loop + Opus publish, and the screen APP-AUDIO leg (its !Send capture thread + publish task). Encode + publish run on the RT audio runtime (“audio is sacred”: a screenshare encode can’t stall the mic), and audio frames publish via the ordered-substream path (count-rolled groups) so successive Opus frames ride one in-order QUIC stream — never one stream per frame.
camera_tx 🔒
Outbound camera: the !Send capture+encode thread (device hot-swap, self-preview tee, adaptive encode) and the async publish task running the 1 s BWE adaptation tick. The capture side reads its decisions off adapt::AdaptShared; the codec is negotiated against the roster and re-driven mid-call by the re-negotiator. Self-preview keeps running even while the ladder pauses video (AudioOnly), so the local tile never freezes.
control 🔒
The control-plane IO shell: ControlClient drives the /control WebSocket (applies each ServerMessage to the SessionView reducer, publishes CallView snapshots, exposes the outbound intent queue, and fans the keyframe/feedback/quality signals out to the per-track listeners), plus dial_relay — the iroh endpoint + QUIC connection the MoQ session rides. Invariant: the returned iroh Endpoint must outlive the connection — the SFU is relay-only reachable, so dropping the endpoint wedges every stream after the handshake.
policy 🔒
Pure, unit-tested call policy — the sans-IO decision layer. Inputs (sizes, instants, samples, roster caps) in, decisions out: codec negotiation + the mid-call CodecSwitch, spatial-layer LayerHysteresis, RestartPacer, ResizeWatch, mic route warmup / dead-silence detection, per-stream playback gain, and the audio gap/cadence math. Nothing here touches sockets, clocks, or devices, so every policy is pinned by unit tests without a live source.
reconnect 🔒
Reconnect policy + the retry loop: the pure exponential Backoff with bounded jitter, the fatal-vs-retryable classify verdict over HoundError, and reconnect_loop — which re-runs establish_live until success, a fatal error, or cancel. Invariant: only an auth/permission outcome is fatal; everything transient retries FOREVER (each attempt re-mints the relay token via voice/join, so a stale token heals on retry).
screen_tx 🔒
Outbound screen share: open → !Send capture/encode thread → publish, plus the LOW simulcast rung and the classified exit protocol back to the supervisor (ScreenCaptureExit: a clean end vs a held resize vs a transient fault — a share must NOT end on a resize or a glitch). Static content is re-emitted on a slow cadence for fresh subscribers, and an idle source is liveness-probed so a terminated window ends the share instead of freezing on its last frame.
session 🔒
The live call: CallSession (join/leave/commands + the cross-reconnect command seams), establish_live (one full control+MoQ session), and the auto-reconnect supervisor — which owns the live session, the backoff, and the screen-share state (so a share is republished against each fresh session). Invariant: recovery is a FULL re-establish (fresh PeerId) — the broken session’s tasks/threads are fully torn down before a rebuild, and only an auth/permission failure (or an explicit leave) ends the call.
simulcast 🔒
Demand-driven simulcast, producer side: LayerControl folds every receiver’s standing quality demand (QualityReport via spawn_quality_listener) into a [hound_media::SimulcastPlanner], driving the shared LOW-layer supervisor; spawn_low_video_publish drains the capture thread’s low encode onto its own moq track, announcing it on the FIRST frame only (dark until actually producing). Invariant: the source encode is never dropped — the low layer is purely additive, so any low-layer fault costs a thumbnail, never the full-quality source. That additive-only design is why the low layer runs unconditionally: a receiver only switches to the low track for a small/hidden tile it explicitly requested.
subscribe 🔒
The subscribe loop: reconcile remote tracks → decode → sink / mix bus. Diffs the SessionView roster into per-track pipelines (start_pipeline), picks source vs low layer per tile demand (debounced through the policy hysteresis), and self-heals: every pipeline carries a PipelineEndGuard whose Drop reports the death so the reconcile reaps + re-subscribes it (restart-paced). Invariant: audio subscribes with arrival-order drain and the smaller moq priority value — audio must strictly outrank video on the shared connection.
video_rx 🔒
Inbound video: the async pull half (receive-side keyframe gate; the nag task lives with the subscribe loop) feeds a blocking decode thread that owns the !Send VideoDecoder and rebuilds it whenever the sender’s per-packet codec byte changes — the sender’s choice drives decode. Decoded frames reach the sink surface-first (zero-copy GPU handle) with materialized I420 as the universal fallback; push_preview is the borrow-based capture-side twin the publish loops use for self-preview.

Structs§

CallParticipant
A control-plane peer with a known identity. The UI joins user_id against the display roster (the same one text channels use) for the name + avatar. Carries no name/avatar itself — identity lives in the roster, not the call plane.
CallSession
One live voice/video call. The supervisor task owns the live session(s) + every capture thread; this struct holds the cross-reconnect command seams + the supervisor handle. Dropping it (or CallSession::leave) cancels + tears down.
CallTrack
One published track, carried with the control-plane enums (not flattened to strings) so the binding compares them type-driven — t.kind == MediaKind::Video — with exhaustiveness on both sides. The enums are Copy + Eq (serde snake_case).
CallView
The call’s roster as the UI renders it: self id, peers, and the tracks (with kind/source/mute). Built from the control plane’s [SessionView] reducer — the same source of truth the media plane subscribes off of. Empty before Welcome.
DeviceListView
The available media devices grouped by kind — the data the Devices panel renders. A kind whose platform backend is absent simply comes back empty (a capability miss, not an error).
DeviceView
One enumerated media device for a picker row. FFI-flat: id is the opaque selection token to pass back to call_select_device for this kind (for audio it equals the cpal device name; for a camera the AVFoundation uniqueID), and is_default flags the host’s current default for that kind.
LevelsView
Speaking levels: the local mic level plus a per-peer map, each 0.0..=1.0 (perceptual, from RMS). The UI renders these as the ring/glow around a tile.
NullVideoSink
A no-op sink (tests / headless): decoded frames are discarded. The call still runs end to end (audio plays, state updates); only video display is dropped.
ScreenSourceView
A shareable screen source for the picker — a display, a window, or a camera streamed as a screen. UI-friendly + FFI-flat: the typed engine selection token is encoded into id (see encode_screen_source) so the UI round- trips an opaque string back into CallSession::screen_share_start.
ScreenThumbnail
A share-picker preview image: opaque RGBA8888, width * height * 4 bytes.

RenderSurfaceHandle

PLAYOUT_TARGET_US 🔒
Per-peer jitter-buffer depth floor the adaptive jitter grows from. 80 ms buys headroom so a brief fill-thread scheduling hiccup under heavy video load can’t drain the buffer before the next frame lands — defense in depth behind the RT audio runtime (which is what actually prevents starvation). The adaptive estimator still grows above this on a jittery link; the extra baseline latency is inaudible for conversation and buys a full Opus frame of slack.
SCREEN_AUDIO_BITRATE 🔒
App-audio Opus bitrate for a shared window’s audio. This is music/system audio, not a mic — encoded in OpusProfile::Music (general-audio) mode, so it needs a music-grade rate: speech-grade rates (~32 kbps) audibly crush music. 96 kbps mono is near-transparent for general audio.
SCREEN_AUDIO_TRACK 🔒
SCREEN_BITRATE 🔒
Screen publish bitrate seed — the initial encoder target before the BWE adaptation settles onto a ladder rung. Matches the 1080p30 SCREEN rung (~2.4 Mbps, see screen_bitrate_bps) — a screen-content target, NOT a camera-video one. A camera-grade seed (several Mbps) would saturate the shared QUIC connection before BWE reacts and starve audio/other video, while screen content (mostly static, text) looks crisp in H.265 at this rate. The BWE climbs from here if the link genuinely has headroom.
SCREEN_FPS 🔒
SCREEN_IDLE_POLL 🔒
Damage-driven screen capture: bounded wait for a frame (a static screen produces none) so the loop can re-emit a cached frame for new subscribers.
SCREEN_KEYFRAME_CADENCE 🔒
Screen keyframe cadence: IDRs on demand only (a subscriber’s RequestKeyframe — sent at subscribe and re-sent by the receiver’s awaiting-keyframe nag — or a PLI), plus a slow safety net. Screen content is mostly static, and a periodic IDR of an unchanged 1080p frame is pure redundancy — a short fixed cadence would dominate the data cost of a static share (hundreds of kbps for zero new information). max_gop_frames = 900: one GOP = one moq group, and moq-net evicts a group past 1024 objects (a fatal CacheFull for the drain) — 900 bounds the GOP safely under that cap even at sustained 60 fps (≈15 s), while the 60 s safety interval governs low-fps/static content (tiny groups).

Traits§

VideoSink
The decode→render seam. The orchestration decodes remote video into tightly packed I420 and hands it here, keyed by a stream key ("self", "cam:{peer}", "screen:{peer}"). The FRB layer supplies an impl whose body calls video_render::push_frame(key, w, h, i420) — so this crate stays free of any texture/FFI dependency. The sink must be cheap and non-blocking (drop-oldest); it is called on a decode task, never on a real-time audio thread.

Functions§

audio_runtime 🔒
The dedicated runtime for the audio RECEIVE/fill path (per-peer Opus decode → mix jitter buffer). Isolated from call_runtime and real-time promoted (each worker runs [promote_current_thread_for_audio_fill] on start) so a CPU-heavy screen-share decode/scale/render load can never deschedule the fill — the RT mix bus would otherwise find the jitter buffer dry at its 10 ms pull tick and the call audibly stutters (mix_starve). The playout (mix) side is already RT; this extends the same guarantee to the producer side. “Audio is sacred”: video can freeze, voice cannot.

Type Aliases§

PeerIdentities
Shared peer_id (u32) → user_id (Uuid) map. Seeded at join from the REST VoiceJoin.peers (+ self), then kept current by the realtime voice events (VoiceJoined/VoiceState/VoiceLeft). An ArcSwap so the call’s reader task reads a cheap snapshot while the realtime reducer task swaps in updates — no lock across the media plane. Cleared on call join/leave so identities never leak.
SharedScreenQuality 🔒
The sharer’s live screen-quality preference (motion↔legibility + a resolution/fps cap), shared between CallSession::set_screen_quality (the writer) and the running screen adaptation tick (the reader, which set_screen_qualitys its ladder when it changes). An ArcSwap so the command writes lock-free while the tick reads a cheap snapshot. Lives on ConnectionParams so it survives a reconnect (the republished share re-seeds its ladder from it).
