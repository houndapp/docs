The !Send screen capture+encode thread: open the engine ScreenShare (+ maybe ScreenAudio) and the H.264 encoder HERE (so neither crosses a thread boundary), report readiness (handing back the audio leg), then pull → self-preview tee → encode → forward. A static source produces no frames, so on an idle poll the last frame is re-emitted (timestamp advanced) at the static-refresh cadence so a fresh subscriber isn’t stuck on a black tile.

The loop classifies its exit and reports it via notify:

  • a window CLOSE / OS “stop sharing” / source gone (next_frame errs cleanly) → ScreenCaptureExit::Ended;
  • a window RESIZE that holds past SCREEN_RESIZE_DEBOUNCE (the captured content size departs its baseline and stays) → ScreenCaptureExit::Resized;
  • a transient next_frame error/panic → ScreenCaptureExit::Retry (after a brief SCREEN_RETRY_BACKOFF).

NB the engine exposes ONE next_frame_timeout result for both “ended” and a transient error (a CaptureError), so we conservatively treat an error as a transient Retry: the supervisor’s restart re-opens cleanly, and a re-open that then fails (the source is truly gone) degrades to a clean Ended there. notify is NOT fired when stop was latched (a deliberate stop/leave/reconnect — the supervisor already owns that transition).
