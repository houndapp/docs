Module video_rx 

Source
Expand description

Inbound video: the async pull half (receive-side keyframe gate; the nag task lives with the subscribe loop) feeds a blocking decode thread that owns the !Send VideoDecoder and rebuilds it whenever the sender’s per-packet codec byte changes — the sender’s choice drives decode. Decoded frames reach the sink surface-first (zero-copy GPU handle) with materialized I420 as the universal fallback; push_preview is the borrow-based capture-side twin the publish loops use for self-preview.

Constants§

FIRST_KEYFRAME_DEADLINE 🔒
How long a fresh video subscription may sit with NO keyframe before the pipeline ends itself so the reconcile re-subscribes (restart-ladder paced). Past the nag window (5 × 2 s): a producer that ignored every re-request is wedged or its broadcast is a zombie from a same-path replacement, and only a fresh subscribe — a new announce cursor + control-plane Subscribe — heals those; waiting on the 60 s safety-net IDR is a frozen tile.
RETRY_AFTER 🔒
How long the decode loop withholds retrying platform-decoder construction for a codec that just failed to build. Without this, the per-packet wire codec byte is sender-controlled — a peer sending AV1 to a box with no AV1 decode makes every single frame re-attempt Engine::new().video_decoder(..) (a VideoToolbox/VAAPI session setup) and warn!, forever. That is a remotely triggerable retry storm (CPU burn + log spam) for the lifetime of the subscription.
VIDEO_STALL 🔒
How long the video pull tolerates delivering NOTHING before treating the subscription as stalled. Above the static-share re-emit cadence (SCREEN_STATIC_REFRESH, 1 s) so a healthy static share never trips it; a mid-GOP loss stall or a silent producer does within seconds instead of a retransmit tail.

Functions§

push_preview 🔒
Push one CAPTURED frame to the render sink for a self-preview tile, preferring the ZERO-COPY surface path: a surface-backed frame (SCK / AVFoundation NV12 IOSurface) is handed to the renderer to sample directly — no NV12→I420 CPU conversion, no ~1.4–3 MB per-frame alloc+copy, no re-upload (~90 MB/s of conversion at a 1080p 30 fps share). Falls back to materialized I420 for non-surface frames (scaled camera frames, other platforms) or a sink without a surface renderer. Borrows the frame — the caller still owns it for encode. (The REMOTE decode path keeps its own ownership-optimized version of this pattern: it consumes the frame, so its no-surface case can MOVE the bytes instead of copying.)
video_decode_loop 🔒
The blocking video decode loop: owns the !Send VideoDecoder, rebuilds it when the sender’s codec changes (the sender’s choice drives decode), and pushes each decoded I420 to the sink.
video_pipeline 🔒
Video decode pipeline. The MoQ subscriber pull is async (must run on the runtime), but the engine VideoDecoder (VideoToolbox/dav1d) is !Send, so it CANNOT live across an .await. The async half therefore only PULLS frames — applying the receive-side keyframe gate so we never feed a delta before the first keyframe — and forwards them to a dedicated blocking decode thread that owns the !Send decoder and pushes decoded I420 to the sink keyed by peer.
