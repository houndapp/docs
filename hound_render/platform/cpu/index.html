Non-macOS (Windows / Linux) video display path — the CPU pixel-buffer analogue of the macOS platform::apple::render zero-copy GPU module, exposing the IDENTICAL push_frame / attach / detach API so the crate-root facade calls one path on every platform.

Where macOS wraps a CVPixelBuffer’s IOSurface as a wgpu::Texture and renders the I420 with a Metal shader, here the native runner plugin (Windows flutter::PixelBufferTexture, Linux FlPixelBufferTexture) allocates a plain RGBA8888 buffer and hands us its raw pointer via attach. A per-stream render thread takes the LATEST pushed I420 from the mailbox (drop-oldest) and converts + scales it into that buffer at ~30 fps via crate::yuv; the plugin independently pings its texture “frame available” on a timer so Flutter re-pulls the bytes. Single-buffer (Rust writes while Flutter reads) — a read can observe a partially written frame, the same accepted tradeoff as the macOS path.

§Buffer lifetime / safety

attach receives a raw *mut u8 to the plugin’s w×h×4 RGBA buffer plus its w/h (so this side knows exactly how many bytes it may write). The Dart VideoTextureHandle calls detachVideoTexture (which stops + JOINS this thread) BEFORE destroyTexture frees the buffer, so the buffer always outlives the thread that writes it — the same ordering contract the macOS module relies on for its CVPixelBuffer retain.

§Texture size matches the decoded frame

The buffer is sized to the DECODED frame’s w×h, passed through attach from the createTexture(width,height) the Dart VideoTextureHandle requests and the plugin allocates. When the decoded resolution changes (a simulcast source↔low switch, a shared-window resize) Dart recreates the texture at the new size and re-attaches, so the conversion target tracks the source aspect with no crop or stretch. A source frame whose size still differs from the buffer (the brief window before a recreate lands) is nearest-neighbor scaled to fit; Flutter’s FittedBox then applies the tile fit (cover/contain), exactly as on macOS.

Functions§

attach
Bind stream_key to the plugin’s w×h RGBA buffer at buffer_ptr and start converting the latest pushed frame into it. Replaces any existing stream under the same key (idempotent re-attach — used both for a fresh tile and to rebind after a decoded-resolution change recreates the buffer at a new size). Returns an error string (mapped to a Dart exception by the FRB layer) on a null pointer.
detach
Stop + join stream_key’s render thread (no-op if absent). After this returns the thread no longer touches the buffer, so the caller may free it. The mailbox is left intact (a producer may still be pushing / a re-attach is coming).

mailboxes 🔒
push_frame
Push the latest decoded I420 frame for stream_key (drop-oldest). Cheap: one lock plus a Vec move; never blocks the producer. Identical contract to the macOS module (the mailbox lives independently of the render thread, so a producer can push before attach / across re-attach).
push_surface
No surface renderer on this platform: always declines, so the caller falls back to push_frame with materialized I420 (no cfg at the call site).
render_loop 🔒
Render-thread body: until stop, take the latest pushed frame and convert+scale it into the plugin’s RGBA buffer at ~30 fps. Until the first frame arrives the buffer is cleared to opaque black once (so a not-yet-producing stream shows black, not garbage). Drop-oldest: a slow consumer just skips stale frames.
