§Per-platform renderers

The concrete implementations behind the crate-root facade, selected at compile time per target — the same shape as hound-engine’s platform module. The facade calls through the imp alias and never carries a #[cfg]; platform-only public surface goes through crate::os, not here.

  • [apple] — the shared wgpu(Metal) render loop drawing into IOSurface-backed CVPixelBuffers (zero-copy NV12 + uploaded I420), plus the dirty-set drain for the Swift display link.
  • [windows] — the D3D11 path: NV12 frames (zero-copy engine textures, or uploaded CPU I420) blitted into the plugin’s shared BGRA texture by the fixed-function video processor, plus the dirty-set drain for the plugin’s ticker.
  • cpu — the Linux path: per-stream threads converting I420 into the plugin’s RGBA pixel buffer.

All expose the same four entry points (attach / detach / push_frame / push_surface); a platform without a surface renderer answers push_surface with false so callers need no fallback cfg. Platform deps live in this crate’s [target.'cfg(...)'.dependencies] tables.

cpu 🔒
Non-macOS (Windows / Linux) video display path — the CPU pixel-buffer analogue of the macOS platform::apple::render zero-copy GPU module, exposing the IDENTICAL push_frame / attach / detach API so the crate-root facade calls one path on every platform.
