§hound-render — decoded video frames → Flutter-registered native textures

The display side of the video pipeline: the call orchestration decodes remote video (and the capture path previews local video) and pushes each frame here by stream_key ("self", "cam:{peer}", "screen:{peer}"); the Flutter side registers a native texture per tile and binds it to the same key via attach. This crate connects the two with one platform-uniform API — the caller (the FRB layer) never carries a #[cfg(target_os = …)].

§Render model

  • Mailboxes — each stream key owns a drop-oldest one-frame mailbox. push_frame/push_surface overwrite it and never block the producer; a slow renderer skips stale frames. Mailboxes live independently of the attached texture, so a producer can push before attach and across re-attach.
  • Shared render loop (macOS) — ONE render thread draws every stream’s latest frame into its target IOSurface via wgpu(Metal), batching all draws into one command submit per wake, and marks each drawn Flutter texture_id “dirty”. The Swift shared CVDisplayLink drains the dirty set through the C ABI export [hound_video_take_dirty] and re-pulls exactly the tiles that got a new frame — repaint is frame-arrival-driven, not vsync-driven.
  • D3D11 path (Windows) — NV12 frames (the engine’s zero-copy decode/ capture textures, or uploaded CPU I420) are blitted into the plugin’s shared BGRA texture by the fixed-function video processor, synchronously in the push; the plugin’s ticker drains the same dirty-set C ABI as macOS.
  • CPU path (Linux) — a per-stream render thread converts the latest I420 into the plugin’s RGBA pixel buffer (crate::yuv) at ~30 fps; the plugin pings its texture on its own timer.

§Platform layout

The concrete renderers live under platform/ and are selected at compile time (platform); macOS-only primitives that callers may legitimately need (the IOSurface ⇄ wgpu interop) are exposed as an explicit escape hatch under [os::macos], never as cfg’d root exports.

Functions§

attach
Bind stream_key to the native buffer the Flutter plugin registered as texture_id, and start rendering pushed frames into it. Idempotent: an attach under an existing key replaces the previous binding.
detach
Unbind stream_key (no-op if absent). When this returns, no render work — CPU thread or in-flight GPU command — still touches the native buffer, so the caller may free/destroy the texture immediately. The stream’s mailbox survives (a producer may still be pushing; a re-attach may be coming).
push_frame
Push the latest decoded I420 frame for stream_key (drop-oldest). i420 is tightly packed w*h + 2*(w/2 * h/2) bytes, stride == width, even extents. Cheap and non-blocking on every platform — safe to call from a decode/capture thread; never from a real-time audio thread.
push_surface
Push a decoded frame that is still GPU-resident, identified by a platform-defined surface pointer — on macOS an IOSurface-backed NV12 CVPixelBufferRef (a retain is taken synchronously), on Windows a borrowed NV12 ID3D11Texture2D (consumed synchronously: the blit into the Flutter target happens inside this call). Valid for the duration of the call. Returns true if the platform renderer took the surface; false where there is no surface renderer, the tile is not attached yet (Windows), or zero-copy is disabled via HOUND_ZEROCOPY_VIDEO=0 — the caller then falls back to push_frame with materialized I420.
