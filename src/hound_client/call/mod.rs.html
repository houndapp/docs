1//! Voice/video **call orchestration** — the UI-free engine that drives a live
2//! call from `HoundClient`.
3//!
4//! It parallels the legacy Tauri orchestration in `src/voice/session/*` but
5//! reuses the engine/transport/control crates directly and stays free of any
6//! UI/FFI machinery (no Tauri `Channel`/`AppHandle`/`emit`). The decode→render
7//! seam is a [`VideoSink`] trait the FRB layer supplies (its closure calls
8//! `video_render::push_frame(...)`); decoded frames reach the right tile keyed by
9//! peer without this crate ever knowing about textures or Dart.
10//!
11//! ## What a join wires up
12//!
13//! ```text
14//! REST voice/join  ──► relay_url + relay_token + sfu_node_id + peers
15//!        │
16//!        ├─► control WS (hound-control)  ──► SessionView reducer ──► watch<CallView>
17//!        │
18//!        └─► dial iroh ──► MoqRelaySession ─┬─► publish mic (Opus)
19//!                                           ├─► publish camera (H.264) [if enabled]
20//!                                           └─► subscribe loop: for each remote track
21//!                                                 video → decode → VideoSink (push_frame)
22//!                                                 audio → decode → mix bus (PlayoutBus)
23//! ```
24//!
25//! ## Module tree
26//!
27//! This module is the shared hub — the public seams ([`VideoSink`],
28//! [`RenderSurfaceHandle`]), the picker enumerations, the observable snapshot
29//! types, the dedicated runtimes, and the cross-module tuning constants. The
30//! orchestration lives in the submodules:
31//!
32//! * [`session`] — [`CallSession`] (join/leave/commands), `establish_live`, the
33//!   auto-reconnect supervisor + screen-share command handling.
34//! * [`reconnect`] — the pure backoff/classification policy + the retry loop.
35//! * [`control`] — the `/control` WebSocket IO shell + the iroh relay dial.
36//! * [`policy`] — the pure, unit-tested decision policies (codec switch, layer
37//!   hysteresis, restart pacing, resize debounce, gain, gap math…).
38//! * [`adapt`] — sender-side BWE/congestion adaptation glue ([`adapt::AdaptShared`],
39//!   the 1 s tick, the keyframe/feedback listeners, the shared encode core).
40//! * [`simulcast`] — demand-driven simulcast (the layer planner + low publish).
41//! * [`audio_tx`] / [`camera_tx`] / [`screen_tx`] — the publish paths.
42//! * [`subscribe`] — the roster-diffing subscribe loop + per-track pipelines.
43//! * [`audio_rx`] / [`video_rx`] — the decode pipelines.
44//!
45//! ## Observable state
46//!
47//! Three `watch` channels, driven the same way as the realtime→reducer→watch task
48//! in [`crate::app`]: [`CallView`] (self id, peers, tracks) is derived from the
49//! control plane's [`SessionView`]; [`ConnectionState`] reflects the control WS +
50//! reconnect attempts; [`LevelsView`] carries mic + per-peer speaking levels.
51//!
52//! ## Auto-reconnect supervisor
53//!
54//! A live call heals a dropped control WebSocket or MoQ/iroh connection **without
55//! the user re-joining** — "nothing drops a call except total network loss."
56//!
57//! * [`reconnect::Backoff`] + [`reconnect::classify`] — the pure, headless policy
58//!   (exp-backoff+jitter; retryable-vs-fatal over [`HoundError`]). Only an
59//!   auth/permission outcome (`Http 401/403` or `Auth`) is fatal; everything
60//!   transient retries FOREVER.
61//! * [`session::LiveSession`] — the per-establish layers (relay/conn/control/tasks).
62//!   Its `monitor` selects over the control-WS-closed signal and the iroh
63//!   `Connection::closed()`, naming which layer dropped.
64//! * [`session::ConnectionParams`] — the Arc-cloneable cross-reconnect state.
65//!   `establish_live` builds one `LiveSession` (re-minting the relay token via
66//!   `voice/join` each time, so no standalone token-refresh task is needed).
67//! * [`session::spawn_supervisor`] — owns `live` + a `Backoff` + the screen-share
68//!   state; monitors, tears down on failure, and rebuilds with backoff. Recovery is
69//!   a FULL re-establish (fresh `PeerId`); the reactive subscribe loop re-diffs the
70//!   roster for free and re-publish is a fresh mic/cam announce.

133// ─── Dedicated call runtime ─────────────────────────────────────────────────────
134
135/// The runtime every call task is spawned onto.
136///
137/// flutter_rust_bridge builds its async runtime with `tokio::runtime::Runtime::new()`
138/// — one worker per core. The call's steady-state tasks (mic publish, per-track
139/// decode, the moq session driver, supervisor/heartbeat) are light and
140/// latency-bound, not CPU-bound, so spreading them over a per-core pool only adds
141/// park/unpark + work-steal coordination that dwarfs the actual Opus/moq work. A
142/// small pool carries the same load with a fraction of that overhead.
143///
144/// All `spawn` sites in this module target this runtime, and `CallSession::join` is
145/// driven on it (so moq-net's internally-spawned session driver lands here too rather
146/// than on the FRB pool). `'static` so it is never dropped — dropping a runtime from
147/// within an async context panics, and FRB commands call us from async context.
148pub(crate) fn call_runtime() -> &'static tokio::runtime::Runtime {
149    static RT: std::sync::OnceLock<tokio::runtime::Runtime> = std::sync::OnceLock::new();
150    RT.get_or_init(|| {
151        tokio::runtime::Builder::new_multi_thread()
152            // 3 workers: 2 carry the steady control + video load with headroom; the
153            // third is margin so a high-bitrate screenshare's moq frame delivery —
154            // which can saturate a worker — cannot head-of-line-block the others on
155            // the same pool. NB: audio fill does not run here — it has its own RT
156            // runtime (`audio_runtime`) so video CPU can't starve voice ("audio is
157            // sacred").
158            .worker_threads(3)
159            .enable_all()
160            .thread_name("hound-call")
161            .build()
162            .expect("build dedicated call runtime")
163    })
164}
165
166/// The dedicated runtime for the audio RECEIVE/fill path (per-peer Opus decode → mix
167/// jitter buffer). Isolated from [`call_runtime`] and **real-time promoted** (each
168/// worker runs [`promote_current_thread_for_audio_fill`] on start) so a CPU-heavy
169/// screen-share decode/scale/render load can never deschedule the fill — the RT mix
170/// bus would otherwise find the jitter buffer dry at its 10 ms pull tick and the call
171/// audibly stutters (`mix_starve`). The playout (mix) side is already RT; this extends
172/// the same guarantee to the producer side. "Audio is sacred": video can freeze, voice
173/// cannot.
174///
175/// Two workers: the decode of a 20 ms Opus frame is sub-millisecond and mostly parked
176/// on the network, so a couple of RT threads comfortably carry many peers without any
177/// busy-loop that would starve the system. (Caveat: the shared moq/QUIC demux driver
178/// still runs on `call_runtime`; the jitter buffer absorbs brief demux delay, and RT
179/// decode here removes the dominant — CPU-bound — starvation source.)
180pub(crate) fn audio_runtime() -> &'static tokio::runtime::Runtime {
181    static RT: std::sync::OnceLock<tokio::runtime::Runtime> = std::sync::OnceLock::new();
182    RT.get_or_init(|| {
183        tokio::runtime::Builder::new_multi_thread()
184            .worker_threads(2)
185            .enable_all()
186            .thread_name("hound-audio-rx")
187            .on_thread_start(hound_engine::api::promote_current_thread_for_audio_fill)
188            .build()
189            .expect("build dedicated audio runtime")
190    })
191}

193// ─── Public seam: the decode → render sink (UI-free) ────────────────────────────
194
195/// The decode→render seam. The orchestration decodes remote video into tightly
196/// packed I420 and hands it here, keyed by a stream key (`"self"`, `"cam:{peer}"`,
197/// `"screen:{peer}"`). The FRB layer supplies an impl whose body calls
198/// `video_render::push_frame(key, w, h, i420)` — so this crate stays free of any
199/// texture/FFI dependency. The sink must be cheap and non-blocking (drop-oldest);
200/// it is called on a decode task, never on a real-time audio thread.
201pub trait VideoSink: Send + Sync + 'static {
202    /// Hand one decoded I420 frame to the renderer for `stream_key`. `i420` is
203    /// tightly packed: `w*h + 2*(w/2 * h/2)` bytes.
204    fn push_frame(&self, stream_key: &str, w: u32, h: u32, i420: Vec<u8>);
205
206    /// Zero-copy fast path: hand one decoded frame that is still GPU-resident
207    /// ([`RenderSurfaceHandle`]) so the renderer can sample it without a CPU
208    /// readback / colour conversion / re-upload. Returns `true` if the sink took
209    /// it. The default returns `false` ("I can't take a surface"), so the caller
210    /// falls back to [`Self::push_frame`] with materialized I420 — meaning software
211    /// decoders, non-GPU sinks (tests/headless), and platforms without a surface
212    /// renderer all keep working unchanged. The handle is valid only for the
213    /// duration of the call; an impl that keeps it must retain synchronously.
214    fn push_surface(&self, stream_key: &str, surface: RenderSurfaceHandle) -> bool {
215        let _ = (stream_key, surface);
216        false
217    }
218}
219
220/// A decoded frame still resident in a platform GPU buffer, handed to the renderer
221/// for a zero-copy sample (no NV12→I420 readback, no re-upload). One variant per
222/// GPU backend; a platform without a surface renderer never builds one (the
223/// [`VideoSink::push_surface`] default declines and the I420 path runs).
224///
225/// Designed once for all platforms: macOS/iOS today; Windows (D3D11 shared NV12)
226/// and Linux (dmabuf NV12) slot in as new variants behind the same seam.
227pub enum RenderSurfaceHandle {
228    /// macOS/iOS: an IOSurface-backed NV12 `CVPixelBuffer` (raw `CVPixelBufferRef`
229    /// as a pointer-sized int). The producer holds a +1 retain for the duration of
230    /// the `push_surface` call; the renderer wraps the two IOSurface planes (luma
231    /// R8, chroma RG8) as Metal textures on wgpu's device.
232    AppleCvPixelBuffer { ptr: usize, width: u32, height: u32 },
233    /// Windows: an NV12 `ID3D11Texture2D` on the engine's shared device (raw COM
234    /// pointer as a pointer-sized int). BORROWED — the frame's texture-pool lease
235    /// is what guards the texture against being rewritten, and that lease lives
236    /// on the frame, so the renderer must consume it synchronously inside the
237    /// `push_surface` call (it blits NV12→BGRA into the Flutter-shared target
238    /// there and then).
239    WindowsD3d11Nv12 { ptr: usize, width: u32, height: u32 },
240}
241
242impl RenderSurfaceHandle {
243    /// The platform surface handle for a GPU-resident frame, or `None` for byte
244    /// frames / platforms without a surface renderer (callers then take the
245    /// materialized-I420 [`VideoSink::push_frame`] fallback).
246    pub fn for_frame(frame: &hound_engine::types::RawFrame) -> Option<Self> {
247        let (width, height) = (frame.dims.width(), frame.dims.height());
248        let surface = frame.surface.as_ref()?;
249        if let Some(ptr) = surface.cv_pixel_buffer_ptr() {
250            return Some(Self::AppleCvPixelBuffer { ptr, width, height });
251        }
252        if let Some(ptr) = surface.d3d11_texture_ptr() {
253            return Some(Self::WindowsD3d11Nv12 { ptr, width, height });
254        }
255        None
256    }
257}

330/// Enumerate the shareable screen sources (displays + windows). UI-free + needs no
331/// active call, so it lives as a free function the facade/FRB call directly. Blocking
332/// (round-trips ScreenCaptureKit's async enumeration); a Screen-Recording TCC denial

334pub fn list_screen_sources() -> Result<Vec<ScreenSourceView>> {

340        .map(|s| ScreenSourceView {

342            title: s.title,
343            app_name: s.app_name,

345        })
346        .collect())
347}

376/// Encode the engine's typed selection token into the opaque `"{kind}:{id}"` string
377/// the UI round-trips. Inverse of [`parse_screen_source`].
378fn encode_screen_source(source: &hound_engine::types::ScreenSource) -> String {
379    use hound_engine::types::ScreenSource;
380    match source {
381        ScreenSource::Display(id) => format!("display:{}", id.as_str()),
382        ScreenSource::Window(id) => format!("window:{}", id.as_str()),
383        ScreenSource::Camera(id) => format!("camera:{}", id.as_str()),
384    }
385}
386
387/// Parse the opaque `"{kind}:{id}"` picker token back into the engine's typed
388/// [`ScreenSource`](hound_engine::types::ScreenSource). Inverse of
389/// [`encode_screen_source`].
390fn parse_screen_source(token: &str) -> Result<hound_engine::types::ScreenSource> {
391    use hound_engine::types::{CameraId, DisplayId, ScreenSource, WindowId};
392    let (kind, id) = token
393        .split_once(':')
394        .ok_or_else(|| HoundError::Media(format!("malformed screen source token {token:?}")))?;
395    match kind {
396        "display" => Ok(ScreenSource::Display(DisplayId::new(id))),
397        "window" => Ok(ScreenSource::Window(WindowId::new(id))),
398        "camera" => Ok(ScreenSource::Camera(CameraId::new(id))),
399        other => Err(HoundError::Media(format!(
400            "unknown screen source kind {other:?} (want display|window|camera)"
401        ))),
402    }
403}

498/// Join `peers` (the control-plane roster) against the `peer_id → user_id` map into
499/// a deterministic (sorted by peer_id) list of [`CallParticipant`]. Peers with no
500/// known identity are simply omitted (the UI falls back to "Peer N"). Shared by the
501/// call reader (control-plane changes) and the realtime reducer (identity changes).
502pub fn build_participants(peers: &[u32], map: &HashMap<u32, Uuid>) -> Vec<CallParticipant> {
503    let mut out: Vec<CallParticipant> = peers
504        .iter()
505        .filter_map(|&peer_id| {
506            map.get(&peer_id)
507                .map(|&user_id| CallParticipant { peer_id, user_id })
508        })
509        .collect();
510    out.sort_by_key(|p| p.peer_id);
511    out
512}

514/// One published track, carried with the control-plane enums (not flattened to
515/// strings) so the binding compares them type-driven — `t.kind == MediaKind::Video`
516/// — with exhaustiveness on both sides. The enums are `Copy + Eq` (serde snake_case).
517#[derive(Clone, Debug, PartialEq, Eq)]
518pub struct CallTrack {
519    pub id: u32,
520    pub peer: u32,
521    pub kind: MediaKind,
522    pub source: Source,
523    pub codec: CtlCodec,
524    pub muted: bool,
525}

527impl CallView {
528    /// Build from the control-plane view alone — participants empty (identities
529    /// unknown). Test-only; the live reader always uses [`from_view_with`].
530    #[cfg(test)]
531    fn from_view(view: &SessionView) -> Self {
532        Self::from_view_with(view, &HashMap::new())
533    }
534
535    /// Build from the control-plane view, joining each peer against the
536    /// `peer_id → user_id` map to populate [`CallView::participants`].
537    fn from_view_with(view: &SessionView, identities: &HashMap<u32, Uuid>) -> Self {
538        let peers: Vec<u32> = view.peers().map(|p| p.id.0).collect();
539        let participants = build_participants(&peers, identities);
540        Self {
541            self_id: view.self_id().map(|p| p.0),
542            peers,
543            tracks: view
544                .tracks()
545                .map(|t| CallTrack {
546                    id: t.id.0,
547                    peer: t.peer.0,
548                    kind: t.kind,
549                    source: t.source,
550                    codec: t.codec,
551                    muted: t.muted,
552                })
553                .collect(),
554            participants,
555        }
556    }
557}

559/// The connection state of the call's control plane (and, by extension, whether
560/// media can flow). Drives a UI banner. `attempt` is the reconnect attempt count.
561#[derive(Clone, Debug, Default, PartialEq, Eq)]
562pub enum ConnectionState {
563    /// Not in a call (initial / after `leave`).
564    #[default]

566    /// Dialing the control WS / relay for the first time.
567    Connecting,
568    /// Control WS up and a `Welcome` received — media flowing.

574}

588/// Per-peer jitter-buffer depth floor the adaptive jitter grows from. 80 ms buys
589/// headroom so a brief fill-thread scheduling hiccup under heavy video load can't
590/// drain the buffer before the next frame lands — defense in depth behind the RT
591/// audio runtime (which is what actually prevents starvation). The adaptive
592/// estimator still grows above this on a jittery link; the extra baseline latency
593/// is inaudible for conversation and buys a full Opus frame of slack.
594const PLAYOUT_TARGET_US: i64 = 80_000;
595/// Self-view / camera publish geometry. The engine scales native → this.
596const CAM_DIMS: Dimensions = Dimensions::new(1280, 720);
597const CAM_FPS: Fps = Fps::new(30);
598/// Camera publish bitrate ceiling.
599const CAM_BITRATE: Bitrate = Bitrate::kbps(1500);
600/// The LOW simulcast layer's encode geometry for the camera. Unlike the screen path
601/// (`LOW_SCREEN_LONG_EDGE`, which must preserve a variable source aspect), camera
602/// capture is always fixed at `CAM_DIMS` (16:9), so this is a plain half-scale
603/// constant rather than a derived function — a small/hidden camera tile pulls this
604/// instead of the full source.
605const LOW_CAM_DIMS: Dimensions = Dimensions::new(640, 360);
606/// Screen-share publish geometry: capture is native, scaled to this 1080p-class
607/// cap preserving aspect (text stays legible without 4K uplink cost).
608const SCREEN_MAX_W: u32 = 1920;
609const SCREEN_MAX_H: u32 = 1080;
610const SCREEN_FPS: Fps = Fps::new(30);
611/// Long-edge budget for the LOW simulcast layer — what a small/unfocused screenshare
612/// tile pulls instead of the full source. 640 px long edge = 640×360 for a 16:9
613/// share, ~8× fewer pixels than 1080p.
614const LOW_SCREEN_LONG_EDGE: u32 = 640;
615
616/// Camera keyframe cadence: a periodic IDR. Camera content changes wholesale
617/// (deltas are already large), so the periodic refresh is cheap insurance for
618/// loss recovery.
619const CAM_KEYFRAME_CADENCE: KeyframeCadence = KeyframeCadence::MOTION;
620/// Screen keyframe cadence: IDRs **on demand only** (a subscriber's
621/// `RequestKeyframe` — sent at subscribe and re-sent by the receiver's
622/// awaiting-keyframe nag — or a PLI), plus a slow safety net. Screen content is
623/// mostly static, and a periodic IDR of an unchanged 1080p frame is pure
624/// redundancy — a short fixed cadence would dominate the data cost of a static
625/// share (hundreds of kbps for zero new information).
626/// `max_gop_frames = 900`: one GOP = one moq group, and moq-net evicts a group
627/// past 1024 objects (a fatal `CacheFull` for the drain) — 900 bounds the GOP
628/// safely under that cap even at sustained 60 fps (≈15 s), while the 60 s
629/// safety interval governs low-fps/static content (tiny groups).
630const SCREEN_KEYFRAME_CADENCE: KeyframeCadence = KeyframeCadence::OnDemand {
631    safety_seconds: 60,
632    max_gop_frames: 900,
633};
634/// Screen publish bitrate seed — the initial encoder target before the BWE
635/// adaptation settles onto a ladder rung. Matches the 1080p30 SCREEN rung
636/// (~2.4 Mbps, see `screen_bitrate_bps`) — a screen-content target, NOT a
637/// camera-video one. A camera-grade seed (several Mbps) would saturate the shared
638/// QUIC connection before BWE reacts and starve audio/other video, while screen
639/// content (mostly static, text) looks crisp in H.265 at this rate. The BWE climbs
640/// from here if the link genuinely has headroom.
641const SCREEN_BITRATE: Bitrate = Bitrate::kbps(2500);
642/// App-audio Opus bitrate for a shared window's audio. This is music/system
643/// audio, not a mic — encoded in `OpusProfile::Music` (general-audio) mode, so it
644/// needs a music-grade rate: speech-grade rates (~32 kbps) audibly crush music.
645/// 96 kbps mono is near-transparent for general audio.
646const SCREEN_AUDIO_BITRATE: Bitrate = Bitrate::kbps(96);
647/// Damage-driven screen capture: bounded wait for a frame (a static screen
648/// produces none) so the loop can re-emit a cached frame for new subscribers.
649const SCREEN_IDLE_POLL: std::time::Duration = std::time::Duration::from_millis(100);
650/// Re-emit the last screen frame at least this often (timestamp advanced) so a
651/// static share still delivers pixels to a freshly-subscribed peer.
652const SCREEN_STATIC_REFRESH: std::time::Duration = std::time::Duration::from_secs(1);
653/// How long a changed window size must hold before the share restarts at the new
654/// geometry — one restart when a live drag-resize settles, not dozens.
655const SCREEN_RESIZE_DEBOUNCE: std::time::Duration = std::time::Duration::from_millis(700);
656/// Backoff the screen capture thread sleeps before signalling `Retry` after a
657/// transient capture error/panic — bounds the restart rate if the source keeps
658/// faulting immediately (the share restart itself paces it further).
659const SCREEN_RETRY_BACKOFF: std::time::Duration = std::time::Duration::from_millis(500);
660/// moq-net track names (one per source) under the per-peer broadcast path.
661const MIC_TRACK: &str = "mic";
662const CAM_TRACK: &str = "cam";
663const SCREEN_TRACK: &str = "screen";
664const SCREEN_AUDIO_TRACK: &str = "screen_audio";
665/// moq-net delivery priorities — **audio must strictly outrank video on the wire** so
666/// a high-bitrate screenshare can never delay/burst the Opus streams on the SHARED
667/// QUIC connection (the "audio is sacred" wire guarantee).
668///
669/// ⚠️ COUNTERINTUITIVE VALUES — they compensate for an inversion in moq-net 0.1.9.
670/// moq-net ranks tracks by *descending* `track` value into a vec and hands quinn the
671/// vec **index** (`lite/priority.rs:138`, `new_priority = idx`, 0 = top rank). But
672/// quinn schedules the **largest** i32 first (`quinn-proto streams: BinaryHeap` max-
673/// heap). So a *smaller* `track` value → higher vec index → larger quinn priority →
674/// **served first**. Audio therefore takes the SMALLER value. If moq-net ever corrects
675/// the mapping, this inverts and audio would drop to lowest priority — the
676/// `audio_strictly_outranks_video_on_the_wire` guard test (`subscribe.rs`) fails
677/// loudly if that regression lands, and `moq-net` is pinned `=0.1.9` against it. Only
678/// the relative order matters (the value is used as an ordinal rank), not the
679/// magnitude. Used on BOTH subscribe (relay→us downlink) and publish (us→relay
680/// uplink), so it governs every audio/video contention point.
681const AUDIO_PRIORITY: u8 = 2;
682const VIDEO_PRIORITY: u8 = 4;
683/// Heartbeat cadence (keeps the backend's voice-membership alive).
684const HEARTBEAT_INTERVAL: std::time::Duration = std::time::Duration::from_secs(15);
685/// Sender-side BWE/congestion adaptation tick cadence: one decision per second
686/// folds RTT + the worst receiver-reported loss into a target bitrate/rung the
687/// encode loop applies.
688const ADAPT_TICK_INTERVAL: std::time::Duration = std::time::Duration::from_secs(1);
