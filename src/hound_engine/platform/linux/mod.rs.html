1//! # The Linux platform backend (V4L2 camera)
2//!
3//! Camera capture on Linux via **V4L2** through the [`v4l`] crate, which is a
4//! safe Rust wrapper over the kernel's `videodev2` ioctl ABI (its default
5//! `v4l2`/`v4l2-sys` features bind the kernel UAPI directly — **no `libv4l`
6//! system library**, only the always-present `linux/videodev2.h` UAPI header).
7//! Because `v4l` already owns the unsafe ioctl boundary, this module needs **no
8//! raw-FFI sys wrapper** of its own (the apple backend has one because it talks
9//! to Obj-C frameworks directly; here the safe boundary is the `v4l` crate). If
10//! a future need for a raw ioctl appears (e.g. a control `v4l` doesn't expose),
11//! add a small `v4l2.rs` sys wrapper following the `apple` style — one
12//! `// SAFETY:` per block, every public fn safe.
13//!
14//! ## What this backend provides
15//! * [`camera::V4l2Camera`] — the [`crate::backend::VideoCapture`] implementation
16//!   (MMAP streaming, format negotiation, conversion to tightly-packed I420,
17//!   scale-to-encode-target), with the same construct-is-start / drop-is-stop
18//!   typestate as the AVFoundation capturer.
19//! * [`camera::enumerate_cameras`] — the Linux device listing (`/dev/video*`),
20//!   wired into the engine's device inventory.
21//!
22//! Mics/speakers already come from cpal (CoreAudio's Linux sibling, ALSA/Pulse)
23//! via `platform::generic` — a Linux desktop starts with working audio calls and
24//! this module adds the camera.
25//!
26//! ## Codecs on Linux (capability posture)
27//! The default is the portable **SW H.264 (openh264)** path, both ways — the
28//! codec ladder's universal floor, which always builds and interops with Apple /
29//! Windows peers. A hardware **VAAPI** backend (via `cros-libva` / `cros-codecs`,
30//! safe Rust over the system `libva.so`) is implemented behind the off-by-default
31//! **`linux-vaapi`** feature; when enabled it supersedes the floor:
32//! * [`video_codec::LinuxVideoEncoder`] — VAAPI HW **H.264** encode, SW openh264
33//!   fallback (no GPU / HW-open failure). HEVC **encode** is `Unsupported`:
34//!   cros-codecs 0.0.6 has no VAAPI HEVC encoder (an upstream gap, not a choice).
35//! * [`video_codec::LinuxVideoDecoder`] — VAAPI HW **H.264 + HEVC** decode, SW
36//!   openh264 fallback for H.264 (HEVC has no SW path — a headless box reports
37//!   HEVC decode unsupported). The decode-caps probe ([`va_display`]) advertises
38//!   HEVC only when the device exposes an HEVC `VLD` entrypoint.
39//!
40//! **Why the feature is off by default:** cros-libva bindgens the system libva
41//! headers at build time but its source only sets the struct fields that existed
42//! in the libva it was written for, so it fails to compile against newer system
43//! libva (e.g. the VP9 encode-param struct gained `seg_id_block_size` in libva
44//! ≳2.21). Until cros-libva tracks current libva (or we vendor a patched copy),
45//! the default Linux build uses the SW floor and the VAAPI path is opt-in for
46//! boxes whose libva matches. When `linux-vaapi` is on and a GPU is present, Linux
47//! does H.264 both ways in HW and decodes HEVC in HW (an Apple peer may send it);
48//! a headless box / HW-open failure falls back to SW H.264 and never loses video.
49//! AV1 stays unwired. NVENC/NVDEC, a VAAPI HEVC encoder, and a libva-current
50//! cros-libva are the documented follow-ups.
51//!
52//! ## Where the testable logic lives (codecs)
53//! Like the camera path, the platform-coupled VAAPI plumbing only runs on a Linux
54//! host with a GPU, so the pure pixel logic is factored into
55//! [`crate::platform::generic::convert`] and unit-tested on every host (incl. the
56//! macOS dev box): `i420_to_nv12` (VA surface upload), `nv12_bytes_to_i420` (the
57//! shared decode readback), and `crop_i420` (coded→visible). The cros-codecs
58//! encode/decode lifecycle itself is validated on-target (see the round-trip tests
59//! in `video_codec`, gated to a live VAAPI device).
60//!
61//! ## Where the testable logic lives
62//! The platform-coupled parts (opening `/dev/video*`, ioctls, mmap) can only run
63//! on a Linux host. So the parts most prone to bugs are factored OUT into pure,
64//! platform-agnostic functions in [`crate::platform::generic`] that are
65//! unit-tested on every host (including the macOS dev box this was written on):
66//! * format-preference ranking — [`crate::platform::generic::camera_format`]
67//!   (NV12 > YUYV > MJPEG, ignore-unrecognised);
68//! * pixel conversion — `yuyv_to_i420` / `nv12_bytes_to_i420` in
69//!   [`crate::platform::generic::convert`];
70//! * frame scaling — `scale_i420` (shared with the AVFoundation capturer).
71//!
72//! What remains Linux-on-target-only is the V4L2 plumbing itself (device open,
73//! `set_format`/`set_params`, mmap stream pump), validated on a Linux box / CI.

84/// `pw_init` once per process (reference-counted upstream; one init keeps
85/// teardown symmetric), shared by every PipeWire bridge thread.
86pub(crate) fn pw_init_once() {
87    static INIT: std::sync::Once = std::sync::Once::new();
88    INIT.call_once(pipewire::init);
89}
