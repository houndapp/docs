1//! # hound-engine — the platform media engine
2//!
3//! Everything device- and codec-shaped in hound lives here: microphone and
4//! speaker I/O, camera capture, video/audio codecs, echo cancellation, and the
5//! mixed playout bus. The layers above (client orchestration, FFI) consume
6//! **only** the high-level facade + [`types`] — they never see `cpal`, `objc2`,
7//! VideoToolbox, or any other platform type, and they cannot construct an
8//! invalid configuration (geometry, rates, device kinds) because the
9//! vocabulary in [`types`] validates at construction.
10//!
11//! ## Layering (top consumes bottom; nothing reaches upward)
12//!
13//! ```text
14//! ┌────────────────────────────────────────────────────────────┐
15//! │ api        — the high-level, platform-agnostic facade      │
16//! │              (Engine, Microphone, PlayoutBus, …)           │
17//! ├────────────────────────────────────────────────────────────┤
18//! │ audio      — portable engine services (APM/AEC, mix bus)   │
19//! │ codec      — portable codecs (Opus, …)                     │
20//! ├────────────────────────────────────────────────────────────┤
21//! │ backend    — the sealed platform SPI (traits)              │
22//! ├────────────────────────────────────────────────────────────┤
23//! │ platform   — the per-platform backends implementing it     │
24//! │              generic: cpal audio I/O, resample, enumerate  │
25//! ├────────────────────────────────────────────────────────────┤
26//! │ types/error — the zero-cost typed vocabulary + taxonomy    │
27//! └────────────────────────────────────────────────────────────┘
28//! ```
29//!
30//! The [`backend`] traits are **sealed**: only this crate can implement them,
31//! so the set of platform backends is closed at the crate boundary and the
32//! app can neither implement nor name a platform type. Capability differences
33//! between platforms are **queried, never panicked**: codec support via the
34//! runtime [`api::Engine::encode_caps`] / [`api::Engine::decode_caps`] probes,
35//! everything else via the typed [`error::CaptureError::Unsupported`] miss a
36//! constructor returns when a capability is absent.
37//!
38//! ## Adding a backend (the desktop/mobile parity path)
39//!
40//! Apple, Linux, and Windows desktop camera backends all exist; this is the
41//! recipe a *new* target (a future mobile/embedded backend) follows. The audio
42//! path (cpal: WASAPI / ALSA-Pulse / CoreAudio) already compiles and runs on
43//! every desktop target through `platform::generic` — a new desktop target
44//! starts with working audio calls. **Software H.264 is cross-platform too**:
45//! `platform::generic` carries SW H.264 (openh264, both ways) — the codec
46//! ladder's universal floor — so any desktop target interops with an Apple peer
47//! at H.264 the moment it has a camera. (There is no SW AV1/HEVC: a device with
48//! no HW video codec cannot run those in real time.) Full parity means adding
49//! the platform's camera + (optionally) a HW codec backend:
50//!
51//! 1. Create `src/platform/windows/` (or `linux/`), gated with
52//!    `#[cfg(target_os = "...")]`, and implement the [`backend`] traits for
53//!    the native APIs (Media Foundation capture + MFT H.264, or
54//!    V4L2/PipeWire + VAAPI). Construction-is-open: a backend value that
55//!    exists is running.
56//! 2. Wire it into the `cfg`-resolved constructors in [`api`] so the runtime
57//!    [`api::Engine::encode_caps`] / [`api::Engine::decode_caps`] probes report
58//!    the new HW codec honestly.
59//! 3. Keep platform deps in `[target.'cfg(...)'.dependencies]` of THIS
60//!    crate's Cargo.toml — the app crate stays platform-dep-free.
61//! 4. Nothing in the app changes: it already consumes the facade and treats
62//!    capability misses as graceful degradation (audio-only).
63//!
64//! ### Backend status
65//!
66//! * **Apple (macOS/iOS)** — AVFoundation camera + VideoToolbox HW
67//!   H.264 encode/decode + HW AV1 decode + ScreenCaptureKit screen/window
68//!   capture with per-app audio (`platform::apple`). Screen capture reports
69//!   `screen_capture = true` (macOS 12.3+) and `app_audio_capture = true`
70//!   (window-scoped audio, macOS 13+, runtime-gated at open).
71//! * **Linux** — camera: V4L2 via the `v4l` crate
72//!   (`platform::linux`), reporting `camera = true`. Video codecs default to the
73//!   portable **SW openh264** H.264 path (both ways). A HW **VAAPI** backend
74//!   (cros-libva / cros-codecs: HW H.264 encode and H.264 + HEVC decode) is
75//!   implemented behind the off-by-default **`linux-vaapi`** feature — off because
76//!   cros-libva only builds against the libva generation it was written for, so a
77//!   default build stays on the always-buildable SW floor; enable the feature on a
78//!   box whose libva matches. HEVC **encode** is unwired regardless (no VAAPI HEVC
79//!   encoder upstream); NVENC/NVDEC, a VAAPI HEVC encoder, and a cros-libva that
80//!   tracks current libva are the **documented follow-ups**.
81//! * **Windows** — camera: Media Foundation via the official `windows`
82//!   crate (`platform::windows`), an `IMFSourceReader` synchronous `ReadSample`
83//!   pull on our own thread (the same no-foreign-callback model as V4L2),
84//!   reporting `camera = true`. Video codecs: HW Media Foundation Transforms
85//!   (H.264/HEVC encode+decode, AV1 decode via the AV1 Video Extension) with
86//!   the SW openh264 H.264 floor as fallback (`platform::windows::mft` /
87//!   `mft_encoder`).
88//!
89//! ### Screen capture (the Apple-first SPI)
90//!
91//! Screen/window sharing is a separate device kind with its own SPI
92//! ([`backend::ScreenCapture`] + the optional [`backend::AppAudio`] leg, surfaced
93//! as [`api::Engine::screen_sources`] / [`api::Engine::open_screen_capture`]).
94//! Screen **video** is native on every desktop: ScreenCaptureKit on Apple
95//! (`platform::apple::screen`), Windows Graphics Capture driven directly on
96//! Windows (`platform::windows::wgc`, zero-copy D3D11), and an
97//! xdg-desktop-portal ScreenCast session + libpipewire driven directly on Linux
98//! (`platform::linux::portal_screen` — the compositor's dialog owns source
99//! selection, by Wayland design) — window or display video as
100//! [`types::RawFrame`]s. The **audio leg**: on Apple, per-app audio
101//! (ScreenCaptureKit, macOS 13+ *window* sources) — the source application's
102//! audio as 48 kHz mono [`types::PcmFrame`]s (the mic contract, so Opus/publish
103//! reuse is trivial); on Windows, WASAPI loopback (per-app process-tree for
104//! window shares, whole-system for display shares); on Linux, PipeWire loopback
105//! (`platform::linux::pw_audio` — whole-system default-sink monitor for display
106//! shares; window shares carry no auto audio because the portal never reveals
107//! the owning app, and [`os::linux`] exposes app-scoped capture by explicit
108//! identity instead).
109//!
110//! **TCC / Info.plist:** ScreenCaptureKit needs the macOS **Screen Recording**
111//! permission, which the system prompts for on first capture — there is **no**
112//! `Info.plist` key to add for it (unlike the camera/mic `NS*UsageDescription`
113//! strings). A denial surfaces as a typed
114//! [`error::CaptureError::PermissionDenied`] (kind `Screen`), so the UI can route
115//! the user to System Settings → Privacy & Security → Screen Recording rather
116//! than showing an empty source list as if nothing were shareable.
117//!
118//! ## Typestate: open is construct, drop is stop
119//!
120//! Device handles ([`api::Microphone`], …) have no separate `start()` —
121//! opening *is* constructing, so a handle that exists is capturing and
122//! `next_frame()` on an unstarted device is unrepresentable (that whole
123//! runtime error class is gone). Dropping a handle releases the device on the
124//! owning thread (the `!Send` contract: construct, drive, and drop on one
125//! dedicated thread).
