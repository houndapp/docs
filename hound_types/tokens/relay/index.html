§Relay token mint/verify

Every participant must present a relay token when connecting to the SFU. The token is issued by the backend and proves that:

  1. The user is authenticated (the backend checked their JWT).
  2. They are a member of the community that owns the voice channel.
  3. The token has not expired.

§Token format

{user_id}:{space_id}:{expiry_unix_secs}:{hex_mac}

Where hex_mac is HMAC-SHA256(secret, "{user_id}:{space_id}:{expiry}").

The backend and the SFU share the same RELAY_HMAC_SECRET environment variable. This secret is never sent over the wire — only the token is. This module is the ONE implementation of the format: the backend calls format_relay_token to mint, the SFU calls verify_relay_token to check, and both link against the exact same code instead of two hand-kept HMAC constructions that could silently drift.

§Why HMAC instead of JWT?

Relay tokens are short-lived (15 minutes) and carry only three fields. HMAC is simpler, has no library complexity, and is easy to verify from any language. The backend’s JWT infrastructure is reused for user auth; the SFU doesn’t need a second JWT dependency just for this.
