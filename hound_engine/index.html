§hound-engine — the platform media engine

Everything device- and codec-shaped in hound lives here: microphone and speaker I/O, camera capture, video/audio codecs, echo cancellation, and the mixed playout bus. The layers above (client orchestration, FFI) consume only the high-level facade + types — they never see cpal, objc2, VideoToolbox, or any other platform type, and they cannot construct an invalid configuration (geometry, rates, device kinds) because the vocabulary in types validates at construction.

§Layering (top consumes bottom; nothing reaches upward)

┌────────────────────────────────────────────────────────────┐
│ api        — the high-level, platform-agnostic facade      │
│              (Engine, Microphone, PlayoutBus, …)           │
├────────────────────────────────────────────────────────────┤
│ audio      — portable engine services (APM/AEC, mix bus)   │
│ codec      — portable codecs (Opus, …)                     │
├────────────────────────────────────────────────────────────┤
│ backend    — the sealed platform SPI (traits)              │
├────────────────────────────────────────────────────────────┤
│ platform   — the per-platform backends implementing it     │
│              generic: cpal audio I/O, resample, enumerate  │
├────────────────────────────────────────────────────────────┤
│ types/error — the zero-cost typed vocabulary + taxonomy    │
└────────────────────────────────────────────────────────────┘

The backend traits are sealed: only this crate can implement them, so the set of platform backends is closed at the crate boundary and the app can neither implement nor name a platform type. Capability differences between platforms are queried, never panicked: codec support via the runtime api::Engine::encode_caps / api::Engine::decode_caps probes, everything else via the typed error::CaptureError::Unsupported miss a constructor returns when a capability is absent.

§Adding a backend (the desktop/mobile parity path)

Apple, Linux, and Windows desktop camera backends all exist; this is the recipe a new target (a future mobile/embedded backend) follows. The audio path (cpal: WASAPI / ALSA-Pulse / CoreAudio) already compiles and runs on every desktop target through platform::generic — a new desktop target starts with working audio calls. Software H.264 is cross-platform too: platform::generic carries SW H.264 (openh264, both ways) — the codec ladder’s universal floor — so any desktop target interops with an Apple peer at H.264 the moment it has a camera. (There is no SW AV1/HEVC: a device with no HW video codec cannot run those in real time.) Full parity means adding the platform’s camera + (optionally) a HW codec backend:

  1. Create src/platform/windows/ (or linux/), gated with #[cfg(target_os = "...")], and implement the backend traits for the native APIs (Media Foundation capture + MFT H.264, or V4L2/PipeWire + VAAPI). Construction-is-open: a backend value that exists is running.
  2. Wire it into the cfg-resolved constructors in api so the runtime api::Engine::encode_caps / api::Engine::decode_caps probes report the new HW codec honestly.
  3. Keep platform deps in [target.'cfg(...)'.dependencies] of THIS crate’s Cargo.toml — the app crate stays platform-dep-free.
  4. Nothing in the app changes: it already consumes the facade and treats capability misses as graceful degradation (audio-only).

§Backend status

  • Apple (macOS/iOS) — AVFoundation camera + VideoToolbox HW H.264 encode/decode + HW AV1 decode + ScreenCaptureKit screen/window capture with per-app audio (platform::apple). Screen capture reports screen_capture = true (macOS 12.3+) and app_audio_capture = true (window-scoped audio, macOS 13+, runtime-gated at open).
  • Linux — camera: V4L2 via the v4l crate (platform::linux), reporting camera = true. Video codecs default to the portable SW openh264 H.264 path (both ways). A HW VAAPI backend (cros-libva / cros-codecs: HW H.264 encode and H.264 + HEVC decode) is implemented behind the off-by-default linux-vaapi feature — off because cros-libva only builds against the libva generation it was written for, so a default build stays on the always-buildable SW floor; enable the feature on a box whose libva matches. HEVC encode is unwired regardless (no VAAPI HEVC encoder upstream); NVENC/NVDEC, a VAAPI HEVC encoder, and a cros-libva that tracks current libva are the documented follow-ups.
  • Windows — camera: Media Foundation via the official windows crate (platform::windows), an IMFSourceReader synchronous ReadSample pull on our own thread (the same no-foreign-callback model as V4L2), reporting camera = true. Video codecs: HW Media Foundation Transforms (H.264/HEVC encode+decode, AV1 decode via the AV1 Video Extension) with the SW openh264 H.264 floor as fallback (platform::windows::mft / mft_encoder).

§Screen capture (the Apple-first SPI)

Screen/window sharing is a separate device kind with its own SPI (backend::ScreenCapture + the optional backend::AppAudio leg, surfaced as api::Engine::screen_sources / api::Engine::open_screen_capture). Screen video is native on every desktop: ScreenCaptureKit on Apple (platform::apple::screen), Windows Graphics Capture driven directly on Windows (platform::windows::wgc, zero-copy D3D11), and an xdg-desktop-portal ScreenCast session + libpipewire driven directly on Linux (platform::linux::portal_screen — the compositor’s dialog owns source selection, by Wayland design) — window or display video as types::RawFrames. The audio leg: on Apple, per-app audio (ScreenCaptureKit, macOS 13+ window sources) — the source application’s audio as 48 kHz mono types::PcmFrames (the mic contract, so Opus/publish reuse is trivial); on Windows, WASAPI loopback (per-app process-tree for window shares, whole-system for display shares); on Linux, PipeWire loopback (platform::linux::pw_audio — whole-system default-sink monitor for display shares; window shares carry no auto audio because the portal never reveals the owning app, and os::linux exposes app-scoped capture by explicit identity instead).

TCC / Info.plist: ScreenCaptureKit needs the macOS Screen Recording permission, which the system prompts for on first capture — there is no Info.plist key to add for it (unlike the camera/mic NS*UsageDescription strings). A denial surfaces as a typed error::CaptureError::PermissionDenied (kind Screen), so the UI can route the user to System Settings → Privacy & Security → Screen Recording rather than showing an empty source list as if nothing were shareable.

§Typestate: open is construct, drop is stop

Device handles (api::Microphone, …) have no separate start() — opening is constructing, so a handle that exists is capturing and next_frame() on an unstarted device is unrepresentable (that whole runtime error class is gone). Dropping a handle releases the device on the owning thread (the !Send contract: construct, drive, and drop on one dedicated thread).
