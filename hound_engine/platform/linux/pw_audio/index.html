PipeWire share-audio legs: default-sink-monitor loopback (display shares) and app-scoped stream capture (per-application audio, the venmic/OBS model).

§PipeWire loopback capture → screen-share audio (Linux)

The Linux audio leg of a screen share, scoped by [AudioScope]:

  • Display share → whole-system audio ([AudioScope::SystemMonitor]): one capture stream with stream.capture.sink, which WirePlumber connects to the default sink’s monitor — everything audible is on the screen being shared, the same contract as the Windows WASAPI render loopback. The stream follows default-sink changes (session-manager policy), so a device swap mid-share keeps audio flowing.
  • App-scoped audio ([AudioScope::App]): the per-application model every shipping Linux capturer uses (OBS’s app capture, Discord’s venmic): watch the PipeWire registry for Stream/Output/Audio nodes whose identity matches the target — process id within the target’s process tree (a browser renders audio in child processes), or application binary/name — and run one capture stream per matching node (target.object), mixed to a single leg. Nodes are picked up and released live as the app starts and stops streams. Wayland’s portal never reveals which app owns a shared window, so the caller supplies the identity explicitly; a scope is never silently widened to system audio (the privacy contract shared with the Windows process-loopback path). Our own process is always excluded — a share must not capture the call’s own playout as an echo.

Either way the leg delivers 48 kHz mono 20 ms [PcmFrame]s — the same contract as the microphone path and the macOS/Windows share-audio legs, so the Opus encode + publish plumbing is reused verbatim.

§Pipeline & threading

One dedicated bridge thread owns every PipeWire object (MainLoop, context, registry, streams — none are Send): it iterates the loop, and all stream process callbacks run in that iteration (streams are connected without RT_PROCESS, so delivery is marshalled off PipeWire’s RT data thread onto the loop — the engine never runs Rust in the graph’s RT path). Each stream asks the graph for 48 kHz mono S16 and PipeWire’s stream adapter does the rate/channel conversion, so the bridge only mixes and frames: per-node sample queues → [AppMixer] (aligned summing with a liveness window, so one stalled app can’t hold back the leg) → the shared [PcmFramer] 20 ms cadence → a drop-oldest [BoundedRing] popped by [AppAudio::next_frame].

An idle scope (no matching app streams; a silent suspended sink) produces no frames and the framer re-anchors on resume — the receiver’s screen_audio track is gappy during silence exactly like macOS/Windows.

§Where the testable logic lives

The graph plumbing needs a live PipeWire daemon (covered by the self-skipping integration tests below, which spawn real pw-play producers). The bug-prone parts are pure and unit-tested on every host: node identity matching ([node_matches]), process-tree expansion ([descendants], [parse_stat_ppid]), and the multi-stream mixer ([AppMixer]).

Structs§

AppMixer 🔒
Sums per-node sample queues into one mono stream with aligned pops: each Self::take emits n = min(len) over the live legs (delivered within LEG_LIVE_US), so concurrently playing apps stay sample-aligned, while a leg that stops delivering drops out of the minimum after the liveness window instead of muting the mix. With a single leg this is a pass-through (its whole queue drains every take). Saturating i32 sum, clamped to i16.
AppTarget 🔒
The identity an app-scoped capture matches Stream/Output/Audio nodes against. Any populated criterion admits a node (pid-tree first, then the name fallbacks — the OBS matching order); an empty target matches nothing.
ClientIdentity 🔒
The application.* identity of one PipeWire client (connection).
Leg 🔒
NodeIdentity 🔒
The identity properties of one registry node, as matched against an AppTarget — after merging in the owning client’s application.* props (see RegistryWatch).
PipeWireAppAudio 🔒
The running PipeWire share-audio leg. Holds the drop-oldest ring the bridge thread fills; AppAudio::next_frame pops it. Dropping it stops the bridge thread, which releases every PipeWire object on its owning thread.
RegistryWatch 🔒
Registry-side matching state for an app scope: the target identity, the (lazily re-expanded) process tree, and the client identity table. A stream node does not reliably carry application.* props itself — they live on the owning Client object, referenced by the node’s client.id — so Client globals are recorded and merged into each node’s identity (the OBS matching model). Runs inside the global callback, so it only records matches; stream creation happens in the loop body.
Shared 🔒
Mutable state shared between the loop body and the stream/registry callbacks (all on the bridge thread — RefCell, no locks).

AudioScope 🔒

Constants§

LEG_LIVE_US 🔒
A mixer leg is “live” if it delivered samples within this window; the aligned mix waits only for live legs, so a paused/stalled app releases the leg after at most this long instead of muting the other apps forever.
LEG_MAX_SAMPLES 🔒
Per-leg queue bound (1 s at 48 kHz). Overflow drops the OLDEST samples — the mix has fallen behind and stale audio is worse than a gap.
LOOP_TICK 🔒
Loop iteration timeout: the mix/frame cadence when no graph events arrive.
RING_FRAMES 🔒
Bounded PCM ring depth (20 ms frames) — ~200 ms, so a brief consumer stall never starves while the buffer stays bounded. Drop-oldest on overflow.
TREE_RESCAN_US 🔒
Re-expand the target’s process tree at most this often when unknown pids appear (browsers spawn audio child processes after the share starts).

Functions§

bridge_loop 🔒
Bridge-thread body: connect to the graph (reporting success/failure), pump the loop + mixer until stopped, then stop the ring so the consumer ends.
capture_format_pod 🔒
The one capture format asked of every stream: S16LE, 48 kHz, mono — the engine’s PCM contract. PipeWire’s stream adapter converts whatever the source node carries, so the bridge never resamples. Pure (a serialized EnumFormat pod), unit-tested below.
descendants 🔒
Expand root to itself plus every descendant, given (pid, ppid) pairs (a /proc snapshot). Pure — unit-tested below.
monotonic_us 🔒
Monotonic microseconds for callback-side throttling (never the capture clock — that belongs to frame timestamps only).
node_matches 🔒
Whether node belongs to the app-scoped capture. Only application output streams qualify; our own process never does (capturing the call’s own playout would feed remote audio back as an echo). Criteria are OR’d: pid in the target’s process tree, or a case-insensitive binary / application-name match — the same identity keys OBS and venmic scope by.
open_capture_stream 🔒
Build + connect one S16/48 kHz/mono capture stream. capture_sink selects the default-sink monitor; otherwise target_serial pins the stream to one node (target.object). Samples land in the shared mixer under leg.
parse_stat_ppid 🔒
The ppid field of a /proc/<pid>/stat line. The comm field (2nd) may itself contain spaces and parentheses, so parsing anchors on the LAST ): the fields after it are state ppid …. Pure — unit-tested below.
proc_parent_pairs 🔒
Snapshot (pid, ppid) for every process in /proc.
