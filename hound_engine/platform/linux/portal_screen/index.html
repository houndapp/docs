Portal + PipeWire screen/window video — the native Linux screen backend.

§Portal + PipeWire screen/window capture (Linux)

The Linux [ScreenCapture] backend, driven natively like its siblings (macOS ScreenCaptureKit, Windows WGC): an xdg-desktop-portal ScreenCast session ([super::portal]) selects the source and hands over a PipeWire video node; this module consumes that node directly with a libpipewire capture stream and delivers scaled, tightly-packed I420 [RawFrame]s stamped on the shared capture clock.

§Source model (Wayland’s rules)

Applications cannot enumerate windows/displays on Wayland — the compositor’s portal dialog owns selection. So [enumerate_sources] returns one entry per castable kind (screen / window, probed from the portal) and the real pick happens in the system dialog at open. The picked kind still drives the share-audio scope exactly like Windows: a display share carries default-sink (whole-system) audio, a window share carries no audio — the portal never reveals the window’s owning application, and widening a window share to system audio would leak other apps (the privacy rule shared with the WASAPI process-loopback path). App-scoped audio by explicit identity is crate::os::linux’s escape hatch.

§Pipeline & threading

The portal thread (zbus) lives in [super::portal]. Here, a bridge thread owns every PipeWire object: connect over the portal fd, one video capture stream on the cast node negotiated down to BGRx/BGRA over shared memory (no dmabuf modifiers are offered, so compositors fall back to memfd — the zero-copy dmabuf→VAAPI import is the documented follow-up), process → stride-aware BGRA→I420 ([convert::bgra_to_i420]) → aspect-fit scale to the encode target → drop-oldest [BoundedRing]. The consumer-side contract (next_frame_timeout Idle/Ended, damage-driven delivery, content_size resize signal, source_alive) matches the other platforms: PipeWire renegotiates the stream format when the shared window resizes (a new param_changed), which is surfaced through [ScreenCapture::content_size], and the compositor ending the share (user hits “stop”, window closes) destroys the node — observed as the stream erroring/unconnecting → a typed capture end, never a hang.

Functions§

bridge_loop 🔒
Bridge-thread body: connect the portal fd, negotiate BGRx/BGRA, report the encode target, then pump frames → I420 → scaled → ring until stopped or the cast ends. Every PipeWire object lives and dies on this thread.
capture_thumbnail 🔒
Thumbnails require capturing, and capturing prompts the portal dialog — a per-row prompt storm. Typed Unsupported; the picker degrades to title-only cards (its documented fallback).
convert_frame 🔒
One negotiated cast frame → tightly-packed I420 at the encode target. BGRx/BGRA take the alpha-ignoring converter; RGBx/RGBA go through the same converter after a channel swap. Stride-aware; None on a short buffer. Pure — unit-tested below.
enumerate_sources 🔒
List the castable source kinds as picker entries — one “Screen” and/or one “Window” row (the compositor’s dialog picks the concrete source at open). An absent portal surfaces CaptureError::Unsupported so the UI degrades the same way as a platform without a screen backend.
video_format_pod 🔒
The stream’s format offer: raw video, BGRx or BGRA (every compositor’s shared-memory screencast format), any size (default = the portal-reported source size), any framerate. No dmabuf modifier property is offered, which steers negotiation to shared memory — mappable buffers the CPU convert can read. Pure (a serialized EnumFormat pod), unit-tested below.
