§The Linux platform backend (V4L2 camera)

Camera capture on Linux via V4L2 through the [v4l] crate, which is a safe Rust wrapper over the kernel’s videodev2 ioctl ABI (its default v4l2/v4l2-sys features bind the kernel UAPI directly — no libv4l system library, only the always-present linux/videodev2.h UAPI header). Because v4l already owns the unsafe ioctl boundary, this module needs no raw-FFI sys wrapper of its own (the apple backend has one because it talks to Obj-C frameworks directly; here the safe boundary is the v4l crate). If a future need for a raw ioctl appears (e.g. a control v4l doesn’t expose), add a small v4l2.rs sys wrapper following the apple style — one // SAFETY: per block, every public fn safe.

§What this backend provides

  • camera::V4l2Camera — the crate::backend::VideoCapture implementation (MMAP streaming, format negotiation, conversion to tightly-packed I420, scale-to-encode-target), with the same construct-is-start / drop-is-stop typestate as the AVFoundation capturer.
  • camera::enumerate_cameras — the Linux device listing (/dev/video*), wired into the engine’s device inventory.

Mics/speakers already come from cpal (CoreAudio’s Linux sibling, ALSA/Pulse) via platform::generic — a Linux desktop starts with working audio calls and this module adds the camera.

§Codecs on Linux (capability posture)

The default is the portable SW H.264 (openh264) path, both ways — the codec ladder’s universal floor, which always builds and interops with Apple / Windows peers. A hardware VAAPI backend (via cros-libva / cros-codecs, safe Rust over the system libva.so) is implemented behind the off-by-default linux-vaapi feature; when enabled it supersedes the floor:

  • [video_codec::LinuxVideoEncoder] — VAAPI HW H.264 encode, SW openh264 fallback (no GPU / HW-open failure). HEVC encode is Unsupported: cros-codecs 0.0.6 has no VAAPI HEVC encoder (an upstream gap, not a choice).
  • [video_codec::LinuxVideoDecoder] — VAAPI HW H.264 + HEVC decode, SW openh264 fallback for H.264 (HEVC has no SW path — a headless box reports HEVC decode unsupported). The decode-caps probe ([va_display]) advertises HEVC only when the device exposes an HEVC VLD entrypoint.

Why the feature is off by default: cros-libva bindgens the system libva headers at build time but its source only sets the struct fields that existed in the libva it was written for, so it fails to compile against newer system libva (e.g. the VP9 encode-param struct gained seg_id_block_size in libva ≳2.21). Until cros-libva tracks current libva (or we vendor a patched copy), the default Linux build uses the SW floor and the VAAPI path is opt-in for boxes whose libva matches. When linux-vaapi is on and a GPU is present, Linux does H.264 both ways in HW and decodes HEVC in HW (an Apple peer may send it); a headless box / HW-open failure falls back to SW H.264 and never loses video. AV1 stays unwired. NVENC/NVDEC, a VAAPI HEVC encoder, and a libva-current cros-libva are the documented follow-ups.

§Where the testable logic lives (codecs)

Like the camera path, the platform-coupled VAAPI plumbing only runs on a Linux host with a GPU, so the pure pixel logic is factored into crate::platform::generic::convert and unit-tested on every host (incl. the macOS dev box): i420_to_nv12 (VA surface upload), nv12_bytes_to_i420 (the shared decode readback), and crop_i420 (coded→visible). The cros-codecs encode/decode lifecycle itself is validated on-target (see the round-trip tests in video_codec, gated to a live VAAPI device).

§Where the testable logic lives

The platform-coupled parts (opening /dev/video*, ioctls, mmap) can only run on a Linux host. So the parts most prone to bugs are factored OUT into pure, platform-agnostic functions in crate::platform::generic that are unit-tested on every host (including the macOS dev box this was written on):

  • format-preference ranking — crate::platform::generic::camera_format (NV12 > YUYV > MJPEG, ignore-unrecognised);
  • pixel conversion — yuyv_to_i420 / nv12_bytes_to_i420 in crate::platform::generic::convert;
  • frame scaling — scale_i420 (shared with the AVFoundation capturer).

What remains Linux-on-target-only is the V4L2 plumbing itself (device open, set_format/set_params, mmap stream pump), validated on a Linux box / CI.
