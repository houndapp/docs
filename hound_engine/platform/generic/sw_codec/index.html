SW video codec backends (openh264) — the non-Apple VideoEncoderBackend / VideoDecoderBackend. Only built where they’re the selected backend (non-Apple). Software video codec backends (openh264) — the non-Apple codec path.

These thin open(...)/caps() wrappers are the SW H.264 floor. They are the selected backend on Windows (encoder), on Linux without the linux-vaapi feature (both ways), and on the bare-fallback targets. Apple (always) and Linux with linux-vaapi reach the SW floor through their own HW-or-SW dispatch enums, holding an [Openh264Encoder]/[Openh264Decoder] directly — so in that one configuration these wrappers are compiled but unused; the cfg_attr keeps that build warning-clean without narrowing the module cfg.

H.264 only. A device with no HW video codec cannot run SW AV1/HEVC in real time, so those return Unsupported and the codec ladder’s universal floor is H.264. These wrap [Openh264Encoder] / [Openh264Decoder] behind the uniform open(codec, …) / caps() interface the platform facade dispatches through, so crate::api never branches on target.
