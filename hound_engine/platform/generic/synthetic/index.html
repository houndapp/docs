§Synthetic video source (camera-free test source)

A [VideoCapturer] that fabricates timestamped I420 frames at the requested resolution/fps without any camera. It lets the encode path be exercised and benchmarked (e.g. can this codec encode {1280×720, 1920×1080} @ {30, 60} in real time within the CPU/thermal budget?) on any machine, regardless of whether a camera is present or the capture backend works — so it is the default source for codec/perf tests.

§Why a moving pattern, not a flat frame

A static frame compresses to almost nothing after the first keyframe, so the encoder would do unrepresentative work and the real-time verdict would be a lie. This source therefore produces genuinely varying content every frame:

  • a diagonal luma gradient that scrolls (one column/frame), giving the encoder spatial detail plus global motion;
  • a solid block that bounces around the frame, giving localised motion vectors / residuals;
  • a slowly cycling chroma background so the U/V planes are not constant.

That keeps the per-frame encode cost in the same ballpark as real camera content (high-frequency detail + motion), which is what the measurement needs.

§Timing contract

[Self::next_frame] paces itself to the requested fps by sleeping the remainder of the frame interval — so captureFps reflects the target rate and the harness measures whether encode keeps up with that cadence (exactly the camera’s role). If the downstream encode is slower than the interval, no sleep happens and frames are produced back-to-back (the honest “can’t keep up” signal). Frames are stamped on the shared [hound_media::CaptureClock] like any real capturer.

§Real-time-safety

All buffers are sized exactly w*h + 2*(w/2)*(h/2) and written by index into a pre-validated Vec — no unwrap, no unchecked slicing, nothing that can panic on the capture thread. Generation is plain integer arithmetic.
