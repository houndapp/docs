Speaker-free synthetic audio SINK — the playout counterpart to synthetic_audio, so a headless/CI run drives a mix bus without opening a real DAC (two real RT mix engines on one host give false starvation verdicts). Same gating; see crate::api::Engine::start_synthetic_playout_bus.

§Synthetic audio sink (speaker-free playout)

An [AudioOutput] that stands in for a real speaker device: a headless client (a bot, or a CI/container run with no audio hardware) still needs a playout bus, but must not open a real DAC — and two real RT mix engines contending for one host’s audio device produce false audio-starvation verdicts. This is the output counterpart to [super::synthetic_audio::SyntheticAudioInput], and the sibling the mix bus opens via synthetic_sink_factory behind the synthetic feature.

§Why paced, and NOT discard-eagerly

The tempting shortcut is a sink whose play() just drops every frame on the floor. That would keep [Self::take_output_underruns] pinned at 0 forever — but 0 there would then mean “no signal”, not “healthy”: the end-of-pipeline AUDIBLE-GAP gauge (the definitive playout-stutter measure the debug report reads) would be dead, reporting perfect even when the mix bus is starving. Note the mix tick’s 10 ms cadence is self-clocked (Instant-paced sleep in the RT loop), NOT driven by this sink — so the sink cannot change the release clock or the per-peer jitter-buffer starvation counters. The ONE honest gauge a sink still owns is this one, so this sink behaves like a real device at the [AudioOutput] boundary: play() pushes into a small bounded ring, and a dedicated background thread drains that ring on its OWN 10 ms real-time clock — exactly mirroring a hardware output callback — counting a genuine underrun for any shortfall it finds. So take_output_underruns stays a meaningful, honest signal instead of a trivial zero.

§Timing contract

The drain thread paces itself the same way as [super::synthetic_audio::SyntheticAudioInput::next_frame]: it sleeps the remainder of the 10 ms interval since its last drain, then pops exactly one [DRAIN_CHUNK_SAMPLES]-sample chunk (10 ms @ 48 kHz). A real device drains on its hardware clock; this stands in for that clock in software.

Shared 🔒
State shared between the sink handle (play, take_output_underruns) and the drain thread. The ring is behind a Mutex<VecDeque<i16>>: this is NOT the real cpal RT callback with its wait-free hardware constraints, just a software stand-in on non-RT threads, so a mutex is an honest simplification.

A speaker-free playout sink. Open is construct (Self::open): a value that exists has its drain thread running.

Constants§

DRAIN_CHUNK_SAMPLES 🔒
Samples the drain thread pops per interval: 10 ms @ 48 kHz mono = 480. The same chunk size the mix bus pushes each tick, so the ring stays ~stable.
DRAIN_INTERVAL 🔒
One drain interval — 10 ms, matching the mix bus’s 10 ms output-side tick so the software drain rate equals the rate play() is fed at.
RING_CAPACITY_SAMPLES 🔒
Bounded ring capacity: ~60 ms of 48 kHz mono (2880 samples). The bus keeps a ~30 ms cushion filled (a silence pre-fill at open) and pushes one 10 ms chunk per tick; 60 ms is double that, giving headroom for the phase drift between the bus’s 10 ms software tick and this thread’s 10 ms drain tick without spurious overflow. A drop-oldest trim keeps a descheduled producer from growing the ring unbounded (mirrors the real sink’s drop-oldest overflow).
