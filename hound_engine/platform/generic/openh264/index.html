§Cross-platform software H.264 (Cisco openh264)

Software H.264 encode + decode via Cisco’s openh264 (the WebRTC baseline real-time codec), wired behind VideoEncode / VideoDecode. This is the path that completes functional video parity wherever the hardware H.264 codecs are absent — Linux/Windows desktops (VideoToolbox is Apple-only; there is no MediaCodec/NVENC backend). The wire codec for all-Apple calls is H.264 (the encoder sends HW H.264, receivers pick the decoder by wire codec), so a Linux client needs H.264 both ways to join a call with a Mac: SW-encode H.264 to send, SW-decode H.264 to receive.

§The interop contract (why this matters)

A Linux↔Mac call only works if the bytes are mutually decodable:

  • Mac VideoToolbox encodes H.264 Annex-B with SPS+PPS prepended on every IDR; this decoder (Openh264Decoder) must decode exactly that.
  • This encoder (Openh264Encoder) must emit the SAME shape — Annex-B with SPS+PPS in-band on every IDR — so a Mac’s VideoToolbox decoder accepts it.

openh264 satisfies both natively: its bitstream is Annex-B (4-byte 00 00 00 01 start codes) and, with the default SpsPpsStrategy::ConstantId, every IDR access unit carries SPS (NAL 7) + PPS (NAL 8) + the IDR slice (NAL 5) — byte-shape-identical to the VideoToolbox encoder’s output. The engine’s h264_* round-trip tests prove the cross-codec interop on this Mac (VT-encoded → openh264-decoded AND openh264-encoded → VT-decoded).

§Latency profile (why these settings)

  • UsageType::CameraVideoRealTime — the real-time usage scenario (no buffering for look-ahead, single reference).
  • Profile::Baseline — no B-frames, no CABAC: lowest decode latency and the widest-compatibility profile, matching the VideoToolbox path’s Baseline_AutoLevel SPS/PPS shape.
  • skip_frames(false) — never drop a frame to hit the rate target. For real-time voice/video we’d rather briefly overshoot bitrate than starve the receiver of a frame (a dropped frame is a freeze until the next one). openh264 prints a one-line warning that bitrate can’t be strictly held without frame-skip; that is the intended tradeoff, and the encoder’s trace level is lowered so the warning does not spam.
  • intra_frame_period caps the GOP so a late joiner is never more than a few seconds from a keyframe; request_keyframe forces an IDR on demand (PLI relay) via ForceIntraFrame.

§Real-time-safety

Runs on the dedicated capture/encode (or decode) thread, never async/UI. The encoder borrows the I420 planes zero-copy through [YUVSlices] (no RGB round-trip, no intermediate allocation beyond openh264’s own bitstream buffer). The decoder copies the decoded planes out respecting openh264’s internal (padded) stride into a tightly-packed I420 buffer — the same stride-respecting plane copy the NV12→I420 path does.

§LICENSING (distribution caveat — TODO before shipping)

Cisco grants a royalty-bearing-by-Cisco H.264 patent license that rides on the binary openh264 releases Cisco publishes; a from-source build like this one (the source Cargo feature) does NOT carry that grant. For distribution we must either ship Cisco’s signed binary libopenh264 (and load it dynamically) or obtain H.264 patent licensing ourselves. This is a pre-distribution decision, not a build blocker — development and CI build the source codec freely.

Structs§

Openh264Decoder 🔒
Software H.264 decoder (Cisco openh264). One per inbound video track. Decodes the Annex-B access units the encoders (this one OR VideoToolbox) emit — SPS/ PPS arrive in-band on each IDR, so there is no separate bootstrap step: the decoder buffers internally until it has seen a keyframe and returns Ok(None) until then. Not Send in our contract.
Openh264Encoder 🔒
Software H.264 encoder (Cisco openh264). One per outbound video track. Not Send in our contract: construct, drive, and drop on the one encode thread.
