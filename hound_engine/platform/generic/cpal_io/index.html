§cpal microphone capture + speaker playback (CoreAudio on macOS)

The runnable device I/O for the audio engine. CpalCapturer opens an input device and bridges its real-time callback to the blocking-pull AudioInput trait through a bounded ring buffer; CpalPlayback opens an output device and drains a ring buffer the AudioOutput trait pushes into. cpal’s CoreAudio backend is the macOS/iOS path; the same code is the Linux (ALSA/Pulse) / Windows (WASAPI) / Android (AAudio) path behind these traits — which is why this lives in platform::generic.

§Lifecycle

Open is construct (CpalCapturer::open / CpalPlayback::open) and stop is Drop — a value that exists is streaming, so a “capturer not started” runtime error class is unrepresentable.

§The real-time contract (the one thing that can go wrong here)

cpal invokes the data callback on a CoreAudio real-time thread. That callback:

  • MUST NOT allocate, lock unboundedly, or block — an xrun is audible.
  • Here it only does a bounded push/pop over an SPSC ring ([rtrb]-style, but we use a Mutex<VecDeque> with a try discipline: the lock is held for O(burst) memcpy only, never across a syscall, and the callback never waits on the consumer). A true lock-free SPSC ring is the noted production refinement.

§Format negotiation

We ASK for 48 kHz mono i16. CoreAudio’s default device config is often 48 kHz already, but the channel count and sample FORMAT are device-dependent (built-in mics frequently expose f32, and some expose stereo). So:

  • We open with the device’s default config’s sample format and convert whatever it delivers (f32/i16/u16/…) to i16 in the callback via cpal’s Sample/FromSample traits.
  • If the device delivers multi-channel, we downmix to mono by averaging channels in the callback before buffering.
  • The device keeps its native RATE; we resample to/from the engine’s 48 kHz at the trait boundary (capture next_frame / playback play, via super::resample::MonoResampler) on worker threads, never in the RT callback. AirPods mics (24 kHz HFP) and 44.1 kHz outputs are common; mislabelling instead of resampling plays every voice pitch-shifted (chipmunk/deep-voice). AudioInput::negotiated still reports the device truth.

§Permission / honest-verdict

On CoreAudio a denied microphone fails at build_input_stream / play time. cpal 0.18 maps that to [cpal::error::ErrorKind::PermissionDenied] (and absence to DeviceNotAvailable), which we translate to the typed AudioError::PermissionDenied / AudioError::NoDevice — a DISTINCT error, never a silent stream of zero-RMS frames.

Structs§

CpalCapturer 🔒
Microphone capture via cpal. The cpal Stream is !Send on CoreAudio, so it is created and held on the thread that calls Self::open; the ring it feeds is Send/Sync and is what AudioInput::next_frame reads. Dropping the capturer stops the stream and releases the device (on the owning thread — the !Send contract).
CpalPlayback 🔒
Speaker playback via cpal. The output callback drains the ring; the trait’s AudioOutput::play pushes into it (non-blocking, bounded). Dropping the sink stops the stream and releases the device.
PlaybackConsumer 🔒
The RT-output-callback (sole consumer) half + shared diagnostics.
PlaybackProducer 🔒
The mix-thread (sole producer) half of the playback ring + shared diagnostics.
PlaybackShared 🔒
Diagnostics shared across the playback ring’s three reachers: the mix thread (owner), the RT output callback (counts underrun), and the stream error callback (marks death). Plain atomics behind an Arc.
Ring 🔒
A bounded mono-i16 ring shared between a cpal RT callback (one side) and the trait’s blocking-pull / non-blocking-push API (the other). The mutex is only ever held for a bounded memcpy, never across a syscall, and the callback never waits on it (it would rather drop than block the audio thread).

Constants§

RING_CAPACITY_SAMPLES 🔒
Ring-buffer capacity (in mono i16 samples). Sized to ~10 frames of 20 ms (≈200 ms) so a brief consumer hiccup does not drop data, but the buffer can never grow unbounded: on overflow the oldest samples are dropped (capture) or the producer drops the frame (playback).
