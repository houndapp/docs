§Media Foundation capture-subtype selection (pure, platform-agnostic)

The Windows analogue of crate::platform::generic::camera_format: the preference ordering the Media Foundation camera backend uses to pick which video subtype to negotiate on an IMFSourceReader, expressed purely over the subtype’s 16-byte GUID data so the ranking — the part most likely to harbour a “we picked a format we can’t convert” bug — is unit-tested on every host, INCLUDING this macOS dev box where the Media Foundation capturer itself cannot compile.

§Why GUID bytes, not the windows crate’s GUID type

Keeping this module free of any windows type is what lets it compile and be tested off-Windows. The Media Foundation video subtype GUIDs all share the fixed tail 00000000-0010-8000-00AA-00389B71 (the “MFVideoFormat” family — a FourCC packed into Data1), so a subtype is fully identified by its Data1 FourCC. We match on the FourCC bytes here and the Windows module maps the chosen MfSubtype back to the concrete windows GUID constant (MFVideoFormat_NV12, …) at the one call site.

§Preference order (best first)

  1. NV12 — biplanar 4:2:0, the format MF camera capture and the MFT H.264 encoder both speak natively; the cheapest path to tightly-packed I420 (a plane de-interleave, no colour math), reusing the exact nv12_bytes_to_i420 the AVFoundation/V4L2 paths use.
  2. YUY2 — packed 4:2:2 (the Windows name for YUYV), the near-universal UVC webcam format; a repack + chroma reduction to I420 with no decoder (the same yuyv_to_i420).

MJPEG and other compressed subtypes are intentionally NOT ranked: unlike the V4L2 path (where the kernel hands a MmapStream of whatever raw format), the MF source reader can be asked to deliver NV12/YUY2 directly and will insert its own decoder/converter MFT for most cameras, so we always request a raw subtype and let MF satisfy it. A camera MF cannot serve as NV12 or YUY2 surfaces as a typed UnsupportedFormat at open, never a silent path.

Enums§

MfSubtype 🔒
A Media Foundation video subtype the engine knows how to turn into I420, ordered by how cheaply it converts (the Ord derive ranks Nv12 < Yuy2, i.e. NV12 is “smallest”/best — preferred = the minimum, mirroring crate::platform::generic::camera_format::CaptureFormat).

Functions§

request_order 🔒
Pick the best subtype to request from the source reader, applying the preference order (NV12 > YUY2). This is the request preference, independent of what the camera natively offers: MF will convert most cameras to the requested raw subtype via an inserted MFT. The Windows backend tries these in order and takes the first the reader accepts.
select_native_subtype 🔒
Given the FourCCs a camera’s native media types advertise, pick the best one we can convert — the fallback used when the reader will not transcode to a requested subtype and we must instead accept a native one. Mirrors crate::platform::generic::camera_format::select_capture_format: ignores unrecognised subtypes, None when nothing is convertible.
