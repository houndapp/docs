§Screen-source filtering, ranking, and audio-leg format math (pure)

The platform-agnostic logic a screen-capture backend leans on, factored out of the Apple ScreenCaptureKit module so the parts most likely to harbour a bug — “we listed our own window”, “we ranked a minimized window above the visible one”, “the stereo→mono downmix clipped” — are unit-tested on every host, including non-Apple ones where the SCK backend cannot compile.

Three concerns live here:

  1. Window filtering (is_shareable_window) — which enumerated windows a picker should show: only on-screen, titled, normal-layer (the menu bar, Dock, wallpaper, overlays all live on non-zero windowLayers), at-least-content-sized windows from apps other than us and the OS-furniture denylist.
  2. Source ranking (source_rank) — the picker sort key: on-screen before off-screen, then larger before smaller (the obvious “share target” is the big visible window).
  3. Audio downmix (downmix_f32_planar_to_mono_i16) — SCK delivers audio as deinterleaved (planar) float32, one buffer per channel; this averages the channels to one mono stream and converts to i16, the shape the crate::backend::AppAudio contract (and the mic path) speaks.

Constants§

MIN_SHAREABLE_EDGE 🔒
The smallest window edge (points) worth offering in a picker. Real app windows are bigger; what lives below this are tooltips, badges, focus proxies, and palette slivers that happen to ride layer 0.
NORMAL_WINDOW_LAYER 🔒
The window layer a normal application window lives on (kCGNormalWindowLevel maps to layer 0 in the windowLayer SCK reports). System chrome — the menu bar, the Dock, the wallpaper, status items, notification banners — all live on non-zero layers, so “layer == 0” is the cheap, robust “this is a real app window” filter that avoids capturing OS furniture.
SYSTEM_BUNDLES 🔒
System apps that own layer-0 windows a user would still never share — OS furniture the layer filter alone doesn’t catch. Compared case-insensitively against the owning bundle id.
