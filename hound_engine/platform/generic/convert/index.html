§Pixel-format conversion (capture → encoder input)

Cameras emit MJPEG / YUYV / NV12 / etc.; the encoders want planar I420 (8-bit YUV 4:2:0). Sources that produce packed RGB (PixelFormat::Rgb frames, e.g. the synthetic test source) convert via rgb_to_i420 here.

This conversion is a hot-path allocation + per-pixel arithmetic cost: it is exactly the work the hardware VideoToolbox path avoids by feeding a CVPixelBuffer straight into the encoder. Where the device can offer YUV natively (YUYV / NV12), the capturer prefers a cheap structural repack (yuyv_to_i420 / nv12_bytes_to_i420) over the RGB round-trip, since that skips the color-matrix math entirely.

Coefficients are BT.601 limited-range (the de-facto default for camera capture / H.264 / AV1 main-profile SD-HD). Chroma is subsampled by a simple 2×2 box average — cheap and visually adequate at half-resolution chroma; a higher-quality filter would cost more CPU per frame for little gain here.

§Scale-to-encode-target (scale_i420)

Cameras rarely open at the exact resolution we ask for — a FaceTime HD cam ignores a 720p request and hands back its native 1080p. The encoder, however, is built for one fixed geometry. To keep the two from ever disagreeing (a dimension mismatch would corrupt the encode), the capturers resize each captured frame from the camera’s negotiated resolution to the requested encode resolution in I420 via scale_i420 ([fast_image_resize] per plane). When native == encode the scale is skipped entirely (no allocation, no copy). The resize is part of every camera frame’s cost on this path, so it lands inside the per-frame encode budget.

Functions§

bgra_to_i420 🔒
Convert packed BGRA (8 bits per channel, the byte order Windows Graphics Capture and most PipeWire screen streams deliver) with an explicit row stride to planar, tightly-packed I420 (BT.601 limited range, matching the rest of the pipeline). The alpha byte is ignored, so this also covers the BGRx / BGR0 layouts (4 bytes/pixel, 4th byte unused).
nv12_bytes_to_i420 🔒
Convert packed NV12 (Y plane followed by interleaved CbCr, the V4L2 NV12 / V4L2_PIX_FMT_NV12 layout) with explicit plane strides to planar, tightly-packed I420. The Linux V4L2 capturer’s preferred path when a camera offers NV12 natively (the same plane layout AVFoundation hands us on Apple), but here the source is a raw V4L2 mmap buffer with a known row stride rather than a CVPixelBuffer, so this is a plain-bytes helper (no CoreVideo).
nv12_bytes_to_i420_cropped 🔒
NV12 → tightly-packed I420 with a display crop.
rgb_to_i420 🔒
Convert packed RGB (width*height*3 bytes, R,G,B) to planar I420.
scale_i420 🔒
Scale a planar I420 frame from src_w×src_h to dst_w×dst_h, resizing each plane independently: Y at full resolution and U/V at half resolution (the I420 chroma subsampling). Used by the AVFoundation capturer (Apple) and the V4L2 capturer (Linux), which both produce I420 at the camera’s native resolution and must hand the encoder the requested encode geometry.
yuyv_to_i420 🔒
Convert packed YUYV (a.k.a. YUY2 / V4L2_PIX_FMT_YUYV) to planar I420.
