§scap screen/window capture (Linux)

The Linux screen-share [ScreenCapture] backend: PipeWire + xdg-desktop-portal, via scap. Video only — the scap path has no audio leg. The other platforms have bespoke engine backends with more surface: macOS ScreenCaptureKit ([super::super::apple]: per-app audio, content-rect resize tracking, one-shot thumbnails) and Windows native WGC ([super::super::windows::wgc]: zero-copy D3D11 + a WASAPI system-audio leg).

§Why scap and not hand-rolled PipeWire

PipeWire + the portal is the correct, modern capture path on Linux — but it is hundreds of lines of unsafe FFI + lifecycle. scap wraps exactly that, so we get the right native path behind one small, maintained, type-safe surface.

§Pipeline & threading

scap’s Capturer is built and driven on a dedicated bridge thread (it owns a platform capture engine that must not move once started; building it on the bridge avoids any Send question). The bridge loops on the blocking get_next_frame(), converts each BGRA frame to I420 (convert::bgra_to_i420), scales it to the encode target (convert::scale_i420), stamps it on the shared [CaptureClock], and pushes it into a [BoundedRing] (drop-oldest). The [ScreenCapture] handle on the capture/encode thread just pulls from the ring — so next_frame_timeout’s Idle/Ended semantics (damage-driven liveness) work identically to the Apple path.

Capture is at the source’s native size (Resolution::Captured); we own the aspect-fit downscale to the 1080p-class cap, matching the SCK backend’s geometry contract.

ScapScreenCapturer 🔒
A running scap screen capture: a bridge thread owns the scap Capturer and fills the ring; this handle pulls I420 frames from the ring. !Send-by-use (drive and drop on one capture/encode thread, like every backend).

Constants§

RING_FRAMES 🔒
Frame ring capacity (whole frames) — small; frames are large, a deep queue is just latency. Overflow drops the OLDEST (the encoder is behind).

Functions§

bgra_to_rgba_thumbnail 🔒
BGRA(/BGRx/BGR0) → opaque RGBA, bilinear-downscaled to aspect-fit (max_w, max_h). Resizes as 4-channel (order-agnostic) then swaps B↔R and forces alpha opaque. Returns None on a malformed/short buffer.
bridge_loop 🔒
The bridge thread body: resolve the target, build the capturer, report the encode target, then drain frames → I420 → scaled → ring until stopped or the source ends. Everything scap-owned lives on this thread.
capture_thumbnail 🔒
One-shot RGBA thumbnail of a source for the share picker, aspect-fit within (max_w, max_h). The scap analogue of the SCK capture_thumbnail: open the target at a low fps, pull the first delivered frame, stop. Cross-platform via the same scap surface the live share uses — so Windows (WGC) gets dialog-free previews like macOS.
convert_frame 🔒
Convert one scap frame to a scaled I420 RawFrame. BGRA/BGRx/BGR0 take the alpha-ignoring path; anything else is counted as a bad frame and dropped.
enumerate_sources 🔒

resolve_target 🔒

The app-audio leg type for this backend: none. scap is video-only on Linux (the only platform that still uses it — Windows is on the native WGC backend, which carries its own WASAPI audio leg). So the open always returns None for audio.
