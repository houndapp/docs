Annex-B parameter-set detection for the HW encoders’ keyframe contract. Built where a HW encoder needs it (Windows MFT, Linux VAAPI).

§Annex-B parameter-set detection (H.264 / HEVC), pure

The HW encoders must honour the wire contract the decoders depend on: every keyframe carries its in-band parameter sets (H.264 SPS+PPS, HEVC VPS+SPS+PPS), so a fresh receiver bootstraps its decoder from a keyframe alone (see apple::vt_decoder / windows::mft — a keyframe without parameter sets makes the decoder return Ok(None) forever → a permanent black tile).

Some encoder Media Foundation Transforms omit the parameter sets on the IDR sample and expose them only once via MF_MT_MPEG_SEQUENCE_HEADER. This module is the pure detection that lets the encoder decide when to prepend its cached sequence header — unit-tested on every host, no platform dependency.

Functions§

contains_parameter_sets 🔒
Does this Annex-B access unit already carry the codec’s sequence parameter set (H.264 SPS = NAL 7; HEVC SPS = NAL 33)? Encoders emit SPS together with the other parameter sets (PPS, and VPS for HEVC), so the SPS is the proxy for “the whole parameter set is in-band here”. AV1 carries its config differently (sequence-header OBU) and is reported as already-present (this path is unused).
first_vcl_offset 🔒
Byte offset of the start code of the first VCL NAL in an Annex-B stream, i.e. where the coded slice begins. Everything before it in an IDR access unit is the parameter-set prefix (VPS/SPS/PPS, plus any SEI) — used to cache the sequence header. None if there is no VCL NAL (a parameter-set-only AU).
is_vcl 🔒
Is nal_type a VCL (slice / picture-data) NAL for codec? Parameter sets and SEI are non-VCL; everything that carries coded picture data is VCL.
nal_types 🔒
Collect the NAL-unit type of each NAL in an Annex-B stream, in order. Handles both 3-byte (00 00 01) and 4-byte (00 00 00 01) start codes. The type encoding is codec-specific: H.264 is nal[0] & 0x1f; HEVC is the 6-bit (nal[0] >> 1) & 0x3f. AV1 is not an Annex-B codec and yields nothing.
starts_with_start_code 🔒
Does the Annex-B stream begin with a start code (00 00 01 or 00 00 00 01)? Used to tell an Annex-B elementary stream from a length-prefixed (AVCC) one.
