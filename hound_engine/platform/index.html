§Per-platform backends

The concrete implementations of the crate::backend SPI, selected at compile time per target. Nothing in here is visible outside the crate (pub(crate)): the app reaches devices and codecs only through crate::api, which wraps these in platform-agnostic handles.

  • generic — the portable pieces that already build on every desktop target: cpal audio I/O (CoreAudio / WASAPI / ALSA-Pulse / AAudio), device-rate resampling, audio device enumeration, the SW H.264 codec (openh264, both ways), the synthetic test source, scap screen capture (Linux), and pixel-format conversion.
  • apple — the Apple-only camera + HW codec backend (AVFoundation capture, VideoToolbox H.264/HEVC/AV1, and the safe sys layer under them).
  • linux — the Linux camera backend (V4L2 via the v4l crate). Video codecs default to the portable SW openh264 H.264 floor; a VAAPI HW backend (cros-libva / cros-codecs: HW H.264 encode, H.264 + HEVC decode) is gated behind the off-by-default linux-vaapi feature (cros-libva’s system-libva build coupling — see the feature note). HEVC encode is unwired regardless.
  • windows — the Windows camera backend (Media Foundation via the official windows crate, IMFSourceReader synchronous pull) plus the Media Foundation Transform HW video codecs: HW H.264/HEVC encode (async MFT) and H.264/HEVC/AV1 decode (DXVA), with the SW openh264 H.264 floor retained.
  • stub — the uninhabited per-role backends for capabilities a target lacks (a typed Unsupported, never a panic).

§Backend selection lives HERE

The per-role backend types are resolved by cfg in this module (CameraBackend, ScreenBackend, AppAudioBackend, VideoDecoderBackend), so crate::api stays platform-agnostic — it names platform::CameraBackend and never carries a #[cfg(target_os = …)] of its own. A capability a target lacks resolves to the matching stub type.

§Adding a backend (the Windows/Linux parity path)

Add a windows/linux module gated on its target_os, implement the crate::backend traits for the native capture/codec APIs, then point the cfg-resolved alias below at it (and, for a new HW codec, wire the runtime crate::api::Engine::encode_caps / crate::api::Engine::decode_caps probes). Platform deps go in this crate’s [target.'cfg(...)'.dependencies] — the app crate never gains one.

One-shot screen-source thumbnail for the share picker: ScreenCaptureKit (Apple, macOS 14+), WGC (Windows), or a scap one-shot capture (Linux); Unsupported where there is no backend. A failure on any platform is non-fatal — the picker degrades to a title-only card.
