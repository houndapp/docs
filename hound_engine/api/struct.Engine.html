Struct Engine 

Source 
pub struct Engine;
Expand description

The platform media engine — the app’s single entry point to devices and codecs. Zero-sized: the backend is selected at compile time, so Engine is free to construct, copy, and keep anywhere.

pub fn encode_caps(self) -> VideoCodecCaps

This device’s video encode capabilities, probed at runtime — the local half of the codec ladder (crate::types::pick_encode_codec). H.264 is always set (HW VideoToolbox where present, else SW openh264 — the universal floor); HEVC/AV1 reflect the actual HW present on this silicon.

Runtime, not compile-time: whether this Mac has a HW HEVC/AV1 encode block is a question only the running machine can answer, so it is probed, never assumed from the target triple. Cheap enough to call at call-setup (the HEVC probe creates and drops a tiny VideoToolbox session); snapshot it once per session rather than per frame.

pub fn open_microphone( self, choice: &DeviceChoice<MicId>, clock: CaptureClock, ) -> Result<Microphone, AudioError>

Open a microphone and start capturing, stamping frames on clock.

Open is construct: the returned Microphone is already delivering (there is no unstarted state to misuse). A named device that has vanished falls back to the host default (logged) rather than failing — the mid-call unplug/hot-swap path. Fails loudly (typed) on no-device / permission-denied — never a silent zero stream.

The handle is !Send (CoreAudio): construct, drive, and drop it on one dedicated capture thread.

pub fn start_playout_bus( self, speaker_rx: Receiver<DeviceChoice<SpeakerId>>, apm: Option<Arc<Apm>>, clock: CaptureClock, playout_floor_us: i64, ) -> Arc<PlayoutBus>

Start the mixed-playout bus for a call: ONE output device stream and ONE coherent AEC render reference, fed by per-peer BusProducers. speaker_rx carries the live speaker choice (hot-swap); apm is the shared echo canceller (None ⇒ pass-through, the mix still plays).

clock is the session’s receive-side capture clock — the SAME Copy clock every per-peer pump pushes timestamps against, so the bus’s real-time pop schedule lines up with the producers’ jitter-buffer timeline. playout_floor_us is the per-peer jitter-buffer depth floor the adaptive jitter grows from.

pub fn open_camera( self, choice: &DeviceChoice<CameraId>, target: Dimensions, fps: Fps, clock: CaptureClock, ) -> Result<Camera, CaptureError>

Open a camera and start capturing at as close to target@fps as the device allows; delivered frames are always scaled to exactly target (so an encoder built from the same Dimensions always agrees), and stamped on clock.

Open is construct: the returned Camera is already delivering, and dropping it stops the session. A named device that has vanished falls back to the default camera (logged). On a platform without a camera backend this is a typed CaptureError::Unsupported — check [Self::capabilities] to degrade gracefully (audio-only).

The handle is !Send (AVFoundation session / V4L2 stream): construct, drive, and drop it on one dedicated capture thread.

pub fn screen_sources(self) -> Result<Vec<ScreenSourceInfo>, CaptureError>

Enumerate the shareable screen sources — displays and windows — for a share picker. Each ScreenSourceInfo carries the typed selection token, a title, the owning app (windows), the native geometry, and visibility.

Requires the macOS Screen Recording permission (TCC): a denied/undetermined state surfaces a typed CaptureError::PermissionDenied so the UI can route the user to System Settings rather than showing an empty list as if nothing were shareable. On a platform without a screen backend this is a typed CaptureError::Unsupported.

Blocking (it round-trips ScreenCaptureKit’s async enumeration to a channel); call it off the UI thread.

pub fn screen_thumbnail( self, source: &ScreenSource, max_w: u32, max_h: u32, ) -> Result<ThumbnailImage, CaptureError>

One-shot RGBA thumbnail of a screen source, aspect-fit within (max_w, max_h) — the picker’s visual preview of what a row would share. Blocking (round-trips the platform’s async screenshot API); call it off the UI thread. Requires macOS 14+ (SCScreenshotManager); older macOS and other platforms return CaptureError::Unsupported and the picker degrades to title-only rows.

pub fn open_screen_capture( self, source: &ScreenSource, config: ScreenCaptureConfig, clock: CaptureClock, ) -> Result<(ScreenShare, Option<ScreenAudio>), CaptureError>

Open a screen/window capture per config, stamping frames on clock.

Open is construct: the returned ScreenShare is already delivering I420 frames scaled to config.dimensions (aspect-preserved), and dropping it stops the session. When config.capture_audio is set and the platform supports the source’s audio scope ([Capabilities::app_audio_capture]; window sources capture the app’s own audio on macOS 13+ and Windows, display sources additionally capture system audio on Windows), the returned tuple’s second element is Some(ScreenAudio) delivering that audio as 48 kHz mono 20 ms frames (the mic contract); otherwise it is None.

A Screen Recording TCC denial surfaces CaptureError::PermissionDenied; a platform without a screen backend returns CaptureError::Unsupported. The handle is !Send (the SCStream is tied to its setup context): construct, drive, and drop it on one dedicated capture thread.

pub fn h264_encoder( self, dims: Dimensions, fps: Fps, bitrate: Bitrate, cadence: KeyframeCadence, ) -> Result<VideoEncoder, CaptureError>

Build a real-time H.264 encoder for dims@fps targeting bitrate, preferring hardware (VideoToolbox on Apple) and falling back to software (openh264, the Cisco WebRTC baseline codec) where no HW encoder exists — i.e. every Linux/Windows desktop today. The geometry/rate preconditions are carried by the parameter types, so an invalid configuration cannot be expressed.

This always succeeds in producing an H.264 encoder (HW or SW) on every supported platform: H.264 is the all-platform wire floor, so any client MUST be able to encode it. The returned VideoEncoder hides which path was selected; the codec on the wire is H.264 either way. Selection order is HW > SW (see [Capabilities::hw_h264_encode] / [Capabilities::sw_h264]).

Thin wrapper over Self::video_encoder(VideoCodec::H264, …). Not Send: lives on the capture/encode thread.

pub fn video_encoder( self, codec: VideoCodec, dims: Dimensions, fps: Fps, bitrate: Bitrate, cadence: KeyframeCadence, ) -> Result<VideoEncoder, CaptureError>

Build a real-time video encoder for a specific codec, preferring hardware:

  • H.264 — HW VideoToolbox on Apple, else SW openh264. Always succeeds (the universal floor); a HW failure falls through to SW.
  • HEVC (H.265) — HW VideoToolbox only. Unsupported on silicon without a HW HEVC block and everywhere off Apple (there is no SW HEVC encoder).
  • AV1 — HW VideoToolbox on M5 Pro/Max + modern discrete GPUs. Not yet wired (the VideoToolbox AV1 encoder awaits validation on AV1-encode HW), so it currently returns Unsupported; [Self::can_encode] reports the same, so the codec ladder never selects AV1 for encode until it lands.

The codec ladder (crate::types::pick_encode_codec) chooses which codec to pass here; callers that just want “an H.264 encoder” use Self::h264_encoder. Not Send: lives on the capture/encode thread.

Platform dispatch (HW VideoToolbox vs SW openh264, per-codec availability) lives in crate::platform::VideoEncoderBackend — this is pure delegation.

pub fn sw_h264_encoder( self, dims: Dimensions, fps: Fps, bitrate: Bitrate, cadence: KeyframeCadence, ) -> Result<impl VideoEncode, CaptureError>

Build the software H.264 encoder (openh264) explicitly — the cross-codec interop test source. Most callers want Self::video_encoder (which prefers HW); this is for when the SW path is specifically required.

pub fn video_decoder( self, codec: VideoCodec, ) -> Result<VideoDecoder, CaptureError>

Build a video decoder for codec (by wire codec), preferring hardware: HW VideoToolbox on Apple (H.264/HEVC/AV1), SW openh264 (H.264 only) elsewhere. HEVC/AV1 off Apple are Unsupported — the receive pipeline drains-and-drops (audio still plays) rather than failing the call.

Platform dispatch lives in crate::platform::VideoDecoderBackend — this is pure delegation. Not Send: lives on the decode thread.
