§Acoustic echo cancellation (full-duplex APM)

Wraps the [sonora] audio-processing module (a pure-Rust port of WebRTC’s modules/audio_processing: AEC3 + noise suppression + AGC2 + high-pass) into the live call so speaker-mode calls (no headphones) do not feed the far-end audio back into the mic: the render reference (what we play out) is subtracted from the capture (the mic).

§Why sonora and not webrtc-audio-processing

webrtc-audio-processing (the tonarino C++ binding) has a -sys crate that hard-requires meson to build the bundled WebRTC C++, and there is no system libwebrtc-audio-processing to dynamically link. sonora is the same WebRTC algorithms ported to pure Rust: it builds with only the toolchain. It is v0.1.0 — newer and less battle-tested than the C++ original — so the toggle below degrades to pass-through if init ever fails.

§The timing contract (the one thing that must be right)

  • Sample rate / channels: the whole endpoint is 48 kHz mono i16 (SAMPLE_RATE/CHANNELS); the APM is configured to match, so there is no resample on either the capture or render path.
  • Frame size: the APM’s native chunk is 10 ms (480 samples @ 48 kHz), NOT our 20 ms Opus frame (960 samples). Every 20 ms PcmFrame is split into two 10 ms passes (APM_FRAME_SAMPLES) — both for capture (Apm::process_capture) and render (Apm::process_render). A frame whose length is not a whole number of 10 ms chunks is passed through unprocessed (defensive; the live path always hands exact 960-sample frames).
  • Stream delay: AEC3 needs an estimate of the capture-vs-render delay to seed its delay search. We set a fixed INITIAL_STREAM_DELAY_MS once; AEC3 is adaptive and refines the true delay itself, so the estimate only has to be in the right ballpark (it is not a hard alignment requirement like a fixed-delay canceller). The render reference is tapped just before playout, so the real loop delay is roughly the playout buffer + device output latency.

§Threading / locking

One Apm is shared (Arc) between the capture thread (which calls Apm::process_capture) and the mix-bus thread (which calls Apm::process_render on each mixed chunk it plays). sonora::AudioProcessing is Send but not Sync, so it lives behind a Mutex. The processing calls are short (a few hundred µs for two 10 ms chunks) and the lock is never held across IO or .await. Neither call happens inside a cpal RT callback — capture processing is on the mic worker thread (between next_frame and the encode channel) and render processing is on the bus thread (between the mix and the playback ring), so the RT audio callbacks are never blocked on this lock.

§Render reference: the single mixed playout bus

AEC3’s render reference must be ONE coherent stream — exactly what the speaker plays. The mixed-playout bus makes it so: all remote peers’ due PCM is summed into one 10 ms chunk and fed to Apm::process_render once per tick, then played on the single CpalPlayback. Because there is exactly one render feed by construction, AEC is correct for ANY number of peers — interleaving independent per-peer references corrupts the canceller. The caller is the bus, not N per-peer pumps.

Structs§

Apm
The live-call audio-processing module: shared echo canceller + noise suppressor + gain control. Capture frames (mic) and render frames (far-end playout) are both fed through it; it subtracts the render reference from the capture so speaker audio does not echo.

Constants§

APM_FRAME_SAMPLES
Samples in one APM processing chunk: 10 ms @ 48 kHz mono = 480. This is the APM’s native frame size (sample_rate / 100), distinct from the 960-sample 20 ms Opus frame the rest of the pipeline carries. Every PcmFrame is split into SAMPLES_PER_FRAME / APM_FRAME_SAMPLES (= 2) chunks for the APM.
INITIAL_STREAM_DELAY_MS
Initial capture↔render stream-delay estimate handed to AEC3, in milliseconds. The true delay is roughly the playout jitter target (60 ms) plus the speaker device’s output latency; 70 ms is a sane seed. AEC3 is delay-adaptive — it searches for and tracks the real delay from this seed — so this only has to be in the right ballpark, not exact. Exposed as a const so the live wiring and the tests agree on the value.
