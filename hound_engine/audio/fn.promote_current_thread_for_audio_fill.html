pub fn promote_current_thread_for_audio_fill()
Expand description

Promote the CURRENT thread to real-time audio scheduling and hold the policy for the thread’s whole lifetime (the handle is stashed in a thread-local).

Intended as a tokio::runtime::Builder::on_thread_start hook for the dedicated audio-receive/fill runtime: each per-peer pump decodes Opus and fills its mix jitter buffer, and if a CPU-heavy screen-share decode/scale/render load deschedules that fill, the RT mix thread finds the buffer dry at its pull tick and the call audibly stutters (mix_starve). The mix bus (the playout side) is already RT; this extends the same guarantee to the producer side so video load can’t starve voice — “audio is sacred”. The work is bounded and mostly parked on the network, so RT promotion is safe (no busy-loop to starve the system).

Best-effort: on failure (no entitlement, unsupported platform) it logs and the thread runs at normal priority — never a hard error. Idempotent per thread.

NOTE: this protects the decode+fill half of the audio receive path. The QUIC socket reads feeding it run on the shared connection runtime; if THAT is the descheduled link under load, this alone won’t fully cure mix_starve.
