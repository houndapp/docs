§Mixed-playout audio bus — one device stream + one coherent AEC reference

All remote peers play through ONE mixed output stream. Echo cancellation requires it: Apm::process_render must be fed the one coherent stream the speaker actually plays — interleaving independent per-peer render references corrupts AEC3’s canceller for any N>1. A single mixed playout makes the render reference exact for any N by construction, and halves the device handles (one stream per call, not one per peer).

§The tick model (PULL-based: the RT thread is the sole release clock)

One dedicated playback thread per call owns the single CpalPlayback. It ticks on a wall-clock 10 ms cadence (the APM chunk size, MIX_CHUNK_SAMPLES) — driven by the OUTPUT side, never by peer frame arrival, because the cpal ring absorbs the jitter between the tick and the DAC. On each tick the bus, for EACH registered producer:

  1. pops every due frame from that peer’s jitter buffer on its OWN real-time clock — playout.pop_due(clock.now() + lookahead) — appending the released PCM to the slot’s sample carry-queue, then drains up to MIX_CHUNK_SAMPLES from the carry,
  2. sums the producers sample-for-sample in i32 then saturating-clamps to i16 (no wrap distortion on overload),
  3. feeds the MIXED 10 ms chunk to Apm::process_render once (the single coherent reference), and
  4. pushes the mixed chunk to the CpalPlayback ring.

§Why pull, not push

A push design — a per-peer pump task (a normal async task) calling pop_due and PUSHING samples into the slot for this RT thread to drain — puts a second, non-real-time scheduler on the release path: under video-encode load the pump gets descheduled, feeds its slot late, and the RT drain finds the slot short — an audible per-peer gap (mix_starves) with the device ring still full (output_underruns == 0). No slot lead or cushion can fix that, because the producer’s timing is the variable. Pull — the RT thread is the ONLY thing that calls pop_due, on its own steady clock — removes that scheduler from the release path entirely. The pump’s sole job is to FILL the jitter buffer (network-paced; its scheduling never affects playout), and the jitter buffer IS the cushion that absorbs the pump’s arrival jitter. Both push (pump) and pop (this thread) read the same Copy [CaptureClock], so the buffer’s sender→local timeline mapping is consistent across the two threads.

When zero producers are registered the bus STILL ticks (pushing silence), so the device stream and the AEC reference stay warm for the whole call — AEC3’s delay/echo state does not have to re-converge each time the room goes briefly quiet or the last peer’s pipeline is mid-rebuild.

§Registration + bounds

A per-peer pipeline, instead of owning a playback thread, registers a BusProducer and PUSHES its decoded, timestamped PCM into the slot’s jitter buffer via BusProducer::push (which also folds the arrival into the slot’s [AdaptiveJitter] and reports back the depth/jitter for stats). The RT thread pulls from that buffer. The jitter buffer is bounded (drop-oldest at its frame cap) and the sample carry-queue is bounded to ~100 ms (SLOT_CAPACITY_SAMPLES) drop-oldest, so one stalled or runaway peer can never grow the bus’s memory. Dropping a BusProducer (its pipeline torn down / aborted) unregisters its slot from the bus, so a left peer stops contributing to the mix with no explicit teardown call.

§Hot-swap / vanish / reopen

The bus thread owns the speaker DeviceChoice hot-swap, the vanished-device fallback (handled inside [CpalPlayback::start]), and the reopen backoff — in ONE place targeting the ONE stream. call_set_speaker nudges the shared watch sender; the bus thread re-opens the single device in place without disturbing the mix.

§!Send discipline

cpal::Stream is !Send on CoreAudio, so the CpalPlayback is constructed, driven, and dropped on the bus’s own thread and never crosses a thread boundary. Producers (Send/Sync) only ever touch the bounded slot queues.

§Decoupling

This module is deliberately free of session-specific types: it takes raw mono i16 PCM (&[i16]) in and plain handles out. The only audio dependencies are Apm (the render-reference sink) and CpalPlayback (the device), both behind narrow calls.

Slot 🔒
One producer’s per-peer slot. The PUMP pushes decoded, timestamped PCM into state.buffer (the jitter buffer); the BUS THREAD pops the due frames on its own real-time clock into carry and drains MIX_CHUNK_SAMPLES per tick. The two mutexes are each held only for a bounded op, never across IO; the producer is the async per-peer pump, the consumer is the bus’s own thread (neither is a cpal RT callback).

Constants§

GLITCH_RING_CAP 🔒
Max glitch events the bus retains between drains (bounded, drop-oldest). The client drains every stats window (1 Hz), so this only needs one window’s events plus headroom.
MIX_CHUNK_SAMPLES
Samples in one mix tick: 10 ms @ 48 kHz mono = 480. Identical to the APM’s native chunk (APM_FRAME_SAMPLES) so the mixed chunk feeds process_render as exactly one block, and the tick cadence matches the canceller’s frame size.
MIX_PULL_LOOKAHEAD_US 🔒
How far PAST strictly-due the RT thread pops each tick: exactly one MIX_TICK (10 ms). A 20 ms audio frame spans two 10 ms ticks, so its due-instant lands between ticks for some phase offsets; a one-tick lookahead releases the next frame into the carry-queue before the carry can bottom out, so the slot always has a full chunk to drain regardless of how the frame-due phase aligns to the tick grid. Because the RT thread (not the pump) owns this read, the lookahead is immune to producer scheduling — it just smooths the frame/tick phase. Net latency is unchanged: a frame released lookahead early simply waits that long in the carry-queue instead of the jitter buffer.
MIX_TICK 🔒
The wall-clock period the bus thread ticks at: 10 ms, matching MIX_CHUNK_SAMPLES at 48 kHz. Output-side cadence — the cpal ring absorbs the difference between this and the DAC’s real pull timing.
MIX_TICK_MS 🔒
MIX_TICK in milliseconds — the chunk duration the output cushion is counted in.
NO_PEER_TAG
A sentinel peer_tag for a producer registered without attribution (the plain PlayoutBus::register) or a bus-global event (a late tick) — the glitch is recorded but not tied to a specific peer.
OUTPUT_CUSHION_MS 🔒
Target depth of the output device ring the bus keeps filled (~3 chunks). The device drains the ring on its hardware clock while this thread refills it on a software clock; without a cushion the ring sits at ~0 and underruns on the slightest pacing jitter (or when encode load deschedules this thread). 30 ms absorbs normal jitter at a small, constant added output latency (AEC3 tolerates the fixed delay); the thread-priority promotion handles longer stalls.
REOPEN_BACKOFF 🔒
Initial rate-limit on device-reopen attempts when no speaker is open (a down device must not hammer start every tick); first retry after ~500 ms.
REOPEN_BACKOFF_MAX 🔒
Ceiling for the reopen backoff. A persistently unavailable output device (e.g. an ALSA hw: held exclusively by another process on Linux — where each failed open/close churn can itself prolong the EBUSY) must not retry at 2 Hz forever, flooding the log and thrashing the device. The interval doubles from REOPEN_BACKOFF up to this cap, then holds — a slow heartbeat that still recovers automatically once the device frees, without the storm.
SLOT_CAPACITY_SAMPLES 🔒
Per-producer slot bound: ~100 ms of 48 kHz mono audio (4800 samples). A drop-oldest policy keeps a stalled/runaway peer from growing the bus’s memory; 100 ms is ~5 mix chunks of slack, enough to ride out a scheduling hiccup without unbounded growth.
TICK_LATE_GLITCH_US 🔒
A mix tick this late (µs) or more is recorded as a GlitchKind::LateTick. Half a tick (5 ms) — below this is ordinary scheduler noise the output cushion absorbs; above it the RT feeder genuinely slipped and the device ring risks draining.

cpal_sink_factory 🔒
The production sink factory: open a CpalPlayback for the choice (with the vanish-fallback inside open), boxed as AudioOutput. None on an open failure (logged) — the bus keeps ticking the AEC reference and retries.
mix_bus_loop 🔒
The bus thread body: own the single output sink (opened via sink_factory, with the speaker hot-swap / vanish-fallback / reopen-backoff), and every MIX_TICK sum the producers into one chunk, feed it to the APM render reference once, and play it. Ticks silence with zero producers so the device stream + AEC reference stay warm. The !Send sink never leaves this thread.
mix_chunk 🔒
Pull one mixed MIX_CHUNK_SAMPLES-sample chunk from the snapshot’s slots at local time now_us: each slot releases its due jitter-buffered frames (on the RT clock) into its carry-queue and accumulates a chunk in i32 (no overflow on N loud peers), then saturating-clamp to i16 (no wrap distortion on overload). A slot with nothing due contributes silence. Returns the mixed chunk.
prefill_output 🔒
Pre-fill a freshly-opened output sink’s ring with OUTPUT_CUSHION_MS of SILENCE. This establishes the device cushion ONCE, at open, so the steady one-chunk-per-tick mix keeps the ring ~stable (RT-paced ticks ≈ the device drain rate) WITHOUT ever draining the per-peer slots faster than real time to refill it. Silence (not slot audio) means the pre-fill costs nothing from any peer’s stream; it is a fixed ~cushion of added output latency, which AEC absorbs.
warmup_reopen_at 🔒
Arm the one-shot super::ROUTE_WARMUP_DELAY timer iff the just-opened sink runs at a non-engine rate (a resampled device — in practice a Bluetooth HFP headset whose SCO link may not be carrying audio yet). 48 kHz devices return None (no warmup reopen), so the common path is untouched. See super::ROUTE_WARMUP_DELAY.
