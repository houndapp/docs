Expand description

The mixed-playout bus for one call. Owns the dedicated playback thread (which owns the single CpalPlayback and runs the 10 ms mix tick) and the producer registry. Per-peer pipelines register a BusProducer and push their due PCM into it; the bus sums them, feeds the mixed chunk to the APM render reference once, and plays it.

Dropping the bus signals its thread to stop (the !Send CpalPlayback is then dropped on its owning thread) — Drop-safe like the per-peer pipelines.

Fields§

§registry: Arc<Registry>§next_id: AtomicU64§playout_floor_us: i64

Jitter-buffer depth floor (µs) for every new slot — the latency floor the adaptive jitter grows from. Set by the call (PLAYOUT_TARGET_US). (The shared capture clock itself is owned by the bus thread, not stored here — only the thread reads now() each tick.)

§stop: Arc<AtomicBool>§mix_starves: Arc<AtomicU64>

Cumulative per-peer mix-STARVATION events shared with every Slot (a primed slot that came up short at tick time — the per-peer audible gap). The app polls this (diffing per window) to surface a live “mix starvation” stat so playout stutter is a measured number, not a vibe. See Slot::starves.

§output_underruns: Arc<AtomicU64>

Cumulative output-ring underrun SAMPLES (the device RT callback played silence because the ring ran dry — the device-side gap). Accumulated each tick from the sink’s AudioOutput::take_output_underruns.

§tick_late_lat: Arc<Mutex<LatencyHistogram>>

How LATE each 10 ms mix tick fired versus its schedule (µs), as a windowed percentile histogram drained by the stats emitter. This is the RT-health signal: if the bus thread is descheduled (e.g. a 60 fps screen-share encode hogging the CPU), ticks slip, the device ring drains, and ALL audio stutters — a p95 well above ~0 localizes the gap to the feeder thread, not the network.

§glitches: Arc<Mutex<VecDeque<AudioGlitch>>>

Bounded ring of discrete audio-glitch events (drop-oldest), recorded by the bus thread at the instant each gap happens and drained by the client each stats window into a rolling history the debug report renders as a glitch log.

§clock: CaptureClock

The session capture clock (the same Copy clock the bus thread stamps glitches against), so the client can compute each glitch’s age (now_us − at_us).

pub fn start( speaker_rx: Receiver<DeviceChoice<SpeakerId>>, apm: Option<Arc<Apm>>, clock: CaptureClock, playout_floor_us: i64, ) -> Arc<Self>

Start the bus: spawn the dedicated playback thread that owns the single CpalPlayback (opening the current speaker_rx choice, with hot-swap + vanish-fallback + reopen backoff) and runs the 10 ms mix tick feeding the shared apm render reference. apm == None ⇒ pass-through (no AEC), the mix still plays.

fn start_with_sink( speaker_rx: Receiver<DeviceChoice<SpeakerId>>, apm: Option<Arc<Apm>>, clock: CaptureClock, playout_floor_us: i64, sink_factory: Box<dyn FnMut(&DeviceChoice<SpeakerId>) -> Option<Box<dyn AudioOutput>> + Send>, ) -> Arc<Self>

Like Self::start but with an injectable SinkFactory so tests drive the tick loop without opening a real audio device. The factory runs on the bus thread (the !Send contract).

pub fn register(&self, label: &'static str) -> BusProducer

Register a new producer slot and return its BusProducer handle. Each per-peer audio pipeline calls this once and pushes its decoded PCM into the returned handle; dropping the handle unregisters the slot. The slot is untagged (NO_PEER_TAG) — its glitches are recorded but not attributed to a peer; use Self::register_tagged to attribute them.

Source

pub fn register_tagged(&self, label: &'static str, peer_tag: u32) -> BusProducer

Like Self::register but tags the slot with peer_tag (the control-plane peer id), so an underflow AudioGlitch recorded for it names the peer in the debug report’s glitch log.

Source

pub fn drain_glitches(&self) -> Vec<AudioGlitch>

Drain the recorded audio-glitch events since the last call (timestamp-ordered, oldest first). The client calls this each stats window and appends them to a rolling history the debug report renders. Clears the ring.

Source

pub fn now_us(&self) -> i64

The session capture-clock time now (µs) — the reference the client uses to age drained AudioGlitches into “t-N s” entries for the report.

Source

pub fn starves_by_kind(&self) -> Vec<(&'static str, u64)>

Cumulative starve counts attributed BY SOURCE KIND (the label passed to Self::register), summed across the currently-registered slots of each kind. Lets the stats loop show WHICH audio is starving — e.g. a screen share’s intermittent app audio (often a metric artifact, inaudible on the call) vs the remote mic (real voice stutter). Diagnostic only; the authoritative aggregate is Self::mix_starves.

Source

pub fn mix_starves(&self) -> u64

Cumulative per-peer mix-starvation events since the call started (a primed slot that came up short at a 10 ms tick — the per-peer playout gap). The app diffs this per stats window. Steady state should hold flat; a rising count is the audio stutter, localized to the per-peer pump→mix cadence (vs the device path, which Self::output_underruns measures).

Source

pub fn output_underruns(&self) -> u64

Cumulative output-ring underrun samples since the call started (the device’s RT callback played silence because the mix did not keep the ring fed — a tick-pacing or device-rate gap affecting ALL audio). Rising ⇒ device-side stutter. The app diffs this per window.

Source

pub fn tick_late_percentiles_us(&self) -> (u64, u64)

Drain the mix-tick lateness percentiles (p50, p95) in µs since the last call — the RT-health signal. ~0/~0 in a healthy run; a high p95 means the bus thread is being descheduled (CPU/encode contention) and dragging the whole playout with it. Called once per stats window (the drain clears the histogram).

Source

pub fn stop(&self)

Stop the bus thread (idempotent). The thread drops its CpalPlayback on its own thread (the !Send contract).
