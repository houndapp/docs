§Hello / HelloAck handshake

The first thing a client does after opening a QUIC connection to the SFU is send a Hello message over a reliable stream (QUIC stream, not datagram). The SFU responds with either HelloAck (success) or HelloNack (failure) on the same stream, then closes it.

After the handshake, all subsequent communication uses:

  • QUIC datagrams — unreliable, for delta video frames and audio
  • New QUIC streams — reliable, for keyframes that must not be lost

§Wire encoding

Messages are length-prefixed JSON so both sides can read the exact number of bytes for each message from the stream:

┌──────────────┬──────────────────────────┐
│ length: u32  │ JSON body (UTF-8)         │
│  (LE, 4 B)   │ (`length` bytes)          │
└──────────────┴──────────────────────────┘

§Codec negotiation

The client sends its codec preferences as ordered lists (best first). The SFU picks the first entry from each list that it supports. All participants in a room use the same codecs — the SFU does not transcode — so if a second client joins with incompatible preferences it receives HelloNack.
