Handle one incoming MoQ relay connection: authenticate, admit, scope, run the moq-net server session.

§Stream ordering (why the token stream is accepted first)

The client opens the relay-token control bi-stream first and awaits the relay’s AUTH_OK before starting its MoQ handshake. The relay accepts that bi-stream (conn.accept_bi) before handing the connection to [moq_net::Server::accept] (which opens its own control stream). QUIC bi-streams are accepted in open order, so the token exchange always lands before the MoQ control stream. If auth fails the connection is closed and the MoQ handshake never runs.

§Ordering of auth vs. admission vs. the verdict byte

authenticate verifies the token but deliberately does NOT write AUTH_OK on success — it hands back the still-open send half instead. That lets this function run the SessionRegistry::admit cap check before committing to a verdict, so an over-cap connection still gets a clean single- byte AUTH_REJECT (the same wire shape as a bad token) rather than silently being admitted and then killed, or racing a verdict it already sent.
