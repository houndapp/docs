Per-connection ops surface + policy enforcement, run for the life of one authenticated relay session (spawned in handle_connection, aborted the instant the connection closes). Three concerns share one loop so there is exactly one task watching a connection’s health:

  • Ingress budget — samples conn.stats().udp_rx.bytes (cumulative QUIC bytes received) every IngressLimits::sample_interval and feeds an IngressBudget to catch a sustained flood (IngressLimits::max_bytes_per_sec).

  • Broadcast quota — counts this connection’s own currently-announced broadcast paths from ingest_announces’ event stream, bounding runaway space/{s}/peer/{user_id} fan-out (IngressLimits::max_broadcasts_per_session).

    Why an announce-event count and not moq-net’s Stats aggregator (the module built for exactly this kind of counting): Stats’s per-broadcast counters are private and only reach the outside world by publishing a JSON snapshot onto a broadcast that a consumer must then subscribe to and parse — there is no synchronous in-process accessor. Standing up a scratch origin plus a self-subscribing background task just to read back two integers is exactly the kind of indirection not worth adding for a number this cheap to derive directly. ingest_announces is a consumer over the SAME shared origin subtree scoped_publish (this connection’s ingest half) publishes into, via the ordinary [OriginConsumer] announce API every other subscriber in this codebase already uses — so counting its live announce/unannounce events is both cheaper and simpler.

    The tradeoff: that subtree is keyed by the authenticated (space, user) pair, not by QUIC connection identity, so if the SAME user holds more than one concurrent session in the SAME space (allowed up to SessionLimits::max_per_user), every one of those connections’ watchdogs sees the group’s COMBINED broadcast count, not just its own. In the common case (one session per user per space) the two coincide exactly. When they don’t, the quota still bounds something real — one identity’s total ingest fan-out into a space — just not quite what “per-session” literally promises.

  • Ops log — every OPS_LOG_INTERVAL, one info! line summarising both, so “which connection is the problem” is answerable by grepping one connection’s lines rather than reasoning about aggregate metrics.

Either policy trip closes the connection with a distinct close code (CLOSE_INGRESS_BUDGET_EXCEEDED / CLOSE_BROADCAST_QUOTA_EXCEEDED) rather than the generic 0 other close paths use, so a client inspecting the code/reason on conn.closed() can tell a policy close from a network failure.
