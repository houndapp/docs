§MoQ relay — the native forward plane

The MoQ forward plane: a thin, forward-only relay that fans every publisher’s encoded media out to the other subscribers in the same space. This is the only media path the SFU serves — clients dial the RELAY_ALPN (moql) ALPN, negotiated via crate::sfu::Sfu.

§Forward-only, no transcode

The relay never decodes media. It re-emits the encoded objects a publisher writes to every subscriber of that track, byte-for-byte. moq-net owns the actual fan-out and per-subscriber prioritisation: each GOP is one QUIC uni-stream and the consumer’s next_group priority-drops a stale group at its boundary, so a receiver never freezes replaying stale deltas while a fresh keyframe group waits. We do not hand-roll routing tables, NACK caches, or SSRC/seq rewrites — that is exactly what MoQ’s group/object model gives us for free.

§How moq-net does the forwarding (the Origin)

moq-net routes through a shared [moq_net::Origin]: a tree of broadcast paths. A [moq_net::Server] session is wired with two halves of that origin:

  • with_consume(OriginProducer) — “consume any remote broadcasts, inserting them into this origin” (the publisher half: a connected publisher announces a broadcast and moq-net inserts it into our shared origin).
  • with_publish(OriginConsumer) — “publish any local broadcasts from this origin to the remote” (the subscriber half: a connected subscriber’s SUBSCRIBE is served from whatever the shared origin holds).

Give every connection both halves of one shared root origin and moq-net itself forwards: publisher A announces space/X/peer/A/video into the origin → subscriber B’s session sees the announce and serves the objects. The relay code never copies a byte; it only owns the origin and the auth gate.

§Auth gate (the mandatory bit)

moq-net itself does not authenticate. Before any media flows we:

  1. Validate the relay-token JWT-equivalent the client presents on a dedicated control bi-stream, reusing crate::auth::verify_relay_token (the same HMAC-SHA256 {user}:{space}:{expiry}:{mac} scheme the backend mints). A bad/expired/missing token is rejected before the MoQ handshake runs. The whole exchange is bounded by RelayConfig::auth_deadline (see handle_connection) — a client that completes the QUIC handshake but never opens/finishes the token stream cannot hold the connection open indefinitely.
  2. Admit the session against SessionLimits — a global cap and a per-user cap, checked right after a successful token verification and before the moq-net handshake runs (see SessionRegistry). Over-cap is a clean AUTH_REJECT, not a dropped connection.
  3. Scope the connection to its space — always, both halves. From the validated crate::auth::RelayTokenClaims we derive the namespace prefix space/{space_id} and hand the moq-net session an origin scoped to it via moq_net::OriginProducer::scope / [moq_net::OriginConsumer::scope]. A scoped producer’s publish_broadcast returns false for any path outside the prefix, and a scoped consumer never sees announcements outside it — so a participant authenticated for space A cannot announce or subscribe to space B’s namespace. moq-net enforces the boundary; we just set it from the token.
  4. Scope the INGEST half to the caller’s own peer namespace, always. The two halves of a session are scoped independently (see handle_connection): the SERVE half (with_publish, subscribing) stays space/{space_id} — a participant must be able to subscribe to anyone in the call. The INGEST half (with_consume, announcing) is tightened to space/{space_id}/peer/{user_id}, so a connection authenticated as user A can only announce broadcasts under its own user_id sub-path — it can never publish (or displace, via moq-net’s route tie-break) under another peer’s path. This is what makes the peer/{user_id}/{peer_id}/{track} tail of NAMESPACE_CONVENTION a real boundary rather than just a naming convention.

This is call-membership + per-user-namespace auth. Per-track allow-listing (which peer may publish which track within a space, driven by hound-control’s SessionServer roster) remains a later refinement.

§Ongoing per-connection budgets (after admission)

Auth and SessionRegistry::admit are one-time checks at connection start. A connection that passes both can still misbehave for the rest of its life — flooding ingress bytes, or announcing an unbounded number of broadcast paths — so handle_connection also spawns a per-connection ingress_watchdog that runs for the session’s whole lifetime, enforcing IngressLimits and logging a periodic ops summary. See that function’s doc for the enforcement details and close-code scheme.

RelayConfig
Runtime knobs for the relay’s connection-acceptance path — threaded through Relay::from_endpoint / crate::sfu::Sfu::from_endpoint. Default carries the production values; tests substitute a short auth_deadline, a tiny SessionLimits, or a tiny/fast IngressLimits to exercise the reject paths deterministically without spinning up hundreds of connections or waiting out production timers.
SessionGuard
RAII guard for one admitted session: decrements the global count and its user’s per-user entry on drop (removing the entry once it reaches zero), so a panic or early return anywhere in the connection handler still releases capacity — admission can never wedge on a phantom session.
SessionLimits
Session admission caps, enforced by SessionRegistry::admit right after a successful auth and before the moq-net handshake runs.
SessionRegistry
Shared session accounting: a global count plus a per-user count, gating admission at SessionLimits and released via SessionGuard. Cheap to clone (every field is Arc-backed) — each accept-loop task holds its own clone.

Constants§

AUTH_DEADLINE
Deadline for the whole relay-token control-stream handshake (authenticate: accept_bi + the two read_exacts). QUIC’s automatic keepalives mean the idle timeout never fires on its own, so a bound here is the only thing stopping a client that completes the QUIC handshake but never opens (or never finishes) the token stream from holding the connection — and the task serving it — open indefinitely. SessionRegistry admission only ever runs after a successful auth (see handle_connection), so this deadline is also what keeps an unauthenticated connection from ever holding a session slot or blocking the SIGTERM drain wait: bounding “unauthenticated” bounds the exposure. Production default (see RelayConfig::default); tests substitute a much shorter value to exercise the timeout deterministically.
AUTH_OK 🔒
One-byte control-stream replies the relay sends after token verification. The client gates its MoQ handshake on reading AUTH_OK.
AUTH_REJECT 🔒
AUTH_REJECT_DRAIN 🔒
How long the relay waits, after sending an AUTH_REJECT, for the client to read the verdict and close the connection before the handler gives up. A safety valve so a vanished client can’t pin the task; we only care that the reject byte had a chance to land.
CLOSE_BROADCAST_QUOTA_EXCEEDED
Close code for exceeding IngressLimits::max_broadcasts_per_session. See CLOSE_INGRESS_BUDGET_EXCEEDED.
CLOSE_INGRESS_BUDGET_EXCEEDED
Application-level QUIC close code ingress_watchdog sends when it closes a connection for exceeding IngressLimits::max_bytes_per_sec — distinct from the generic 0 code the auth/goodbye paths use, so a client inspecting the code/reason on conn.closed() can tell “the relay enforced a policy” apart from “auth failed” or “the network died”.
MAX_TOKEN_LEN 🔒
Maximum relay-token length accepted on the control stream. A valid token is 36 + 1 + 36 + 1 + ~10 + 1 + 64 ≈ 150 bytes; this bound stops a malicious client pinning memory with a giant “token”.
NAMESPACE_CONVENTION
The broadcast-namespace convention every announce/subscribe must obey.
OPS_LOG_INTERVAL 🔒
Cadence of the per-connection ops summary line ingress_watchdog logs. Fixed, not configurable: it’s per-connection and SessionLimits already bounds how many connections exist, so log volume is bounded by construction without a knob.
RELAY_ALPN
MoQ ALPN the relay accepts. Must be a moq-net ALPN: in raw mode web-transport-iroh reports the iroh connection’s ALPN as the MoQ protocol and moq-net rejects an unrecognised one with Error::UnknownAlpn. moql is moq-net’s version-negotiation ALPN and matches the client (hound_transport::ALPN). This is the only ALPN the SFU endpoint advertises.
RELAY_ALPN_PREFIX
Marker documenting that RELAY_ALPN is the moq-net negotiation ALPN, not a free-to-choose hound-* value (the latter fails the handshake).
