§Media packet header

Every media packet on the wire is PacketHeader (fixed 28 bytes, all little-endian) followed by an opaque payload fragment. The header is RTP-inspired but our own, carried inside iroh QUIC datagrams (QUIC already provides encryption and transport, so there is no SRTP/RTP-over-UDP here).

§Layout (28 bytes, little-endian)

Offset  Size  Field          Notes
──────  ────  ─────────────  ─────────────────────────────────────────────
 0       1    version        Protocol version. Receivers reject mismatches.
 1       1    codec          Payload codec (see Codec).
 2       1    track          Which A/V track (see TrackKind).
 3       1    flags          bit0 = keyframe, bit1 = end-of-frame (marker).
 4–5     2    sender         SFU-assigned peer id of the sender.
 6–7     2    reserved       Must be 0 (alignment / future use).
 8–11    4    packet_seq     Per-(sender,track) packet counter. Loss/NACK/
                             reorder detection at the transport level.
12–15    4    frame_seq      Per-(sender,track) frame counter. Groups the
                             fragments of one encoded frame.
16–17    2    frag_index     0-based fragment index within the frame.
18–19    2    frag_count     Total fragments for the frame (≥ 1).
20–27    8    timestamp_us   Capture time, microseconds, on the sender's
                             single shared audio+video clock (for A/V sync).

packet_seq is for the transport (loss/NACK/ordering); frame_seq + frag_index/frag_count are for reassembling a frame. Keeping both is deliberate: the transport can detect/repair packet loss without parsing frame structure, and reassembly is explicit rather than inferred from the marker bit + sequence gaps.
