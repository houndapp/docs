§Sender-side video adaptation (the BWE → encoder glue)

The deterministic, sans-IO core that turns the signals a publisher can observe without any new transport surface into encoder decisions. It wraps three hound_media pieces and is the single place the cam-publish path consults each tick:

  • RateController — loss-based AIMD BWE. Produces a target send bitrate.
  • QualityLadder — maps that bitrate to a discrete video Rung (or Quality::AudioOnly when even the lowest rung won’t fit). Audio is sacred; video is the release valve.
  • KeyframeThrottle — coalesces a PLI storm into one IDR and spaces forced keyframes so a burst of receiver requests can’t spike the bitrate.

§Why these signals

Real BWE wants receiver-reported loss. VideoAdaptation folds three signals into one loss-equivalent each tick:

  • Encoder/publisher backpressure. The capture→publish channel is bounded; when the encoder outruns the publisher (a slow/congested uplink) frames are dropped at try_send. The dropped-fraction over an interval is a direct, local overload signal — it behaves like loss for the AIMD controller.
  • RTT. iroh::endpoint::Connection::rtt is a cheap delay signal. A sharp rise over the connection’s own baseline is bufferbloat — congestion before outright drops — so we fold an RTT-inflation term into the loss-equivalent.
  • Receiver-reported loss. The receiver→producer feedback loop (the MediaFeedback/MediaFeedbackReceived control messages) reports the loss fraction each subscriber observes on the downstream leg the relay hides — the one signal the sender cannot see locally. When several receivers report, the orchestrator pre-reduces to the worst leg (serve the worst receiver) and passes that single fraction in.

All three collapse to a single loss-equivalent fraction fed to the AIMD controller, keeping the controller’s contract (one loss number per interval) intact. The mapping is the only policy here.

§Determinism

Everything here is pure: time enters as now_us, every signal is an explicit input, and the only outputs are returned data (AdaptDecision). No clocks, no IO, no threads. The orchestrator samples the live signals and feeds them in; this core decides. That keeps the policy unit-testable with synthetic signals and no devices (see the tests below).

How an RTT rise maps to loss-equivalent. Tuned conservatively: the delay signal nudges the controller toward backing off before drops appear, but drops (real overload) dominate when both fire.
AdaptDecision
What the encoder/publisher should do after a tick. Pure data — the orchestrator applies it (calls set_bitrate, pauses/resumes publish, forces a keyframe).
AdaptSignals
The signals one publish tick observes. Built by the orchestrator from the bounded-channel drop counters, the iroh connection RTT, and (when receivers report) the worst receiver-reported loss fraction this interval.
GeometryDwell
Dwell-based hysteresis for the encoder-REBUILD geometry decision (pure, sans-IO — see GEOMETRY_REBUILD_DWELL_US for why this must not track the per-tick Rung directly). Self::observe feeds the freshly-decided rung HEIGHT each tick (None for Quality::AudioOnly — no video decision, so the last committed geometry carries through unchanged; encode is paused anyway) and returns the geometry the encoder should be built at right now: unchanged unless the SAME candidate has held for GEOMETRY_REBUILD_DWELL_US continuously — any intervening DIFFERENT candidate (a still-settling climb, or genuine flapping) resets the timer, so only a genuinely stable rung ever triggers a rebuild.
VideoAdaptation
Sender-side video adaptation: the BWE controller, the quality ladder, and the keyframe throttle behind one sans-IO façade.

Constants§

GEOMETRY_REBUILD_DWELL_US
How long a candidate encode geometry must hold, unchanged, before the encoder actually rebuilds at it. Without a dwell, ordinary AIMD noise — or the delay-based estimator pinning low for a beat before recovering — would rebuild the encoder every few seconds. ~5 ticks at the 1 s adaptation cadence the client drives VideoAdaptation::on_tick at.

fit_encode_dims 🔒
Fit a rung’s target HEIGHT to the SOURCE capture’s own aspect ratio, even- clamped. The camera and screen quality ladders both store a nominal 16:9 width on each Rung purely for the bitrate model (see screen_rungs) — a screen source is not always 16:9 (a portrait phone mirror, a near-square shared window), so encoding at the rung’s raw width would stretch it. Deriving the width from the source’s own aspect instead — the same approach the low simulcast layer uses for its fixed low-resolution encode — keeps every encode at the right aspect regardless of what the ladder’s rung table says. For the camera ladder, whose capture geometry and rungs are both already 16:9, this reproduces the rung’s own width exactly.
