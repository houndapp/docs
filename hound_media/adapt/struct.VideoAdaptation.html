Expand description

Sender-side video adaptation: the BWE controller, the quality ladder, and the keyframe throttle behind one sans-IO façade.

Fields§

§cfg: AdaptConfig§rate: RateController§ladder: QualityLadder§keyframe: KeyframeThrottle§rtt_baseline_us: Option<i64>

Lowest RTT yet observed — the connection’s own healthy baseline. The delay term measures inflation above this, so a high-but-stable RTT link (a far relay) is not perpetually penalised; only growth signals congestion.

§last_quality: Quality

The last decided quality, so Self::on_tick can report quality_changed and the orchestrator only acts on transitions.

Source

pub fn new(start_bps: u64, ladder: QualityLadder, cfg: AdaptConfig) -> Self

Build the adaptation stack for a camera publisher. start_bps seeds the rate controller (typically the static cam target the encoder launched at) and the ladder ceiling caps it at the publisher’s native resolution rung so we never try to upscale past what the camera produces.

Source

pub fn standard(cam_bitrate_bps: u64) -> Self

The standard camera ladder for the common case. cam_bitrate_bps is the encoder’s static launch rate, used to seed the rate controller. The ladder’s top rung is 720p — exactly the cam capture config — so no ceiling cap is needed: the climb naturally tops out at what the camera produces.

Source

pub fn screen_with(screen_bitrate_bps: u64, quality: ScreenQuality) -> Self

A screen-share ladder for the sharer’s ScreenQuality — their motion↔legibility priority plus max-resolution/fps caps. screen_bitrate_bps is the encoder’s static launch rate, used to seed the rate controller. The default ScreenQuality is the legibility ladder (drop fps before resolution — the opposite trade-off to Self::standard). Everything else (the AIMD controller, the RTT/backpressure/remote-loss fold, the keyframe throttle) is identical to the camera path; only the bitrate→rung map differs.

Source

pub fn set_screen_quality(&mut self, quality: ScreenQuality)

Swap the quality ladder live when the sharer changes their ScreenQuality on an active share (motion↔legibility, or a resolution/fps cap). Only the bitrate→rung mapping changes: the AIMD controller’s current estimate is preserved, so the next Self::on_tick re-picks a rung from the new ladder at the live bitrate (and writes the new target_fps/geometry downstream). No encoder touch here — the capture thread reacts on the next tick.

Source

fn loss_equivalent(&mut self, sig: &AdaptSignals) -> f64

Fold this tick’s signals into a single loss-equivalent for the AIMD controller. Drops dominate; RTT inflation adds a capped nudge. Pure.

Source

pub fn on_tick(&mut self, sig: &AdaptSignals, now_us: i64) -> AdaptDecision

Advance the adaptation one tick: fold the signals into a loss-equivalent, run the AIMD controller, pick a ladder rung, and report the decision. The orchestrator applies the result (set encoder bitrate, pause/resume, etc.).

Source

pub fn on_pli(&mut self)

A receiver PLI arrived for our track (relayed via control). Coalesced — many PLIs before the next Self::poll_keyframe still mean one IDR.

Source

pub fn poll_keyframe(&mut self, now_us: i64) -> bool

Whether a forced keyframe is due now (a pending PLI and the min interval elapsed). true at most once per interval, coalescing a storm into one IDR. The orchestrator calls encoder.request_keyframe() on true.

Source

pub fn on_keyframe_emitted(&mut self, now_us: i64)

Record that a keyframe was emitted (forced or the encoder’s own periodic one) so the throttle measures its interval from the real emission and a pending request is satisfied without a redundant forced IDR.
