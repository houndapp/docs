§Receiver-side downlink demand

Complements crate::layer::tile_quality_request (rendered-tile-size demand) with the BANDWIDTH half of demand-driven simulcast: when this receiver’s downlink can’t keep up with a video track, ask the producer for the low layer. The two demands are folded with crate::layer::most_constrained by the caller — this module only decides the bandwidth half.

§Signal: frame-sequence gaps, not the sender’s own bitrate

Each wire frame (crate::fragment::Frame) carries a per-track frame_seq. A gap in the sequence a receiver observes means frames were lost between the sender and this receiver — unlike a raw delivered-bytes rate, it is not confounded by the sender’s own encoder adapting its bitrate (which changes how BIG frames are, not how many of them arrive). FrameSeqWindow folds arrivals into a windowed (received, expected) count; DownlinkDemand turns that ratio into a QualityRequest.

§Thresholds: retuned after a cratering regression

An earlier receive-side decider (DownlinkEstimator/QualityRequestDecider, removed as dead code — see commit history) capped a receiver at 0.85× the source bitrate. That turned a MINOR dip into a resolution cliff: a healthy 1:1 720p call would crater to 360p on a transient wobble, because 0.85 is barely below “normal” jitter in the estimate. This decider caps only a genuinely poor link:

  • drop to QualityRequest::Capped once delivery falls below 1/4 of source
  • restore QualityRequest::Source only once delivery climbs back above 1/2 of source

The wide [1/4, 1/2] band is deliberate anti-flap hysteresis — a link wobbling anywhere in that band (including the specific case of ~76% delivery, which the OLD 0.85 threshold would have capped) holds its current demand. Note also: with a single receiver, the sender’s own loss-based AIMD (crate::congestion::RateController, driven by MediaFeedback reports) already right-sizes the one stream it’s publishing — this decider only earns its keep once a call has enough heterogeneous receivers that no single sender-side rate can serve everyone (see the module docs on the call site for how demand is gated).

Sans-IO: time enters only as an explicit now_us argument (mirrors crate::congestion::RateController / crate::send_budget::SendBudget); no clock reads, no I/O.

DownlinkConfig
Tunables for DownlinkDemand.
DownlinkDemand
The receiver’s bandwidth-driven demand for one subscribed video track. Folds arriving frames’ frame_seq into a FrameSeqWindow and, once per DownlinkConfig::eval_interval_us, turns the window’s delivery ratio into a QualityRequest with the [source/4, source/2] hysteresis band described in the module docs. Pure/sans-IO — Self::observe_frame takes now_us and source_bps explicitly.
FrameSeqWindow
Per-track receive-completeness tracker: folds each arriving frame’s wire frame_seq into a windowed (received, expected) count. One instance per subscribed video track — a fresh instance per subscription, so a resubscribe (a new producer’s independent frame_seq space) starts clean.

Constants§

CAP_CHATTER_DEN 🔒
CAP_CHATTER_NUM 🔒
A new Capped value while already capped is only re-emitted (and the demand updated) once it has moved at least this much relative to the standing value — otherwise ordinary window-to-window wobble in a poor-but-stable link would spam the producer with a new cap every eval tick.
MAX_FRAME_GAP 🔒
Bound on a single observed frame_seq gap. A restart (a fresh producer’s frame_seq starts back at 0) or a corrupt/wild value must not be misread as millions of lost frames — far above any gap a live call could produce (a stalled pipeline is reaped and re-subscribed long before frame_seq could legitimately jump this far).
MIN_CAP_BPS 🔒
A Capped request never asks for less than this — mirrors crate::congestion::RateConfig’s default min_bps floor, so a genuinely terrible link still asks for a minimally-useful low layer instead of degenerating toward zero.
