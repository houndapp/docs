§Delay-based bandwidth estimation (sans-IO)

Loss-based control (the crate::congestion::RateController AIMD) only reacts once a queue has already overflowed and packets are being dropped — by then the call has already stuttered. A bottleneck queue announces itself earlier: as it fills, each packet waits a little longer than the last, so the one-way delay trends upward while loss is still zero. Sensing that trend lets a receiver back its quality request off before the link actually drops frames.

This is the delay-based half of Google Congestion Control (GCC), distilled to the pieces that matter here and kept sans-IO (timestamps in, a usage signal + estimate out — no clock, no sockets, fully unit-tested):

  • TrendlineEstimator folds each frame’s inter-arrival delay variation d = (recv_i − recv_{i−1}) − (send_i − send_{i−1}) into a smoothed accumulated delay and fits a least-squares slope over a sliding window. A positive slope means delay is growing — a queue is building.
  • an adaptive threshold gamma that tracks the trend’s own scale, so the detector neither false-positives on a quiet link nor goes numb on a noisy one, plus an overuse-duration gate so a single spike isn’t called congestion.
  • DelayBasedBwe runs AIMD over that Usage signal: multiplicative-decrease on Overuse, additive-increase on Normal, hold on Underuse (a queue is draining — don’t push more in). Its estimate is combined with the loss-based one by taking the minimum (whichever controller is more worried wins), exactly as GCC does.

Clocks need not be synchronised between sender and receiver: only differences of timestamps are used, so any fixed offset cancels (same property the crate::jitter::JitterEstimator relies on).
