Expand description

Fits the trend (slope) of accumulated one-way delay over a sliding window and classifies it against an adaptive threshold into a Usage.

Pure/sans-IO: feed Self::observe each frame’s (recv_us, send_us); read Self::usage.

EWMA-smoothed accumulated delay; the regression is fit over this.

§window_size: usize§first_recv_us: Option<i64>§gamma: f64

Adaptive overuse threshold (unitless, compared against the modified trend).

§overuse_us: f64

Consecutive time (µs) the trend has been over threshold (overuse-duration gate).

§overuse_count: u32

Consecutive over-threshold observations (overuse-count gate). Together with overuse_us this debounces a one- or two-frame jitter spike, which at a 30 fps cadence would otherwise satisfy the time gate alone on its very first frame.

§prev_trend: f64§last_update_us: Option<i64>§num_deltas: usize

Total deltas folded (the modified trend scales by min(this, MIN_DELTAS) so a short window still yields a comparable magnitude — WebRTC’s kMinNumDeltas).

§state: Usage

The latest verdict from Self::classify.

The modified trend scales by min(num_deltas, MIN_DELTAS), so the magnitude is cadence-independent once enough deltas have accrued (WebRTC’s kMinNumDeltas).

Trend must stay over threshold this long before Overuse fires (debounce).

Source

const OVERUSE_COUNT: u32 = 2

…AND over threshold for at least this many consecutive observations. WebRTC uses > 1; video frames are far coarser than its packet-group cadence, so a slightly larger count keeps a brief jitter burst from reading as congestion.
