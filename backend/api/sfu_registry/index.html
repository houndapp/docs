The multi-node SFU registry: which SFU nodes are live, and which one is “best” for a new voice join right now.

§Why a registry, not a single slot

A blue/green SFU rollout runs the old and new pod side by side for a while (existing calls drain off the old one; new joins should land on the new one). A single sfu_node_id: Option<String> slot (the pre-rollout-support shape) can only ever name one node, so the last registration to land wins — during a rollout that’s a coin flip between old and new pod, and it offers no way to say “this node is going away, don’t route new joins to it.” The registry keeps every node’s last-seen registration plus its draining flag and picks the best live one at read time.

§In-process only

Like super::control_calls, this is in-memory and per-process — it does not survive a backend redeploy and does not coordinate across horizontally-scaled replicas. That’s fine: the SFU self-heals by re-registering every REGISTER_INTERVAL (services/sfu/src/main.rs), so a backend restart clears the registry and it repopulates within one interval, same as the single-slot predecessor.

§Liveness

The SFU re-registers every 30s. A node is “live” if it registered within LIVENESS_TTL (3 missed cycles). Liveness is evaluated at read time (select_best) rather than by a background reaper — a stale entry just stops being selectable; nothing needs to actively evict it, and the map never grows unbounded because node IDs are stable per-pod-lifetime (a replaced pod re-registers under a fresh ephemeral identity, but the old entry ages out and is simply never selected again — see the SfuNode churn note below if that ever needs bounding).

Constants§

LIVENESS_TTL 🔒
A node is live if it registered within this many registration intervals. The SFU re-registers every 30s; 90s tolerates two missed cycles (a slow backend, a blip) before presuming the node dead.

Functions§

seed 🔒
Seeds the registry at boot from the SFU_NODE_ID env var, so a dev/compose setup that only sets the env (never calls BACKEND_URL self-registration) keeps working exactly as the old single-slot field did. The seeded entry expires via LIVENESS_TTL like any other node unless something re-registers it before then.
select_best 🔒
Picks the best node to route a new voice join to: the newest (by first registration) live, non-draining node; if every live node is draining, the newest live node anyway (serving a call on a draining node beats refusing it outright — the node keeps accepting connections while it drains, see services/sfu/src/main.rs). None if no node has been seen within LIVENESS_TTL.
