Browser-deferred passkey login.

On platforms without a system passkey provider (Linux) — or for headless callers like houndctl — the WebAuthn ceremony runs in the user’s browser against the hosted auth page (services/auth), which finishes the ceremony with the backend and redirects to a loopback callback carrying the token pair. This crate owns that flow’s client half:

  1. BrowserLogin::begin binds an ephemeral 127.0.0.1 listener and mints a CSRF state nonce; BrowserLogin::url is the page to open.
  2. The caller opens the URL (open_in_browser) and shows it as a fallback.
  3. BrowserLogin::wait accepts the redirect, verifies state, answers the browser with a success page, and yields the Tokens.

Tokens are the caller’s problem from there (houndctl persists them to its config file; the client engine adopts them into its Session).

CallbackResult
What the auth page’s loopback redirect carried: a session; a Steam link token — the Steam ID is unknown to the backend (create-or-link decision pending) or ownership proof was requested with Mode::SteamLink; or a bare “linked” confirmation from the email-link ceremony (BrowserLogin::begin_email_link), which mints no tokens because the app already holds a session.
Mode
Which flow the auth page opens on. SignIn/SignUp pick the initial passkey view (the page lets the user switch). The Steam* modes redirect straight into Steam OpenID: Steam to sign in, SteamLink to prove Steam-account ownership for linking to an already-signed-in account.

Accept connections until one is the /callback redirect, then extract the token pair (or Steam link token). Non-callback requests (e.g. a favicon probe) get a 404 and the wait continues; a wrong state is answered 400 and aborts the flow.
